## Glossary

- **Form 1040 Line 3c** — "Check if your child's dividends are included in 1 ☐ Line 3a   2 ☐ Line 3b." A **pair of checkboxes** new to the tax-year-2025 revision of Form 1040. Each checkbox is a single Yes/No disclosure — there is no dollar amount on Line 3c. Box 1 discloses that some of the amount on Line 3a came from a child; box 2 discloses the same for Line 3b.
- **Form 1040 Line 3a** — "Qualified dividends." The dollar amount on which Line 3c box 1 reports. See `XLS/Computations/3a.md`.
- **Form 1040 Line 3b** — "Ordinary dividends." The dollar amount on which Line 3c box 2 reports. See `XLS/Computations/3b.md`.
- **Form 8814** — "Parents' Election To Report Child's Interest and Dividends." When a parent elects under Internal Revenue Code §1(g)(7) to include a qualifying child's investment income on the parent's own return, Form 8814 produces a per-child set of intermediate values. The single value relevant to Line 3c is **Form 8814 line 9**, which is the qualified-dividend portion of the child's income above the $2,700 Form 8814 line-5 base amount.
- **Form 8814 line 9** — Computed per child as `line 6 × (line 2b / line 4)`, where line 6 = total above-base income, line 2b = child's qualified dividends from 1099-DIV box 1b, and line 4 = child's total qualified-election income. The IRS instructions for line 9 explicitly say: "Include the amount from line 9 on the parent's Form 1040 line 3a (Qualified dividends) AND on line 3b (Ordinary dividends). Check the appropriate boxes on Form 1040 line 3c."
- **Symmetric trigger** — Both Line 3c checkboxes fire on the same single condition (Form 8814 line 9 > 0). This is a consequence of the Form 8814 architecture, not a coincidence — line 9 is the ONLY child dividend amount that flows to either parent line, and it flows to BOTH lines simultaneously because qualified dividends are a strict subset of ordinary dividends per Internal Revenue Code §1(h)(11).
- **TRUE-or-null semantics** — The application represents each checkbox as a Java `Boolean` field. The value is either `Boolean.TRUE` (render as checked) or `null` (do not render — equivalent to "unchecked"). The value is never `Boolean.FALSE` because rendering an explicit "unchecked" box would be visually indistinguishable from the default unchecked state but would inflate the JSON payload.
- **Semantic PDF field map** — The CSV file `f1040_field_mapping_semantic.csv` that maps raw IRS PDF field identifiers (like `c1_33[0]`) to human-readable semantic names that the UI consumes when filling the PDF for export.

## 1. IRS rule for Line 3c (computed before reading the code)

This section states what the IRS expects the Line 3c checkboxes to display, based on the 2025 instructions and the cited authorities. The next section (§2) describes what the code actually does, and §3 compares the two.

### 1.1 Plain-English equation

Conceptually, Line 3c is asking: "Are any of the dividend amounts shown on Lines 3a and 3b actually your child's dividends that you elected to report on your own return using Form 8814?"

The two boxes are independent disclosures (the IRS designed Line 3c as two separable checkboxes — see §1.6), but in the actual Form 8814 architecture they always fire together because the only child dividend amount that lands on parent's Lines 3a or 3b is Form 8814 line 9, which goes to **both** Lines 3a and 3b symmetrically.

In ordinary arithmetic (treating TRUE as 1 and FALSE/null as 0):

```
form8814_qualified_dividends_total = sum across all elected children of
                                     Form 8814 line 9
                                     (= line 6 × line 2b / line 4 per child,
                                      summed across the form8814List)

line_3c_box_1_check_if_in_3a       = (form8814_qualified_dividends_total > 0)
                                     // TRUE when the parent elected Form 8814
                                     // for at least one child with qualified
                                     // dividends; otherwise blank.

line_3c_box_2_check_if_in_3b       = (form8814_qualified_dividends_total > 0)
                                     // Same condition as box 1, because line 9
                                     // flows to BOTH Line 3a and Line 3b per
                                     // Form 8814 line 9 instructions.
```

There are no other inputs and no dollar amounts. The two outputs are pure Boolean disclosures.

### 1.2 Invariants the IRS expects

1. **Line 3c is blank when no Form 8814 is filed.** If the parent did not elect to report any child's investment income, neither checkbox should appear marked.
2. **Line 3c box 1 fires if and only if Form 8814 contributed any qualified dividends to Line 3a.** Because Form 8814 line 9 is the only path from a child to parent's Line 3a, this reduces to "fire when Form 8814 line 9 total > 0."
3. **Line 3c box 2 fires under the same condition as box 1**, because Form 8814 line 9 (qualified-dividend portion above the $2,700 base) flows symmetrically to BOTH Line 3a and Line 3b — qualified dividends are a subset of ordinary dividends per Internal Revenue Code §1(h)(11), so they appear in both totals.
4. **Line 3c is blank when Form 8814 line 4 ≤ $2,700** (the line-5 base amount for tax year 2025). In this case lines 6, 9, and 10 are not computed per the Form 8814 simplified-path instructions, so there is no line 9 contribution and no Line 3c disclosure to make.
5. **Line 3c is blank when the child has only interest income (no dividends).** Form 8814 line 9 = $0 in this case (the ratio `line 2b / line 4` is `0 / something`), so no qualified dividends flow → no checkbox.
6. **Line 3c box 1 alone fires never** under the current IRS architecture (because line 9 always goes to both Lines 3a and 3b together). Similarly, box 2 alone fires never. So the only two valid states are "both checked" or "both blank."

### 1.3 Input variables (one row per amount the IRS rule consumes)

| Variable name | Where the user enters it | What it represents |
|---|---|---|
| `line2bQualifiedDividends` (per child) | `form-child-interest-dividends.xlsx` (Form 8814 manual entry) OR derived from a child's 1099-DIV `qualifiedDividendsAmount` when the 1099-DIV `recipientTIN` matches a dependent's SSN | The child's qualified dividends from 1099-DIV box 1b — the numerator in the Form 8814 line 9 ratio. |
| `line1aTaxableInterest`, `line2aOrdinaryDividends`, `line3CapGainDistributions` (per child) | `form-child-interest-dividends.xlsx` OR derived from child statements | Components of Form 8814 line 4 (denominator in the line 9 ratio). The IRS rule for Line 3c depends only indirectly on these — they determine whether line 9 will be > 0. |
| `dependents[*].ssn` | `form-identification-dependent.xlsx` | Used to match child statements (1099-INT, 1099-DIV) to a specific dependent so Form 8814 can be auto-derived from statements. |

**No direct Line 3c user input field exists.** The user does not enter "check Line 3c box 1 — Yes/No" anywhere. Both checkboxes are derived 100% from the Form 8814 line 9 computation.

### 1.4 Reference data — constants used in the equation

| Constant | Value (TY 2025) | Where it lives in code | IRS source |
|---|---|---|---|
| Form 8814 line 5 base amount | **$2,700** | `ReferenceData.FORM8814_BASE_AMOUNT` | 2025 Form 8814 instructions, line 5 |
| Form 8814 child-income gross-income ceiling | **$13,500** | `ReferenceData.FORM8814_GROSS_INCOME_LIMIT` | 2025 Form 8814 instructions, "Parents' election" eligibility |

The base amount drives whether line 9 is computed at all (only when line 4 > $2,700). The gross-income ceiling drives whether Form 8814 can be filed at all (denied at or above $13,500 — child must file own return). Neither constant directly figures in the Line 3c arithmetic, but both gate whether the line-9 trigger ever fires.

### 1.5 Decision tree (which path applies)

```
Is any Form 8814 filed?
├── No  → Both Line 3c boxes blank; STOP
└── Yes ↓

For each filed Form 8814:
  Is the child's line 4 > $2,700?
  ├── No  → This Form 8814 contributes $0 to form8814_qualified_dividends_total
  └── Yes ↓
      Is the child's line 2b (qualified dividends) > 0?
      ├── No  → line 9 = 0 → this Form 8814 contributes $0
      └── Yes → line 9 = line 6 × line 2b / line 4 > 0
                → this Form 8814 contributes a positive amount

Sum all per-child line 9 amounts → form8814_qualified_dividends_total.

Is form8814_qualified_dividends_total > 0?
├── Yes → Line 3c box 1 = TRUE; Line 3c box 2 = TRUE.
└── No  → Both Line 3c boxes blank.
```

**PDF location:** PDF fields `c1_33[0]` (box 1) and `c1_34[0]` (box 2) on page 1 of Form 1040 at approximate coordinates `(283.8, 187.5)` and `(363, 187.5)` respectively.

### 1.7 Note on the IRS-design intent vs the practical reality

The IRS designed Line 3c as **two conceptually independent checkboxes** (per the 2025 Form 1040 layout — they are drawn as two separate checkboxes with separate captions "1" and "2"). The intent appears to anticipate a future scenario where a child's ordinary dividends might flow to Line 3b independently of any qualified-dividend flow to Line 3a — for example, if the IRS ever splits Form 8814 to route line 2a ordinary divs separately to Line 3b without going through line 9.

In the current (2025) Form 8814 architecture, the only child dividend amount that flows to either parent line is line 9, and line 9 ALWAYS goes to both lines together. So box 1 alone or box 2 alone can never validly fire under 2025 rules. The application correctly implements the 2025 architecture; the symmetric-trigger behavior is intentional and matches `lines/3abc.md` §5.1 and the symmetric-trigger breadcrumb at `TaxReturnComputeService.java:7728-7745`.

### 1.8 Worked examples

**Example A — Parent has no Form 8814:** No `child-interest-dividends` entries; no dependent has a 1099-INT or 1099-DIV with a matching SSN. → `form8814List` empty. → `form8814_qualified_dividends_total` = null. → Both Line 3c boxes blank.

**Example B — Single child, interest only ($3,000):** Form 8814 line 4 = $3,000 (interest only); line 2b = $0 (no dividends). → Line 9 ratio = $0 / $3,000 = 0 → line 9 = $0. → `form8814_qualified_dividends_total` = 0 (or null). → Both Line 3c boxes blank.

**Example C — Single child, qualified dividends ($800), other income too:** Child's 1099-DIV box 1a = $2,000 + 1099-DIV box 1b = $800 + 1099-INT box 1 = $1,200. → Form 8814 line 4 = $1,200 + $2,000 = $3,200; line 6 = $500; line 9 = $500 × ($800 / $3,200) = $125. → `form8814_qualified_dividends_total` = $125 > 0. → Line 3c box 1 = TRUE; Line 3c box 2 = TRUE.

**Example D — Two children, both with qualified dividends:** Each child's line 9 = $300 (per Example C math); aggregate = $600 > 0. → Line 3c box 1 = TRUE; Line 3c box 2 = TRUE.

**Example E — Child gross income at the $13,500 ceiling:** Form 8814 election denied; blocking flag `FORM8814_CHILD_GROSS_INCOME_TOO_HIGH` fires; no Form 8814 instance created. → `form8814_qualified_dividends_total` = null. → Both Line 3c boxes blank.

## 2. Code implementation (what the code actually does)

### 2.1 Backend — symmetric trigger derivation

The Line 3c checkboxes are derived inside `computeDividendIncome` in `TaxReturnComputeService.java`, immediately after Line 3a and Line 3b are finalized:

| Code location | Action |
|---|---|
| `:387-388` (`prepare()`) | Aggregate per-child Form 8814 line-9 amounts into `form8814QualifiedDividendsTotal` by summing `Form8814.line9QualifiedDividends()` across the `form8814List`. |
| `:7621` | `boolean hasChildQualifiedDividends = hasPositiveAmount(form8814QualifiedDividendsTotal);` |
| `:7622-7625` | If `hasChildQualifiedDividends`, add `form8814QualifiedDividendsTotal` symmetrically to BOTH `line3a` and `line3b` (the dollar contribution; this is the upstream of the disclosure). |
| `:7752-7753` | Return the two `Boolean` fields on the `DividendComputation` record: `hasChildQualifiedDividends ? Boolean.TRUE : null` for **each** field. |
| `:7728-7745` | A 17-line breadcrumb explaining that the SYMMETRIC TRIGGER (both checkboxes share the single `hasChildQualifiedDividends` condition) is intentional, not a DRY violation. |

The TRUE-or-null pattern is enforced at the record construction at `:7752-7753` — never `Boolean.FALSE`.

### 2.2 Backend — wiring into the Income output

`computeInterestIncome` at `:7156-7157` reads the two fields off the `DividendComputation` record and writes them through to the Income object construction at `:6702` and `:6705` — where the values are also defensively coalesced to `null` when received as anything other than `Boolean.TRUE`.

### 2.3 Frontend — PDF export

`form-tax-return-1040.component.ts:756-757` reads the two fields off the income JSON and writes them into the PDF field-value map:


### 4.4 Compute order and dependencies

1. **Upstream**:
   - `dependents/*` records — must exist before child statements can be matched to a dependent.
   - 1099-INT / 1099-DIV statements with `recipientTIN` matching a dependent SSN — the statement-driven path.
   - `child-interest-dividends` personal-form entries — the manual path (overrides statement-derived values per dependent).
   - `computeChildInterestDividends` → builds `form8814List`.
   - `buildForm8814` per child → computes `line9QualifiedDividends` (Part I gated by line 4 > $2,700; election denied at line 4 ≥ $13,500).
   - `prepare()` aggregates `form8814QualifiedDividendsTotal = Σ Form8814.line9QualifiedDividends`.
2. **Self**:
   - `computeDividendIncome` reads `form8814QualifiedDividendsTotal`, derives `hasChildQualifiedDividends`, returns the two `Boolean` checkbox fields on the `DividendComputation` record.
   - `computeInterestIncome` propagates them through to `Income.line3cChildDividendsInLine3a` / `line3cChildDividendsInLine3b`.
3. **Downstream**:
   - JSON output → `form1040.income.line3cChildDividendsInLine3a` / `line3cChildDividendsInLine3b`.
   - UI → `form-tax-return-1040.component.ts` populates the two PDF field-value-map keys.
   - PDF export → `pdf-lib` looks up the field by semantic name in `f1040_field_mapping_semantic.csv` (CSV rows 23-24 carry the matching names as of 2026-06-03 Gap 1 closure; lookup now resolves end-to-end).

### 4.5 Blocking flags

Line 3c does not emit any flags of its own. The upstream Form 8814 flags that gate whether Line 3c can fire are:

| Flag | Severity | Condition |
|---|---|---|
| `FORM8814_CHILD_GROSS_INCOME_TOO_HIGH` | Blocking | Any elected child has gross income ≥ $13,500. Form 8814 is not generated and the child contributes nothing to `form8814QualifiedDividendsTotal`, so Line 3c stays blank. |

## 6. Common mistakes (Line 3c)

These mistakes are NOT present in the current code, but are listed because they have appeared in earlier audits and the verified-correct items in §3.3 protect against them. (Gap 1 from §3.2 was closed on 2026-06-03; no compute-side mistakes remain.)

- Treating Line 3c as a dollar amount instead of two Boolean checkboxes.
- Firing box 1 alone when Form 8814 contributes qualified dividends — INCORRECT; box 2 also fires because line 9 goes to both Lines 3a and 3b.
- Firing box 2 on Form 8814 line 2a > 0 (full ordinary divs) — INCORRECT; only line 9 (qualified subset) flows to parent's lines, so the trigger is line 9 > 0, not line 2a > 0.
- Leaving Line 3c blank when Form 8814 IS filed with qualified dividends — a "missing disclosure" issue that would draw IRS attention.
- Setting `Boolean.FALSE` instead of null — inflates the JSON payload and risks an "explicit unchecked" render in some PDF tools.
- Forgetting that Form 8814 line 4 ≤ $2,700 means Part I is skipped → line 9 = null → Line 3c stays blank even though Form 8814 was nominally "filed."
- Forgetting that the $13,500 ceiling denies the election entirely → no Form 8814 instance → no Line 3c contribution.
