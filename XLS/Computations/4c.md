## Glossary

- **Form 1040 Line 4c** — In the IRS 2025 Form 1040, "Line 4c" is the **disclosure annotation space** next to Lines 4a and 4b where the taxpayer writes one or more labels explaining why the gross (Line 4a) differs from the taxable amount (Line 4b). The IRS does NOT print explicit "box 1 / box 2 / box 3" labels on the form; the application models the disclosure as three internal Boolean flags plus a free-form write-in string. The Tax Return JSON output names them `line4cBox1Rollover`, `line4cBox2Qcd`, `line4cBox3Other`, and `line4cBox3Text`.
  - **box 1 / "Rollover"** — fires when any part of the distribution was rolled over (Internal Revenue Code §408(d)(3)). The IRS instructions tell the user to write "Rollover" next to Line 4b.
  - **box 2 / "QCD"** — fires when any Qualified Charitable Distribution applies (Internal Revenue Code §408(d)(8)). The IRS instructions tell the user to write "QCD" next to Line 4b.
  - **box 3 / "other write-in"** — fires when a Health Savings Account Funding Distribution applies (Internal Revenue Code §408(d)(9), label "HFD") OR the taxpayer has any other less-common IRA write-in code. `line4cBox3Text` carries the free-form text — typically `"HFD"` alone, sometimes `"HFD; XYZ"` when the taxpayer also has a separately-coded "other" write-in.
- **Exception 1 — Rollover** — Internal Revenue Code §408(d)(3). The taxpayer moved IRA money to another retirement account within 60 days. Triggers box 1.
- **Exception 2 — Form 8606 basis tracking** — Internal Revenue Code §72 pro-rata basis recovery applied via Form 8606 (Part I traditional / Part II conversion / Part III Roth). **Exception 2 has NO Line 4c box** because Form 8606 itself serves as the IRS-required disclosure attachment.
- **Exception 3 — Qualified Charitable Distribution** — Internal Revenue Code §408(d)(8). Direct transfer from a traditional IRA to a qualified charity for taxpayers age 70½ or older. Triggers box 2.
- **Exception 4 — Health Savings Account Funding Distribution** — Internal Revenue Code §408(d)(9). Once-in-a-lifetime direct transfer from an IRA to a Health Savings Account. Triggers box 3 with `"HFD"` text.
- **Breakout statement** — When **more than one Line 4 exception applies**, IRS rules require the taxpayer to attach a written statement explaining how each amount is attributable to each exception. The application emits the blocking flag `LINE4_EXCEPTION_BREAKOUT_STATEMENT_REQUIRED` until the user confirms (`exceptionBreakoutStatementPrepared = true`) that the statement is attached.
- **2025 waiver** — Per the 2025 IRS Form 1040 instructions: a breakout statement is **not required** when the only combination of exceptions is **Exception 2 (Form 8606) plus exactly one other exception**. Form 8606 is the breakdown in that case.
- **Married Filing Separately (MFS)** — Filing status. The application's MFS guard suppresses the spouse-side computation so spouse-attributed Line 4c facts cannot leak onto the taxpayer's return.

## 1. IRS rule for Line 4c (computed before reading the code)

### 1.1 Plain-English equation

Conceptually, Line 4c asks: "Why does Line 4a (gross) differ from Line 4b (taxable)? Disclose every exception that applies."

The three internal boxes are **independent** — each fires per a different Internal Revenue Code section. Up to all three can be true simultaneously. Each fires at the **return level** when any spouse on a joint return has the underlying exception fact.

```
box_1_rollover  =  taxpayer_had_any_rollover  OR  spouse_had_any_rollover

box_2_qcd       =  taxpayer_had_any_qualified_charitable_distribution
                   OR
                   spouse_had_any_qualified_charitable_distribution

box_3_other     =  taxpayer_had_health_savings_account_funding_distribution
                   OR  taxpayer_had_other_line_4c_write_in
                   OR  spouse_had_health_savings_account_funding_distribution
                   OR  spouse_had_other_line_4c_write_in

box_3_text      =  (each non-empty contributor joined with "; ")
                   where contributors are:
                     "HFD"  (literal, when a Health Savings Account Funding
                             Distribution applies to either spouse)
                     user-entered `line4cOtherWriteInText`  (when the user
                             explicitly entered an "other" write-in)
                   The composed text is built per person, then merged across
                   spouses with a dedup step (semicolon-separated set).
                   The composed string is null when no contributor is present.

Married Filing Separately suppresses the spouse path entirely (all four
spouse-side outputs are forced to false / null).
```

Line 4c itself uses **no monetary constants**. The 2025 waiver rule ("Exception 2 + exactly one other") is a Boolean policy, not a dollar threshold. The following constants are referenced by the related Line 4a / 4b cap-enforcement flags that fire alongside Line 4c on the same return but do not change Line 4c outputs themselves:

| Constant | Value (TY 2025) | Where it lives in code | IRS source |
|---|---|---|---|

| Qualified Charitable Distribution split-interest-entity lifetime cap (blocking advisory) | $54,000 per person | `ReferenceData.QCD_SIE_LIFETIME_CAP_PER_PERSON_2025` | Internal Revenue Code §408(d)(8)(F) |
| 60-day rollover window | 60 days | Not enforced in code (taxpayer self-attests) | Internal Revenue Code §408(d)(3)(A) |
| 2025 multi-exception breakout-statement waiver | "Exception 2 + exactly one other" | Encoded as the Path-B condition `line4ExceptionCategoryCount == 2 AND hasException2 AND nonException2Count == 1` at `TaxReturnComputeService.java:8027` | 2025 IRS Form 1040 instructions, "Lines 4a and 4b" |

### 1.4 Decision tree (which boxes fire and when)

```
For each person (taxpayer; spouse only when filing status ≠ Married Filing Separately):

    hasRollover    =  hadIraRollover  OR  totalIraRolloverAmount > 0
    hasQcd         =  hadQcd           OR  totalQcdAmount > 0
    hasHfd         =  hadHfd           OR  totalHfdAmount > 0
    hasOtherWrite  =  hasOtherLine4cWriteInCode == true
    hasBox3Other   =  hasHfd  OR  hasOtherWrite

    person.line4cBox3Text =
        if hasHfd:                "HFD"  +  (if hasText(otherText)) "; " + otherText
        else if hasText(otherText): otherText
        else:                      null

Aggregate across persons at the orchestrator:

    line_4c_box_1_rollover  =  taxpayer.hasRollover  OR  spouse.hasRollover
    line_4c_box_2_qcd       =  taxpayer.hasQcd       OR  spouse.hasQcd
    line_4c_box_3_other     =  taxpayer.hasBox3Other  OR  spouse.hasBox3Other
    line_4c_box_3_text      =  joinLine4cOtherText(taxpayer.line4cBox3Text,
                                                   spouse.line4cBox3Text)
        — preserve insertion order via LinkedHashSet
        — split spouse text by ";" before merging (dedup safety net)
        — join surviving tokens with "; "
        — return null when no tokens survive

(under Married Filing Separately, spouse-side flags are all false and spouse text is null,
 so the OR-aggregation reduces to taxpayer-only)

Breakout statement gate (governs the LINE4_EXCEPTION_BREAKOUT_STATEMENT_REQUIRED blocking flag,
 not the Line 4c output itself):

    line4ExceptionCategoryCount  =  box1?1:0 + box2?1:0 + box3?1:0 + hasException2?1:0
    nonException2Count           =  box1?1:0 + box2?1:0 + box3?1:0

    if user set moreThanOneLine4ExceptionAppliesOnReturn:
        statementRequired = not onlyException2AndOneOtherAppliesOnReturn   (user-asserted path)
    else:
        waiverCase = (line4ExceptionCategoryCount == 2 AND
                      hasException2 AND
                      nonException2Count == 1)
        statementRequired = (line4ExceptionCategoryCount > 1) AND not waiverCase   (auto-derived path)

    if statementRequired AND not exceptionBreakoutStatementPrepared:
        emit blocking flag LINE4_EXCEPTION_BREAKOUT_STATEMENT_REQUIRED
```

### 1.6 What MUST NOT contribute to Line 4c (explicit exclusions)

Per the IRS 2025 Form 1040 instructions and Publication 590-B:

| Item | Why excluded |
|---|---|
| 1099-R with `iraSepSimple` UNCHECKED (pension / annuity / 401(k) / 403(b)) | Lines 4a/4b/4c are Individual Retirement Account only by statute; non-Individual-Retirement-Account retirement plan disclosures use the parallel Line 5c boxes (PSO / Rollover / etc.) |
| Exception 2 (Form 8606 basis tracking) | Form 8606 IS the IRS-required attachment for basis-recovery cases; it serves as the disclosure that would otherwise require a Line 4c box |
| Form 8606 line 15c / 18 / 25c amounts | Line 4c records the **fact** that an exception applies, not the dollar amount; amounts live on Line 4a (gross) and Line 4b (taxable) |
| Married Filing Separately spouse-side facts | The spouse files separately; spouse's exception facts must not leak onto the taxpayer's Line 4c |
| Box 7 distribution codes alone | Box 7 affects Form 8606 generation and Line 4b computation; it does not itself trigger Line 4c box 1/2/3 (only the corresponding user-entered fact does) |
| The total Qualified Charitable Distribution amount that exceeds the $108,000 cap | The over-cap portion becomes taxable on Line 4b but does NOT create a separate Line 4c disclosure; only `hadQcd`/`totalQcdAmount > 0` triggers box 2 |
| A trustee-to-trustee transfer | Not a distribution; no 1099-R should be issued; no Line 4c box fires |

### 1.7 Worked examples

**Example A — Fully taxable simple case, all Line 4c boxes blank:**
- One IRA 1099-R: gross $5,000, taxable $5,000, code 7, no Roth codes.
- No rollover, no Qualified Charitable Distribution, no Health Savings Account Funding Distribution, no Form 8606 triggers, no other write-in.
- Outputs: box 1 = false; box 2 = false; box 3 = false; `line4cBox3Text` = null; no blocking flag; no advisory flags.

**Example B — Pure rollover, box 1 fires:**
- `hadIraRollover = true`, `totalIraRolloverAmount = $20,000`.
- Outputs: box 1 = **true**; box 2 = false; box 3 = false; `line4cBox3Text` = null.
- Breakout-statement gate: 1 exception category present → no statement required.
- Note: if the user also set `rolloverIntoQualifiedPlanOrCompletedInFollowingYear = true`, the blocking flag `IRA_ROLLOVER_ATTACHMENT_REVIEW` fires per `rules.md §18` (CP2000 risk).

**Example C — Pure Qualified Charitable Distribution, box 2 fires:**
- `hadQcd = true`, `totalQcdAmount = $50,000`.
- Outputs: box 1 = false; box 2 = **true**; box 3 = false; `line4cBox3Text` = null.

**Example D — Pure Health Savings Account Funding Distribution, box 3 fires with "HFD":**
- `hadHfd = true`, `totalHfdAmount = $5,000`. No `line4cOtherWriteInText`.
- Outputs: box 1 = false; box 2 = false; box 3 = **true**; `line4cBox3Text` = `"HFD"`.

**Example E — Health Savings Account Funding Distribution + user "other" write-in, box 3 text composes:**
- `hadHfd = true`, `hasOtherLine4cWriteInCode = true`, `line4cOtherWriteInText = "XYZ"`.
- Per the per-person composition rule: `"HFD" + "; " + "XYZ"` = `"HFD; XYZ"`.
- Outputs: box 3 = **true**; `line4cBox3Text` = `"HFD; XYZ"`.

**Example F — Multi-exception (rollover + Qualified Charitable Distribution + Health Savings Account Funding Distribution), breakout-statement gate fires:**
- `totalIraRolloverAmount = $10,000`, `totalQcdAmount = $10,000`, `totalHfdAmount = $5,000`.
- Outputs: box 1 = true; box 2 = true; box 3 = true; `line4cBox3Text` = `"HFD"`; `line4ExceptionCategoryCount = 3`; not waiver case.
- Blocking flag `LINE4_EXCEPTION_BREAKOUT_STATEMENT_REQUIRED` fires until `exceptionBreakoutStatementPrepared = true`.

**Example G — Exception 2 + one other (the 2025 waiver), no breakout statement needed:**
- `hasNondeductibleTraditionalBasis = true` (Exception 2 — Form 8606 fires) AND `totalIraRolloverAmount = $1,000` (Exception 1 — box 1 fires).
- `line4ExceptionCategoryCount = 2`; `hasException2 = true`; `nonException2Count = 1`; **waiver applies**.
- Outputs: box 1 = true; box 2 = false; box 3 = false; no breakout-statement flag (Form 8606 itself is the IRS-required disclosure).

**Example H — Married Filing Separately spouse-leakage guard:**
- Filing status: Married Filing Separately. Taxpayer has rollover; spouse has Qualified Charitable Distribution; both filed separate returns.
- On the taxpayer's return: only taxpayer's `hasRollover` contributes. Outputs: box 1 = **true**; box 2 = false; box 3 = false. Spouse's Qualified Charitable Distribution does NOT leak in.
- On the spouse's return: only spouse's `hasQcd` contributes. Outputs: box 1 = false; box 2 = **true**; box 3 = false.

**Example I — Both spouses have Health Savings Account Funding Distribution (dedup behavior):**
- Filing status: Married Filing Jointly. Taxpayer: `hadHfd = true`, no other write-in. Spouse: `hadHfd = true`, no other write-in.
- Per-person texts: taxpayer = `"HFD"`; spouse = `"HFD"`.
- Merge via `joinLine4cOtherText`: insert `"HFD"` from taxpayer → set = `{"HFD"}`; split spouse's `"HFD"` by `;` yields `["HFD"]` → add `"HFD"` → set unchanged. Result: `"HFD"`.
- Outputs: box 3 = true; `line4cBox3Text` = `"HFD"` (NOT `"HFD; HFD"`).

## 2. Code implementation (what the code actually does)

The Line 4c compute path lives in `TaxReturnComputeService.java`. Two methods participate (sharing the same per-person pipeline as Lines 4a and 4b):

| Method | Approximate location | Role for Line 4c |
|---|---|---|
| `computeIraDistributions()` | `:7910–8270` | Orchestrator. Runs the Married Filing Separately suppression, calls `computeIraForPerson` twice, OR-aggregates the three Boolean boxes across spouses, calls `joinLine4cOtherText` to merge the per-person write-in strings, derives `line4ExceptionCategoryCount` + `nonException2Count`, runs the two-path statement-required logic, emits the breakout-statement blocking flag when warranted, packages outputs onto `Income`. |
| `computeIraForPerson()` | `:8218–8514` | Per-person worker. Computes per-person `hasRollover`, `hasQcd`, `hasHfd`, `hasOtherLine4cWriteIn`, composes `line4cBox3Text` from `"HFD"` literal + user's `line4cOtherWriteInText`, returns these on the `IraPersonComputation` record. |
| `joinLine4cOtherText()` | `:12480–12496` | Private static helper. Asymmetric dedup: first string added whole; second string split on `";"` and each token added separately; LinkedHashSet preserves insertion order; tokens joined with `"; "`. Returns null when no tokens survive. |

### 2.1 Step-by-step code transcription

```
At computeIraForPerson:

    Read user-form Booleans (each false when iraForm is null per MFS guard):
        hasRollover           := boolean OR amount-positive  via :8282-8290
        hasQcd                := boolean OR amount-positive
        hasHfd                := boolean OR amount-positive
        hasOtherLine4cWriteIn := getBoolean(iraForm, "hasOtherLine4cWriteInCode")

    Compose per-person line4cBox3Text  (:8464-8471):
        line4cBox3Text := null
        if hasHfd:
            line4cBox3Text := "HFD"
        otherText := getString(iraForm, "line4cOtherWriteInText")
        if hasText(otherText):
            if hasText(line4cBox3Text):
                line4cBox3Text := line4cBox3Text + "; " + otherText
            else:
                line4cBox3Text := otherText

    Return IraPersonComputation(... hasRollover, hasQcd,
                                hasHfd || hasOtherLine4cWriteIn,  ← hasBox3Other
                                ..., line4cBox3Text, ...)

At computeIraDistributions (orchestrator) :7985-7989:

    line4cBox1   := taxpayer.hasRollover()    OR  spouseResult.hasRollover()
    line4cBox2   := taxpayer.hasQcd()         OR  spouseResult.hasQcd()
    line4cBox3   := taxpayer.hasBox3Other()   OR  spouseResult.hasBox3Other()
    hasException2:= taxpayer.hasException2()  OR  spouseResult.hasException2()
    line4cBox3Text := joinLine4cOtherText(taxpayer.line4cBox3Text(),
                                          spouseResult.line4cBox3Text())

Compute category counts :7991-8004:

    line4ExceptionCategoryCount := count of {box1, box2, box3, hasException2} that are true
    nonException2Count          := count of {box1, box2, box3}                that are true

Breakout-statement gate :8023-8038:

    if user explicitly set moreThanOneLine4ExceptionAppliesOnReturn:
        statementRequired := not user-set-onlyException2AndOneOtherAppliesOnReturn
    else:
        waiverCase := (line4ExceptionCategoryCount == 2  AND
                       hasException2  AND
                       nonException2Count == 1)
        statementRequired := line4ExceptionCategoryCount > 1  AND  not waiverCase

    if statementRequired AND not exceptionBreakoutStatementPrepared:
        flags.add(TaxReturnFlag("LINE4_EXCEPTION_BREAKOUT_STATEMENT_REQUIRED",
                                 "...attachment statement breaking out...",
                                 true   ← BLOCKING))

At joinLine4cOtherText (helper) :12480-12496:

    parts := new LinkedHashSet<String>()
    if hasText(first):
        parts.add(first.trim())                 ← whole-string insert
    if hasText(second):
        for each token in second.split(";"):
            if hasText(token):
                parts.add(token.trim())          ← per-token insert
    if parts.isEmpty():
        return null
    return String.join("; ", parts)
```

### 2.2 Married Filing Separately guard

Same single-guard cascade as Lines 4a and 4b. The orchestrator at `:7940` sets `iraIncomeSpouse = isMfsReturn ? null : iraIncomeSpouseRaw`. When `iraIncomeSpouse` is null, the spouse's `computeIraForPerson` reads all user-form Booleans as false (via `getBoolean(null, ...)`), so `spouseResult.hasRollover() = spouseResult.hasQcd() = spouseResult.hasBox3Other() = false` and `spouseResult.line4cBox3Text() = null`. The OR-aggregation at `:7985-7989` therefore reduces to taxpayer-only. Verified by lock-in test `mfsExcludesSpouseIraFromLine4a` (also exercises Line 4c boxes).

### 2.3 The three boxes are INDEPENDENT (NOT symmetric like Line 3c)

Per `:7979-7983` design note: Line 4c uses three independent triggers because each Internal Revenue Code exception is standalone. Contrast with Line 3c (child dividend disclosure) which uses two checkboxes from a **single** condition (`hasChildQualifiedDividends` feeds BOTH Line 3a and Line 3b symmetrically per Internal Revenue Code §1(h)(11)). The two design patterns are deliberately distinct — Line 4c boxes 1, 2, 3 correspond to Internal Revenue Code §408(d)(3), §408(d)(8), and §408(d)(9) respectively, with no shared trigger.

### 2.4 Box 3 text composition rule

Per `computeIraForPerson:8464-8471`:

1. Start with `line4cBox3Text = null`.
2. If `hasHfd = true`, set `line4cBox3Text = "HFD"`. (The IRS form requires the literal text "HFD" for the Health Savings Account Funding Distribution disclosure.)
3. Read the user's `line4cOtherWriteInText`. If present and non-empty:
   - If `"HFD"` is already in the string, append `"; " + userText`.
   - Otherwise, set the string to `userText` alone.
4. Result is the per-person `line4cBox3Text`, returned on `IraPersonComputation`.

Per-person results are then merged at the orchestrator via `joinLine4cOtherText`.

### 2.5 The asymmetric dedup design in `joinLine4cOtherText`

The merge function treats the two inputs asymmetrically by design (documented in the 4c #5 audit):

- **First argument** (typically taxpayer's composed text): added as a single token (no split).
- **Second argument** (typically spouse's composed text): split by `";"` and each piece added separately as its own token.

The LinkedHashSet provides set semantics (duplicates dropped) and preserves insertion order. The asymmetry is a known design choice — the both-spouses-`"HFD"` edge case (Example I above) deduplicates correctly because spouse's `"HFD"` is one of the split tokens. A pathological case where the user's per-person text already contains an embedded `";"` would compose taxpayer's whole string into one token and spouse's matching whole string into separate sub-tokens (potentially producing `"HFD; XYZ; HFD"` if both spouses entered `"HFD; XYZ"`); this edge case is documented as acceptable behavior given how rare both-spouses-with-same-multi-token-write-in is in practice.

### 2.6 Canonical zero-vs-null compliance

Each box is a Boolean — there is no "zero versus null" distinction for the three flags themselves (the `IraPersonComputation` record stores `boolean` primitives, defaulting to false). The composed `line4cBox3Text` is `String` (nullable) and follows the canonical convention: **null when no contributor, never empty-string**. PDF rendering treats a null text as absent (blank disclosure on Form 1040), whereas empty-string would render an unwanted whitespace gap.

The `Income` setters at `:6715-6718` accept the Boolean fields directly. The `line4cBox3Text` setter at `:6718` preserves null.

### 2.7 Two-path breakout-statement-required design

The application supports two paths because the auto-derived detection may miss edge cases the user can see:

- **Path A — user-asserted** (priority): when the user explicitly sets `moreThanOneLine4ExceptionAppliesOnReturn = true`, the code uses only the user's `onlyException2AndOneOtherAppliesOnReturn` waiver flag to decide whether the statement is required.
- **Path B — auto-derived** (fallback when user has not asserted): the code counts the exception categories from the computed boxes + Form 8606 trigger, applies the 2025 waiver rule, and decides automatically.

Per the 4c #7 audit, the user-asserted path overrides the auto-derived path even when the auto-derived path would draw a different conclusion. This is deliberate: the user has visible information (a less-common Line 4 write-in code that does not map to one of the boxes, for example) that the validator cannot see.

## 3. Comparison — does the code match the IRS rule?

### 3.1 Item-by-item matching

## 6. Common mistakes (Line 4c specifically)

- **Treating Line 4c as a single field instead of three independent triggers.** Each box corresponds to a distinct Internal Revenue Code exception; do not unify the logic into a single "any exception" Boolean.
- **Including Form 8606 (Exception 2) as a Line 4c box.** Form 8606 IS the IRS-required disclosure attachment for basis-tracking cases; it does NOT belong on Line 4c. The application correctly tracks `hasException2` as a separate signal that participates in the breakout-statement count but never becomes a box.
- **Forgetting the literal "HFD" text for Health Savings Account Funding Distribution.** The IRS instructions require the specific text "HFD" — the application hard-codes it at `:8466` rather than letting the user type a free-form string. (The user's separate `line4cOtherWriteInText` is appended only after "HFD".)
- **Producing empty-string instead of null for an empty composed text.** PDF rendering treats null as "blank" and empty-string as "whitespace gap" — the application correctly returns null at `:12492-12494` when no contributor is present.
- **Forgetting to dedup the both-spouses-Health-Savings-Account-Funding-Distribution case.** Without the LinkedHashSet dedup in `joinLine4cOtherText`, the composed text would be `"HFD; HFD"`. The application's helper deduplicates correctly.
- **Letting Line 4c outputs leak across spouses on Married Filing Separately.** The orchestrator's single-guard cascade (Lines 4a/4b/4c) at `:7940-7949` suppresses the spouse path entirely. Verified by lock-in test.
- **Forgetting the 2025 waiver in the breakout-statement gate.** Without the waiver, multi-exception returns with `Form 8606 + 1 other` would over-fire the blocking flag. The Path B encoding at `:8027` is exact.
- **Computing the statement-required gate from the boxes ALONE (excluding Exception 2).** The `line4ExceptionCategoryCount` at `:7991-8003` correctly INCLUDES `hasException2`; Exception 2 counts as an exception category for the breakout-statement gate even though it has no box.
- **Letting a Line 4c output feed Line 9 total income.** Line 4c is disclosure-only; only Line 4b (taxable amount) feeds Line 9. The application correctly omits Line 4c from the `totalIncome` aggregation.
