## Glossary

- **Form 1040 Line 5c** — Tax Year 2025 Form 1040 line that carries three disclosure checkboxes and a write-in text area attached to the pension/annuity reporting on Lines 5a and 5b. Line 5c does NOT carry a dollar amount itself; it is metadata that tells the IRS which special tax treatments applied to the pension distributions reported above.
- **Line 5c box 1 — Rollover** — Checked when any portion of the pension/annuity distribution shown on Lines 5a/5b was rolled over into another qualified plan or an Individual Retirement Account. Required disclosure when the Line 5b taxable amount has been reduced by a rollover.
- **Line 5c box 2 — Public Safety Officer (PSO) exclusion** — Checked when the taxpayer (or spouse on a joint return) is an eligible retired Public Safety Officer and has elected the Internal Revenue Code §402(l) premium exclusion of up to $3,000 per person. The election reduces the Line 5b taxable amount.
- **Line 5c box 3 — Other** — Checked when the taxpayer needs to enter a write-in code in the adjacent text field. IRS examples include certain special rollover treatments and other less common situations covered by IRS Publication 575.
- **Line 5c write-in text** — Free-text field next to the three checkboxes. Per IRS Publication 575 the user enters a short code or descriptive abbreviation when box 3 applies. Examples include "QCD" (for IRA distributions reported here in error, though that's normally a Line 4 marker), specific transaction codes from the source 1099-R, or other Publication 575-referenced abbreviations.
- **Per-return cardinality** — Form 1040 Line 5c is a SINGLE row of three checkboxes plus one write-in text field for the entire return, even on Married Filing Jointly. When both spouses have qualifying disclosures, the boxes are OR-aggregated (any spouse triggering the disclosure causes the box to be checked) and the text is concatenated.
- **Married Filing Separately (MFS)** — Filing status. The application's MFS guard suppresses the spouse-side pension reads so the spouse's rollover / PSO election / write-in text cannot leak onto the taxpayer's separate return.
- **`joinOtherWriteInTokens`** — Application helper that combines taxpayer and spouse write-in text strings into one space-separated string of unique tokens (`LinkedHashSet` preserves insertion order; duplicates are dropped). Shared with Line 4c for the same purpose.
- **`hasText`** — Application helper that returns true when a string is non-null and contains at least one non-whitespace character.

## 1. IRS rule for Line 5c (computed before reading the code)

### 1.1 Plain-English equation

Line 5c is three independent checkboxes plus one write-in text field. Each box is set when its specific trigger condition is met. The text is the user's free-text write-in for box 3.

```
Step 1 — per-person triggers
  For each person (taxpayer, spouse on a joint return):

    rollover_applies_for_person  =  TRUE  if  hadPensionRollover == TRUE
                                          OR  totalPensionRolloverAmount > 0

    pso_election_for_person      =  TRUE  if  isEligibleRetiredPublicSafetyOfficer == TRUE
                                          AND electsPsoPremiumExclusion == TRUE
                                          (double-gate — both must be true)

    write_in_text_for_person     =  pensionForm.line5cOtherWriteInText
                                     (the user-provided string; may be null/empty)

    write_in_qualifies_for_person =  TRUE  if  hasOtherLine5cWriteInCode == TRUE
                                          OR  hasText(write_in_text_for_person)
                                          OR  line5cOtherWriteInAmount > 0
                                     (used internally to flag this person as having an
                                      "exception" for the Line 5a blank-when-fully-taxable
                                      test; see Line 5a §1.4)

    write_in_text_returned_for_person =
                                     write_in_text_for_person  if  write_in_qualifies_for_person
                                     null                       otherwise
                                     (the per-person worker only returns the text when at
                                      least one of the three sub-conditions is met)

Step 2 — return-level aggregation (orchestrator)
  line_5c_box_1  =  rollover_applies_for_taxpayer   OR  rollover_applies_for_spouse
  line_5c_box_2  =  pso_election_for_taxpayer       OR  pso_election_for_spouse
  line_5c_text   =  joinOtherWriteInTokens(write_in_text_returned_for_taxpayer,
                                            write_in_text_returned_for_spouse)
                    (space-separated unique tokens; preserves insertion order;
                     empty/null operands ignored)
  line_5c_box_3  =  hasText(line_5c_text)

  On Married Filing Separately, the spouse-side per-person worker is suppressed by the
  MFS guard, so the spouse contributions are always false/null and the aggregations reduce
  to taxpayer-only.

Step 3 — render
  PDF rendering:
    box 1 prints a checkmark when line_5c_box_1 == TRUE
    box 2 prints a checkmark when line_5c_box_2 == TRUE
    box 3 prints a checkmark when line_5c_box_3 == TRUE
    text field prints the joined string when line_5c_box_3 == TRUE; empty otherwise

  JSON output stores all four fields under form1040.income; null/false values are
  omitted from the wire format when the field is unset.
```

### 1.3 Reference data — constants used in the equation

| Constant | Value (Tax Year 2025) | Where it lives in code | IRS source |
|---|---|---|---|
| Public Safety Officer annual exclusion cap | $3,000 per person | Hard-coded at `TaxReturnComputeService.java:9403` | Internal Revenue Code §402(l); IRS Publication 575 |

Line 5c itself has no monetary constants in the disclosure logic — the boxes are pure on/off based on whether the underlying trigger condition fires. The Public Safety Officer cap is referenced because box 2 firing implies the §402(l) exclusion has been applied at Line 5b under that cap.

### 1.4 Decision tree (when each Line 5c box is set)

```
Per-person worker (computePensionForPerson):

  hasRollover_for_person  =  pensionForm.hadPensionRollover == TRUE
                              OR  pensionForm.totalPensionRolloverAmount > 0

  hasPsoElection_for_person =  pensionForm.isEligibleRetiredPublicSafetyOfficer == TRUE
                               AND pensionForm.electsPsoPremiumExclusion == TRUE

  line5cBox3Text_for_person =  pensionForm.line5cOtherWriteInText   (may be null)

  hasOtherLine5cWriteIn_for_person =  pensionForm.hasOtherLine5cWriteInCode == TRUE
                                       OR  hasText(line5cBox3Text_for_person)
                                       OR  pensionForm.line5cOtherWriteInAmount > 0

  effective_text_returned_for_person =
                                  line5cBox3Text_for_person  if  hasOtherLine5cWriteIn_for_person
                                  null                        otherwise

  Per-person worker returns:
    hasRollover, hasPsoElection, hasAnyException (which includes hasOtherLine5cWriteIn),
    effective_text_returned_for_person  (named line5cBox3Text on the record)

Orchestrator (computePensionAnnuities):

  line_5c_box_1  =  taxpayer.hasRollover()       OR  spouseResult.hasRollover()
  line_5c_box_2  =  taxpayer.hasPsoElection()    OR  spouseResult.hasPsoElection()
  line_5c_text   =  joinOtherWriteInTokens(taxpayer.line5cBox3Text(),
                                            spouseResult.line5cBox3Text())
  line_5c_box_3  =  hasText(line_5c_text)

Final wiring at buildIncome:
  income.setLine5cBox1Rollover(line_5c_box_1)
  income.setLine5cBox2Pso(line_5c_box_2)
  income.setLine5cBox3Other(line_5c_box_3)
  income.setLine5cBox3Text(line_5c_text)
```

### 1.5 Symmetric write-in text joining (joinOtherWriteInTokens)

When both spouses provide write-in text, the orchestrator joins them with `joinOtherWriteInTokens`. The helper:

1. Splits each input by whitespace.
2. Adds each token to a `LinkedHashSet` (preserves insertion order, eliminates duplicates).
3. Joins the unique tokens back into one space-separated string.

This means:
- If taxpayer writes "CODEX" and spouse writes "CODEY" → joined = "CODEX CODEY"
- If both write the same token "CODEX" → joined = "CODEX" (deduplicated)
- If taxpayer is null and spouse writes "CODEY" → joined = "CODEY"
- The order matches the order of the operands (taxpayer first, then spouse).

Same helper is shared with Line 4c (Internal Revenue Account disclosure write-in). The symmetric tokenization fix (4c.md Gap 1) eliminated an earlier asymmetric quirk where the order depended on which operand was non-null first.

### 1.6 Outputs produced

| Output JSON path | Java field | Meaning |
|---|---|---|
| `form1040.income.line5cBox1Rollover` | `Income.line5cBox1Rollover` (Boolean) | Line 5c box 1 — true when any spouse has rollover |
| `form1040.income.line5cBox2Pso` | `Income.line5cBox2Pso` (Boolean) | Line 5c box 2 — true when any spouse has Public Safety Officer election |
| `form1040.income.line5cBox3Other` | `Income.line5cBox3Other` (Boolean) | Line 5c box 3 — true when the joined write-in text is non-empty |

### 1.7 What MUST NOT trigger a Line 5c box (explicit exclusions)

| Item | Why NOT a Line 5c trigger |
|---|---|
| Form 5329 early-distribution additional tax | Form 5329 is a separate penalty; its presence is disclosed on Schedule 2 and Form 1040 Line 23, not Line 5c |
| Form 4972 lump-sum distribution election | Form 4972 is disclosed by the "Form 4972" write-in on Form 1040 Line 16 box 2, not Line 5c |
| Simplified Method or General Rule basis recovery | Per IRS Publication 575, basis recovery is reflected directly in the Line 5b taxable amount; it does NOT trigger a separate Line 5c box |
| Disability pension routed to Line 1h | Such entries are excluded from the per-person pension worker entirely; they do not flow through the Line 5c logic |
| Individual Retirement Account distributions (Line 4 path) | IRA distributions report on Lines 4a/4b/4c; their disclosures use the Line 4c boxes (different rollover/Qualified Charitable Distribution/Health Funding Distribution semantics), not Line 5c |
| Form RRB-1099-R rollover (not applicable) | Per IRS rules railroad-retirement distributions are not qualified-plan rollover-eligible, so they cannot trigger Line 5c box 1 |

### 1.8 Worked examples

**Example A — All three boxes triggered (single taxpayer):**
- Pension form: `hadPensionRollover=true`, `isEligibleRetiredPublicSafetyOfficer=true`, `electsPsoPremiumExclusion=true`, `line5cOtherWriteInText="CODEX"`
- Per the equation: box 1 = true (rollover); box 2 = true (PSO double-gate); box 3 = true (text "CODEX" present); text = "CODEX"

**Example B — Married Filing Jointly with both spouses contributing different write-ins:**
- Taxpayer: `line5cOtherWriteInText="CODEX"`
- Spouse: `line5cOtherWriteInText="CODEY"`
- Per the equation: box 3 = true; text = "CODEX CODEY" (joined, unique tokens)

**Example C — Married Filing Jointly with duplicate write-ins:**
- Taxpayer: `line5cOtherWriteInText="QCD"`
- Spouse: `line5cOtherWriteInText="QCD"`
- Per the equation: box 3 = true; text = "QCD" (LinkedHashSet deduplicates)

**Example D — PSO elected but no pension activity:**
- 1099-R: none seeded
- Pension form: `isEligibleRetiredPublicSafetyOfficer=true`, `electsPsoPremiumExclusion=true`, `totalQualifyingPsoPremiumsPaid=$2,000`
- Per the equation: box 2 = true (PSO double-gate fires regardless of pension activity — see §3.2 Gap 1 discussion)
- Line 5a = null, Line 5b = null (no pension activity), but Line 5c box 2 = true

**Example E — Rollover flag set but no rollover amount:**
- Pension form: `hadPensionRollover=true`, `totalPensionRolloverAmount=null`
- Per the equation: box 1 = true (flag alone triggers); Stage 3 of the Line 5b chain has nothing to subtract; Line 5b unchanged

**Example F — Married Filing Separately spouse-leakage guard:**
- MFS
- Taxpayer: no rollover, no PSO, no write-in
- Spouse: `hadPensionRollover=true`, `electsPsoPremiumExclusion=true`, `line5cOtherWriteInText="CODEX"`
- MFS guard suppresses spouse-side reads → all spouse contributions = false/null
- Line 5c box 1 = false, box 2 = false, box 3 = false, text = null

**Example G — Box 3 fires from text alone (no flag, no amount):**
- Pension form: `hasOtherLine5cWriteInCode=null`, `line5cOtherWriteInText="QCD"`, `line5cOtherWriteInAmount=null`
- Per-person worker: `hasOtherLine5cWriteIn = false OR hasText("QCD")=true OR false → true`
- `line5cBox3Text returned = "QCD"`
- Orchestrator: text = "QCD", box 3 = true

**Example H — Flag set but text empty (silent no-disclosure case):**
- Pension form: `hasOtherLine5cWriteInCode=true`, `line5cOtherWriteInText=null`, `line5cOtherWriteInAmount=null`
- Per-person worker: `hasOtherLine5cWriteIn = true` (the flag alone qualifies) → returns `line5cBox3Text=null`
- Orchestrator: text = null, box 3 = false
- See §3.2 Gap 3 — the user set the flag but provided no text; the form shows no box 3 disclosure.

## 2. Code implementation (what the code actually does)

The Line 5c compute path lives in `TaxReturnComputeService.java`. Three methods participate:

| Method | Approximate location | Role for Line 5c |
|---|---|---|
| `computePensionForPerson()` | `:9136-9419` | Per-person worker. Reads the pension form's rollover/PSO/write-in fields; sets `hasRollover`, `hasPsoElection`, and `effective line5cBox3Text` on the returned record. |
| `computePensionAnnuities()` | `:8908-9120` | Orchestrator. OR-aggregates `hasRollover` and `hasPsoElection` across spouses; joins write-in text via `joinOtherWriteInTokens`; derives box 3 from `hasText(line5cText)`. |
| `buildIncome()` | `:6751-6754` | Final wiring. Calls `income.setLine5cBox1Rollover/...Box2Pso/...Box3Other/...Box3Text` on every compute (these are simple Boolean/String setters with no null gating). |

### 2.1 Step-by-step code transcription

```
At computePensionForPerson (per-person):

    hasRollover = Boolean.TRUE.equals(getBoolean(pensionForm, "hadPensionRollover"))
                  || hasPositiveAmount(getAmount(pensionForm, "totalPensionRolloverAmount"))

    isEligiblePso = Boolean.TRUE.equals(getBoolean(pensionForm, "isEligibleRetiredPublicSafetyOfficer"))
    electsPso     = Boolean.TRUE.equals(getBoolean(pensionForm, "electsPsoPremiumExclusion"))
    hasPsoElection = isEligiblePso && electsPso

    line5cBox3Text_raw = getString(pensionForm, "line5cOtherWriteInText")
    hasOtherLine5cWriteIn =
            Boolean.TRUE.equals(getBoolean(pensionForm, "hasOtherLine5cWriteInCode"))
        ||  hasText(line5cBox3Text_raw)
        ||  hasPositiveAmount(getAmount(pensionForm, "line5cOtherWriteInAmount"))

    hasAnyException = hasRollover
        || hasPsoElection
        || hasOtherLine5cWriteIn
        || hasTaxableNotDetermined
        || needsSimplifiedMethod
        || needsGeneralRule

    return PensionPersonComputation(
        ... gross/taxable fields ...,
        hasRollover,                                                  // → orchestrator box 1
        hasPsoElection,                                               // → orchestrator box 2
        hasAnyException,                                              // → Line 5a blank-rule
        hasOtherLine5cWriteIn ? line5cBox3Text_raw : null,            // → orchestrator text
        ... early-distribution + entry-count fields ...
    )

At computePensionAnnuities (orchestrator), after both per-person workers run:

    line5cBox1 = taxpayer.hasRollover() || spouseResult.hasRollover()
    line5cBox2 = taxpayer.hasPsoElection() || spouseResult.hasPsoElection()
    line5cText = joinOtherWriteInTokens(taxpayer.line5cBox3Text(),
                                         spouseResult.line5cBox3Text())
    line5cBox3 = hasText(line5cText)

At buildIncome (final wiring):

    if (pension != null) {
        income.setLine5cBox1Rollover(pension.line5cBox1Rollover());
        income.setLine5cBox2Pso(pension.line5cBox2Pso());
        income.setLine5cBox3Other(pension.line5cBox3Other());
        income.setLine5cBox3Text(pension.line5cBox3Text());
    }
```

### 2.2 Married Filing Separately guard

The orchestrator at `:8910` sets `pensionIncomeSpouse = isMfsReturn ? null : pensionIncomeSpouseRaw` and `spouseSsn = isMfsReturn ? null : normalizeSsn(spouse.ssn)`. When `pensionIncomeSpouse` is null:
- `spouseResult.hasRollover()` returns `false` (the per-person worker initializes the field to false when the form is null and no entries attribute via SSN fallback).
- `spouseResult.hasPsoElection()` returns `false`.
- `spouseResult.line5cBox3Text()` returns `null`.

Net effect: orchestrator aggregation reduces to taxpayer-only, and Line 5c reflects taxpayer-only disclosure.

### 2.3 Design difference vs Line 4c

Line 4c (Internal Revenue Account disclosure) uses a slightly different box-3 trigger than Line 5c:
- **Line 4c** box 3 fires on the orchestrator-computed `hasBox3Other` boolean — an explicit OR across `taxpayer.hasBox3Other()` and `spouseResult.hasBox3Other()`. This is necessary because Line 4c has the Health Funding Distribution auto-prepend to its write-in text, which means the text can appear independently of whether the user explicitly set a write-in. The text-independent box-3 trigger preserves disclosure even when the user's per-person `hasOtherLine4cWriteInCode` flag is false but the auto-prepend has injected text.
- **Line 5c** has no auto-prepend (Public Safety Officer is disclosed via box 2, not text). So the simpler text-derived `box3 = hasText(line5cText)` trigger is equivalent and avoids the need for a separate orchestrator-computed boolean.

Both designs are IRS-correct for their respective lines. See `XLS/Computations/4c.md` for the Line 4c contrast.

### 2.4 No multi-exception breakout-statement requirement

Unlike Line 4c (which has a breakout-statement requirement when more than one IRA exception applies — see `XLS/Computations/4c.md` §1.7), Line 5c has no equivalent multi-exception statement requirement. Per IRS 2025 pension rules, each Line 5c box is disclosed via its own checkbox; no consolidated attachment is needed when multiple boxes apply. The code does not (and should not) emit a `LINE5_EXCEPTION_BREAKOUT_STATEMENT_REQUIRED` flag.

### 2.5 Boolean wire format

`Income.line5cBox1Rollover`, `Income.line5cBox2Pso`, and `Income.line5cBox3Other` are typed `Boolean` (nullable) but the application always sets them to a non-null `Boolean.TRUE` or `Boolean.FALSE` via the orchestrator's primitive-boolean computation. The PDF renderer prints a checkmark when the value is `true`; otherwise the box is unmarked.

`Income.line5cBox3Text` is `String` (nullable). When the joined text is empty, `line5cBox3 = false` and `line5cBox3Text` is the empty string (or null after `joinOtherWriteInTokens` returns null for empty inputs). The PDF renderer prints the string in the adjacent text field when box 3 is checked.

## 3. Comparison — does the code match the IRS rule?

### 3.1 Item-by-item matching

| IRS rule (§1) | Code (§2) | Verdict |
|---|---|---|
| Line 5c box 1 (rollover) fires when any rollover applies | `line5cBox1 = taxpayer.hasRollover() || spouseResult.hasRollover()` at `:9008`; per-person `hasRollover = hadPensionRollover OR totalPensionRolloverAmount > 0` at `:9355-9356` | **Match** |
| Line 5c box 2 (Public Safety Officer) fires when any spouse has eligibility + election | `line5cBox2 = taxpayer.hasPsoElection() || spouseResult.hasPsoElection()` at `:9009`; per-person `hasPsoElection = isEligiblePso && electsPso` at `:9401` | **Match** — double-gate verified |
| Line 5c box 3 (other) fires when joined write-in text is non-empty | `line5cBox3 = hasText(line5cText)` at `:9011` | **Match** |
| Line 5c write-in text is joined across spouses with deduplication | `joinOtherWriteInTokens(taxpayer.line5cBox3Text(), spouseResult.line5cBox3Text())` at `:9010` | **Match** — symmetric tokenization via shared helper |
| Married Filing Separately suppresses spouse-side contributions | Single-guard cascade at `:8910` + spouse SSN nullification | **Match** |
| Disability-routed entries excluded from Line 5c | The per-person worker skips disability-routed entries via `entryIdsRoutedToLine1h`; rollover/Public Safety Officer/write-in are read from the PENSION FORM (not the 1099-R), so disability routing doesn't affect them anyway — but the per-person worker doesn't run if the pension form is null on MFS spouse side | **Match** |
| Public Safety Officer cap ($3,000) does not affect Line 5c (it's a Line 5b mechanic) | Line 5c box 2 fires on the election alone; the $3,000 cap is applied at Stage 4 of the Line 5b chain | **Match** |
| Line 5c does not contribute to Line 9 | Line 9 aggregation at `:6620-6622` includes line5b, not line5c | **Match** — Line 5c is metadata-only |

## 6. Common mistakes (Line 5c specifically)

- **Forgetting to check box 1 when a rollover is partial.** Per IRS Form 1040 Line 5c box 1 instructions, the box must be checked when ANY portion of the distribution was rolled over, not only when the entire distribution was rolled over. The application's `hasRollover = hadPensionRollover OR totalPensionRolloverAmount > 0` is correct.
- **Forgetting the Public Safety Officer double-gate.** Both `isEligibleRetiredPublicSafetyOfficer` AND `electsPsoPremiumExclusion` must be true for box 2 to fire and the §402(l) exclusion to apply at Line 5b. Eligibility alone (without affirmative election) does NOT trigger box 2.
- **Letting spouse Line 5c contributions leak onto the taxpayer's separate return on Married Filing Separately.** The single-guard cascade at the orchestrator prevents this.
- **Putting a Form 5329 early-distribution disclosure on Line 5c.** Form 5329 is its own form; the additional tax flows to Schedule 2 line 8 and Form 1040 line 23. Line 5c is not the disclosure path for Form 5329.
- **Putting a Form 4972 lump-sum election disclosure on Line 5c.** Form 4972 is disclosed via the "Form 4972" write-in on Form 1040 Line 16 box 2 — NOT Line 5c.
- **Marking box 3 without providing write-in text.** The IRS Form 1040 Line 5c box 3 expects an adjacent code/text. The application's design correctly suppresses box 3 when text is empty even if the `hasOtherLine5cWriteInCode` flag is set — but this can be confusing to users; see Gap 3 above.
- **Letting the same write-in token appear twice when both spouses entered it.** The `joinOtherWriteInTokens` helper deduplicates via `LinkedHashSet`; if you see duplicates on the printed form, the helper has a bug.
- **Confusing Line 5c box 1 (pension rollover) with Line 4c box 1 (Internal Revenue Account rollover).** They are independent disclosures — Line 4c is for IRA distributions, Line 5c is for pension/annuity distributions. The application's mirror filter at the per-person worker correctly routes each Form 1099-R to exactly one path.
