# Form 1040 Line 8 — Other Income from Schedule 1, line 10

**Tax year:** 2025
**Form 1040 label:** "Other income from Schedule 1, line 10"
**Author of this file:** generated from IRS publications + reading the code in `TaxReturnComputeService.java`. The goal is to compare the IRS rule against what the code actually does and call out every gap, mistake, or missing field.

This document is organized so a reader with basic arithmetic and programming knowledge can follow it. Abbreviations are expanded in parentheses the first time each is used.

## 1. The IRS computation in plain English

### 1.1 The headline rule

Form 1040 line 8 is **always exactly the amount on Schedule 1 (Form 1040), line 10**. Nothing on Form 1040 itself adds, subtracts, or otherwise changes that number. It is a straight copy.

```
Form 1040 line 8  =  Schedule 1 line 10
```

So to figure out Form 1040 line 8 the work is really to figure out Schedule 1 line 10.

### 1.2 Schedule 1 line 10 in plain English

Schedule 1 has two parts. Part I (lines 1 through 10) collects extra kinds of income that do not appear on Form 1040 lines 1 through 7. Line 10 is the total of Part I.

In plain English:

> Take everything on Part I lines 1 through 7, then add the grand total of all the "other income" sub-lines that sit underneath line 8 (lines 8a, 8b, 8c, ... up to 8z). The result is line 10.

```
Schedule 1 line 10  =  line 1
                    +  line 2a           (line 2b is the agreement date, not an amount)
                    +  line 3            (Schedule C net business income or loss)
                    +  line 4            (other gains or losses)
                    +  line 5            (Schedule E rental / royalty / partnership / S-corp / trust)
                    +  line 6            (Schedule F net farm income or loss)
                    +  line 7            (unemployment compensation, net of any 2025 repayment)
                    +  Schedule 1 line 9 (the grand total of all line-8 sub-lines)
```

And:

```
Schedule 1 line 9   =  line 8a + line 8b + line 8c + line 8d + line 8e + line 8f
                    +  line 8g + line 8h + line 8i + line 8j + line 8k + line 8l
                    +  line 8m + line 8n + line 8o + line 8p + line 8q + line 8r
                    +  line 8s + line 8t + line 8u + line 8v + line 8z
```

The 2025 Schedule 1 form does not have lines 8w, 8x, or 8y — the lettering jumps from 8v directly to 8z. Line 8z is the catch-all write-in.

### 1.4 The top-of-Schedule-1 Form 1099-K entry space

The 2025 Schedule 1 has a **separate entry space at the very top** (before line 1) for Form 1099-K amounts that were either:

- reported in error, or
- received for the sale of personal items at a loss.

This top entry space is a **disclosure-only matching field**. It is **not** part of the line 9 sum and not part of the line 10 sum. The IRS uses it to reconcile their copy of the Form 1099-K to the return; the amount itself does not affect tax.

### 1.6 Validation rules the IRS effectively requires

1. `Schedule 1 line 9` must equal the sum of lines 8a through 8z.
2. `Schedule 1 line 10` must equal lines 1 + 2a + 3 + 4 + 5 + 6 + 7 + line 9.
3. `Form 1040 line 8` must equal Schedule 1 line 10 exactly.
4. If line 2a is non-zero then line 2b (the agreement date) must be filled in.
5. If line 4 is non-zero then either Form 4797 or Form 4684 must be filed.
6. If line 5 is non-zero then Schedule E must be filed.
7. If line 8d is non-zero then Form 2555 must be filed.
8. If line 8e is non-zero then Form 8853 must be filed.
9. If line 8f is non-zero then Form 8889 must be filed.
10. If line 7 has been reduced for a repaid 2025 overpayment then the line 7 repayment checkbox must be checked and the repayment amount filled in.
11. Schedule 1 must be filed if any Part I line is used, OR any Part II line is used, OR the top-of-form Form 1099-K disclosure space is filled in — even if Form 1040 line 8 ends up at zero.

## 2. The computation as the code actually performs it

**Entry point in code:** `TaxReturnComputeService.java`, method `computeOtherIncomes(...)` at approximately line **12771**.

**Wire-up:** the result of `computeOtherIncomes` (a record called `OtherIncomeComputation`) carries `line8FromSchedule1` — that value is then copied to `Form1040.Income.otherIncomeSchedule1` inside `buildIncome(...)` (around line 6607). `applySchedule1AdditionalIncome(...)` at line 16862 copies the per-sub-line numbers into the Schedule 1 output model so they can be rendered on the PDF.

### 2.1 The code's overall formula

The code computes the same shape the IRS specifies:

```
schedule1Line10 = line1 + line2a + line3 + line4 + line5 + line6 + line7 + schedule1Line9
schedule1Line9  = line8a + line8b + line8c + line8d + line8e + line8f
                + line8g + line8h + line8i + line8j + line8k + line8l
                + line8m + line8n + line8o + line8p + line8q + line8r
                + line8s + line8t + line8u + line8v + line8z
form1040Line8   = schedule1Line10
```

It does this with two helpers:

- `addNonNull(a, b)` — preserves nulls: `null + null = null`, `null + 5 = 5`. Used inside `line10` so that when NO Schedule 1 input exists at all, `line10` stays `null` and the orchestrator returns no Schedule 1.
- `sumAmounts(...)` — treats nulls as zero: used inside `line9` so that an isolated line 8 sub-line works even if the others are null.

### 2.3 The Married Filing Separately guard

At the very top of `computeOtherIncomes(...)` the code nulls out all spouse-side inputs on a Married Filing Separately return:

```java
Map<String, Object> otherIncomeSpouse =
    isMfsReturn ? null : otherIncomeSpouseRaw;
BigDecimal employmentFormInmateWagesSpouse =
    isMfsReturn ? null : employmentFormInmateWagesSpouseRaw;
```

This single guard cascades to every sub-line above, so Married-Filing-Separately taxpayers never see leaked spouse Schedule 1 amounts on their own return. A unit test (`mfsExcludesSpouseOtherIncomeFromLine8`) locks this in.

### 2.4 The "no Schedule 1 needed" short-circuit

If every Part I input is null AND there are no Form 8814 items AND no W-2 box 11 amounts AND no employment-form inmate wages, the orchestrator returns `null` from `computeOtherIncomes` — Schedule 1 is then not generated and Form 1040 line 8 is left null (rendered as blank, not zero).

## 4. Mistakes, shortcomings, and missing fields

The numbered list below ranks gaps from highest to lowest impact on a typical return.

### 4.1 ✅ CLOSED 2026-06-07 — Tax-benefit-rule advisory added (full worksheet deferred)

**IRS rule.** Publication 525 ("Taxable and Nontaxable Income"), and the worksheet inside the 2025 Form 1040 instructions for Schedule 1 line 1, work through a tax-benefit-rule analysis. Roughly:

- If the taxpayer did **not** itemize last year, the whole refund is non-taxable; line 1 is zero.
- If the taxpayer itemized last year but elected to deduct **general sales tax** instead of state and local income tax, line 1 is zero.
- If the taxpayer itemized last year and was subject to the State And Local Tax (SALT) deduction cap of $10,000 ($5,000 if married filing separately), only the portion that actually reduced last year's tax bill is taxable now.

**Code behavior.** The user is asked to enter the already-taxable portion directly into `taxableStateLocalRefundsLine1`. The code does no worksheet.

**Impact.** The user (or their preparer) must do the worksheet manually. If they enter the full Form 1099-G box 2 amount without the worksheet adjustment, line 1 will be over-stated.

**Missing inputs that would let the code do this worksheet:**
- last year's itemized-vs-standard election
- last year's State And Local Tax deduction line breakdown (income tax vs. sales tax election)
- last year's SALT cap excess
- last year's tax before vs. after the SALT deduction

**Suggested status.** New non-blocking advisory flag: when `taxableStateLocalRefundsLine1` is non-zero but the user did not itemize in the prior year, prompt them to confirm.

**Fix applied 2026-06-07:** Per-side non-blocking advisory `OTHER_INCOME_LINE1_VERIFY_TAX_BENEFIT_RULE_{TAXPAYER,SPOUSE}` fires whenever line 1 is non-zero (either via user-fill `taxableStateLocalRefundsLine1` or via uploaded 1099-G box 2). The message walks the user through the three Pub. 525 exit cases: (a) did not itemize last year → refund non-taxable; (b) itemized with sales-tax election → refund non-taxable; (c) itemized with SALT cap → only the portion that actually reduced last year's tax is taxable. ### 4.5 Verified correct — Form 8814 line 12 routing to Schedule 1 (with one remaining PDF-rendering question)

**Original concern.** An earlier draft of this document worried that adding Form 8814 line 12 to Schedule 1 line 10 might double-count amounts that already flow to Form 1040 lines 2b, 3b, or 7. **That concern was wrong** — after tracing the code on 2026-06-07 the routing matches the 2025 IRS Form 8814 instructions exactly.

**The actual IRS Form 8814 routing.** For each child elected on Form 8814, the child's gross income (line 4 = line 1a interest + line 2a ordinary dividends + line 3 capital-gain distributions) is split as follows:

1. **First $2,700** (`ReferenceData.FORM8814_BASE_AMOUNT`) is "absorbed" by the form and produces **Form 8814 line 15** — an extra tax (0% / 10% / capped at `FORM8814_EXTRA_TAX_MAX`) that flows to **Form 1040 line 16** with the box-1 checkbox.
2. **Income above $2,700** (Form 8814 line 6 = line 4 − $2,700) is split proportionally three ways:
   - **Form 8814 line 9** = `line 6 × (line 2b qualified dividends ÷ line 4)` → routes to the parent's **Form 1040 line 3a (qualified dividends)** AND to **line 3b (ordinary dividends)** (symmetric addition, because qualified dividends are a subset of ordinary dividends — they show up on both totals).
   - **Form 8814 line 10** = `line 6 × (line 3 capital-gain distributions ÷ line 4)` → routes to the parent's **Form 1040 line 7a** (the capital-gain line, or Schedule D line 13 if Schedule D is filed).
   - **Form 8814 line 12** = `line 6 − line 9 − line 10` (the remainder — child's interest + non-qualified ordinary-dividend portion) → routes to the parent's **Schedule 1 line 8z** with description "Form 8814". The 2025 Form 8814 instruction for line 12 literally reads: "Include this amount in the total on Schedule 1 (Form 1040), line 8z."

So Form 8814 line 12 **does** flow to Schedule 1, specifically to line 8z. The original gap was based on a misreading.

**Code behavior — verified.** The 3-way split is implemented correctly:
- `TaxReturnComputeService.java:11377` — `line12 = line6 − line9 − line10` matches the IRS formula.
- `TaxReturnComputeService.java:7873-7895` — `form8814QualifiedDividendsTotal` (Form 8814 line 9, summed across all child forms) is added symmetrically to both parent's line 3a and line 3b inside `computeDividendIncome(...)`.
- `TaxReturnComputeService.java:10845-10865` — `form8814Line10Total` (Form 8814 line 10) is added to parent's line 7a.
- `TaxReturnComputeService.java:11398` — `applyForm8814Line12ToSchedule1(...)` adds each child's Form 8814 line 12 to Schedule 1's `otherIncomeForm8814` field and then bumps `totalOtherIncome` (Schedule 1 line 9) and `totalAdditionalIncome` (Schedule 1 line 10) by the same amount.
- `TaxReturnComputeService.java:2848` — Form 8814 line 15 extra tax is summed across child forms and added to Form 1040 line 16 with the box-1 checkbox.

There is **no** double-count.

**PDF rendering — verified correct on 2026-06-07.** Cross-checked the Schedule 1 Angular component at `C:\us-tax\us-tax-ui\src\app\forms\form-tax-return-schedule1.component.ts:802-818` (function `composeLine8z(schedule)`):

```typescript
private composeLine8z(schedule: Schedule1View): { description: string; amount: number } {
  const inc = schedule.additionalIncome ?? {};
```

## 7. Plain-English summary for a non-technical reader

1. **Form 1040 line 8 is just a copy.** It is the same number as Schedule 1 line 10.
2. **Schedule 1 line 10 is one big sum** of eight other numbers: lines 1, 2a, 3, 4, 5, 6, 7, and the sum of all the line-8 sub-lines (which itself is called line 9).
3. **Some sub-lines are negative on purpose** (8a Net Operating Loss, 8d Foreign Earned Income Exclusion, 8s Medicaid waiver offset). The code stores all three as negatives, regardless of the sign the user enters, by taking the absolute value and then flipping the sign.
4. **The code's biggest open questions** are: (a) whether the Olympic-medal exclusion on Schedule 1 line 24c is being computed automatically when the user has Olympic income on line 8m (Section 4.2), and (b) whether the printed Schedule 1 PDF shows the Form 1099-K disclosure (Section 4.6). Form 8814 line 12 routing was suspected to be a double-count in an earlier draft, but **verification confirmed BOTH the arithmetic AND the Portable Document Format rendering are correct** (Section 4.5).
5. **For everything else**, the code's math matches the IRS rule. Married Filing Separately data leakage is guarded against by a single null-out at the top of `computeOtherIncomes`. The validation flags catch missing-statement situations and out-of-scope schedules.

If those three open questions are resolved (or confirmed not to be bugs), the Line 8 / Schedule 1 Part I path produces a defensible return for the supported scenarios.
