# 1) Line identities (2025 Form 1040)

- **Line 11a**: `Adjusted gross income (AGI)`
- **Line 11b**: `Amount from line 11a (adjusted gross income)`

Structural placement:
- **line 11a** appears at the bottom of page 1 and is the actual AGI computation line;
- **line 11b** appears at the top of page 2 and is a direct copy of that AGI amount for downstream tax-and-credit computations.

Some generic IRS pages still describe AGI as being on "Form 1040 line 11." On the 2025 form, that AGI amount is shown on **line 11a** and then repeated on **line 11b**.

# 3) Backend wiring

## 3.1 Computation site

Lines 11a and 11b are computed inline inside `buildAdjustments` at `TaxReturnComputeService.java:7703-7732`:

```java
BigDecimal line10 = roundMoney(incomeAdjustments == null ? null
    : incomeAdjustments.line10FromSchedule1Line26());
BigDecimal line9 = income == null ? null : roundMoney(income.getTotalIncome());
// Form 1040 line 11a is a pure compute line: AGI = line 9 minus line 10.
// The 2025 form does not force zero-flooring here, so preserve negative results.
BigDecimal line11a = line9 == null ? null
    : roundMoney(subtractNonNegativeAllowNegative(line9, line10));
// Line 11b is an IRS copy line that repeats the AGI from line 11a.
BigDecimal line11b = line11a;
if (line10 == null && line11a == null) {
    return null;
}
Adjustments adjustments = new Adjustments();
adjustments.setAdjustmentsSchedule1(line10);
adjustments.setLine11aAdjustedGrossIncome(line11a);
adjustments.setLine11bAmountFromLine11aAdjustedGrossIncome(line11b);
adjustments.setAdjustedGrossIncome(line11a);   // legacy alias
```

Key implementation properties:
- `subtractNonNegativeAllowNegative` (NOT `subtractNonNegative`) — the method name encodes the rule that the result is **signed**;
- `BigDecimal line11b = line11a` — **same reference**, not a value copy, so the IRS copy-line invariant cannot drift;
- When `line10` and `line11a` are both null, the entire Adjustments object is suppressed so the Form 1040 cells render blank rather than $0;
- The legacy `adjustedGrossIncome` alias is set to the same value for backward compatibility with downstream callers (Form 6251 AMT, Schedule A) that still read `getAdjustedGrossIncome()`.

## 3.2 Form 8814 augmentation sync

When the Form 8814 augmentation increments Schedule 1 line 10, the augmentation site also repoints **both** `line11aAdjustedGrossIncome` and `line11bAmountFromLine11aAdjustedGrossIncome` to the new AGI value (`TaxReturnComputeService.java:968-970`). This keeps the page-2 carryforward in lockstep with the page-1 figure.

## 3.3 No standalone MFS guard

Lines 11a / 11b have no orchestrator. They inherit MFS protection from the 14 upstream feeders (13 income orchestrators feeding line 9 plus `computeIncomeAdjustments` for line 10). On MFS, both line 9 and line 10 are taxpayer-only, so line 11a / 11b are taxpayer-only by construction.

# 5) Critical AGI vs. MAGI distinction

This is the most important implementation rule for these lines.

## 5.1 What lines 11a and 11b are

- **Line 11a** is the taxpayer's AGI.
- **Line 11b** is the exact same AGI amount copied forward.

## 5.2 What lines 11a and 11b are **not**

- Line 11b is **not** "the taxpayer's MAGI."
- The IRS uses different MAGI definitions for different deductions, credits, and worksheets.
- In many places, the IRS starts with line 11b and then adds back specific excluded or deducted items.

## 5.3 Examples of 2025 downstream uses

- **Schedule 1-A, Part I** starts with line 11b, then adds back Puerto Rico excluded income, **Form 2555 line 45**, **Form 2555 line 50**, and **Form 4563 line 15** to compute its MAGI.
- **Schedule A** state-and-local-sales-tax worksheet uses line 11b plus certain nontaxable items as an income measure.
- IRA deduction, student loan interest, education benefits, and credits each define MAGI differently.

Implementation consequence:
- store **11a / 11b as AGI only**;
- compute MAGI **separately for each downstream rule** instead of creating one global `magi` field from line 11b.

Verification: `grep magi/MAGI` in `Adjustments.java` returns no match — the backend complies with this rule.

# 6) Downstream dependencies that matter in 2025

## 6.1 Core Form 1040 flow

- `line12e` — dependent standard-deduction worksheet uses AGI for the earned-income ceiling.
- `line13a` — QBI taxable-income limit uses line 11b.
- `line13b` — Schedule 1-A MAGI base = line 11b + add-backs.
- `line14` — composite deduction.
- `line15 = max(0, line11b - line14)`. **Line 15 has the zero-floor rule; line 11a / 11b do not.**

## 6.2 Accessor-preference asymmetry

Downstream consumers split between two accessors (semantically identical because of the same-reference assignment):
- **Prefer `getLine11bAmountFromLine11aAdjustedGrossIncome()`** — line 12e, 13a, 13b, 15, Schedule 8812, Form 8863, Form 8880, Form 8959, EIC. Page-2 logical locality.
- **Prefer `getLine11aAdjustedGrossIncome()` or legacy `getAdjustedGrossIncome()`** — Schedule A (medical floor 7.5% × AGI), Form 6251 AMT, Form 8962 PTC via `computeScheduleAgi()`.

The result is identical (same BigDecimal); the asymmetry is a stylistic inconsistency tracked as a low-priority cleanup.

# 7) Validation rules

## 7.1 Core numeric validation

```text
if Form1040.line9 is missing -> line11a = null (no AGI persisted; Adjustments suppressed when line10 also null)
if Form1040.line10 is missing -> treated as 0 by subtractNonNegativeAllowNegative
```

## 7.2 Copy-line validation

```text
Form1040.line11b == Form1040.line11a
```

This is enforced structurally by the same-reference assignment. The lock-in test `line11bAlwaysEqualsLine11aInvariant` covers positive, negative, and null AGI scenarios.

## 7.3 Sign handling

- Do **not** force line 11a to zero.
- Do **not** force line 11b to zero.
- Preserve the sign of the computed AGI.

The lock-in test `allowsNegativeAdjustedGrossIncomeOnForm1040Line11aAndLine11b` covers the negative-AGI path.

## 7.4 Semantic validation

- Do **not** treat line 11b as a precomputed universal MAGI.
- Do **not** add back excluded Puerto Rico income or Form 2555 amounts at line 11a or 11b.
- Apply those add-backs only in the specific downstream worksheet or schedule that requires them.

# 8) Implementation guardrails

## 8.1 Correct compute order

```text
line9 -> line10 -> line11a -> line11b -> downstream schedules and thresholds
```

## 8.2 Safe storage model

Preferred semantic model:

```text
agi = Form1040.line11a
agiCarryforward = Form1040.line11b
```

Downstream derived values should be separate, for example:

```text
schedule1aMagi
studentLoanInterestMagi
iraDeductionMagi
scheduleASalesTaxWorksheetIncome
```

## 8.3 Avoid these implementation mistakes

Do **not**:
- recompute line 11b from a separate formula;
- label line 11b as `MAGI` in code or UI;
- build one shared MAGI amount and reuse it everywhere;
- apply the line 15 zero-floor rule to line 11a or 11b;
- forget to sync `line11a` / `line11b` / `adjustedGrossIncome` together (they always carry the same value).

# 9) Practical developer cheat sheet

```text
Form1040.line11a = Form1040.line9 - Form1040.line10
Form1040.line11b = Form1040.line11a           # same BigDecimal reference
```

Meaning:
- **11a** = AGI calculation line.
- **11b** = AGI carryforward (page 2 top).

Key 2025 rule:
- **11b is still AGI, not MAGI.**

Backend invariants:
- `subtractNonNegativeAllowNegative` preserves negative AGI;
- legacy alias `adjustedGrossIncome` carries the same value;
- when line 9 is null and line 10 is null, the Adjustments object is suppressed so the Form 1040 cells render blank;
- on the Form 8814 augmentation path, both `line11a` and `line11b` are repointed atomically.

# 10) Bottom line

For tax year 2025:
- **line 11a** is the actual AGI computation;
- **line 11b** equals **line 11a** exactly (same reference);
- neither line has a zero-floor rule;
- line 11b is an AGI carryforward only — not a universal MAGI.
