Four structural facts that drive the line 16 implementation:
- **line 16 is not just one tax lookup** — it is the sum of the tax on taxable income **plus** any applicable **Form 8814 / Form 4972 / box-3 write-in taxes**;
- **Form 2555 has priority** over the ordinary Tax Table / Tax Computation Worksheet / QDCG / Schedule D / Form 8615 path — the **Foreign Earned Income Tax Worksheet** becomes the controlling line-16 method;
- **Schedule J is still an official 2025 line-16 method**, but is treated as **out of scope** in this codebase (farming / fishing income blocked upstream — see 16 #9 observation);
- **line 16 can be positive even when line 15 is zero** if any Box 1, Box 2, or Box 3 taxes apply (spec §2.3 critical zero rule).

# 1) Line identity (2025 Form 1040)

- **Line 16**: `Tax`
- Line 16 has three checkboxes / write-in areas:
  - **Box 1**: Form 8814 tax (child interest and dividends)
  - **Box 2**: Form 4972 tax (lump-sum distributions)
  - **Box 3**: certain specific write-in taxes (5 codes per spec §10.3 — all five now wired per V57)

The amount entered on line 16 includes both:
- the tax figured on **line 15 taxable income** using the applicable IRS method, and
- any specifically listed **additional taxes** that belong on line 16.

# 2) Exact line-16 structure

## 2.1 High-level formula

```text
Form1040.line16 =
    tax_on_taxable_income (regularTax)
  + Form8814_box1_if_any
  + Form4972_box2_if_any
  + box3_writein_total_if_any
```

`regularTax` is preserved on `TaxAndCredits.regularTax` (without box add-ons) because **Form 1040 line 17 (AMT) comparison uses `regularTax`, not the total**: `line17 = max(0, tentative_AMT − regularTax)`.

## 2.2 Box routing

```text
Box 1 → Form 8814 tax           — per-child summation across form8814List
Box 2 → Form 4972 tax           — taxpayer + spouse line30Total (MFS-guarded)
Box 3 → 5 codes: ECR / 962 / 1291TAX / Form 8978 / 965INC
```

Box 3 code field becomes a **comma-separated list** when multiple codes apply (per V57 multi-code support).

## 2.3 Important zero rule

If **Form 1040 line 15** is zero or less:
- the ordinary tax on taxable income is **0** (Branch 1 fires);
- but **Form1040.line16 is not automatically zero**, because Box 1, Box 2, or Box 3 taxes may still apply.

This critical rule means line 16 = 0 only when line 15 ≤ 0 AND no Box 1/2/3 add-ons fire.

# 3) Tax on taxable income: 2025 method hierarchy

Use the following decision order for the `tax_on_taxable_income` (regularTax) portion of line 16. **First match wins**.

## 3.1 7-branch decision tree

```text
Branch 1 (ZERO):
    if line15 <= 0:
        regularTax = 0
        method = "ZERO"

Branch 2 (FOREIGN_EARNED_INCOME):
    elif Form2555Taxpayer != null or Form2555Spouse != null:    (MFS guard nulls spouse)
        regularTax = ForeignEarnedIncomeTaxWorksheet.line6
        method = "FOREIGN_EARNED_INCOME"

Branch 3 (FORM_8615 — kiddie tax):
    elif kiddieIncomeTaxpayer.hasKiddieTaxUnearnedIncome == TRUE:
        regularTax = kiddieIncomeTaxpayer.childFinalTaxLine18 (manual entry)
                  or computeTaxBracket(line15, status) with WARN log if null
        method = "FORM_8615"  (or "TAX_TABLE" / "TAX_COMPUTATION_WORKSHEET" on fallback)

Branch 4 (SCHEDULE_D_TAX_WORKSHEET):
    elif isScheduleDTaxWorksheetRequired(scheduleD):
        regularTax = computeScheduleDTaxWorksheet(line15, status, ...)
        method = "SCHEDULE_D_TAX_WORKSHEET"

Branch 5 (QDCG_WORKSHEET):
    elif isQDCGWorksheetRequired(form1040, scheduleD):
        regularTax = computeQDCGWorksheet(line15, status, ...)
        method = "QDCG"

Branches 6 / 7 (TAX_TABLE / TAX_COMPUTATION_WORKSHEET):
    else:
        regularTax = computeTaxBracket(line15, status)
        method = line15 < $100,000 ? "TAX_TABLE" : "TAX_COMPUTATION_WORKSHEET"
```

Note: branches 6 and 7 share the same `computeTaxBracket` helper — the `$100,000` boundary is **label-only** in this codebase. The helper produces the identical value for both sides of the boundary; the `computationMethod` label is the only difference, exposed for PDF disclosure and E2E test assertions.

## 3.2 General rule

- If taxable income is **less than $100,000** AND no special method applies, use the **Tax Table** (label).
- If taxable income is **$100,000 or more** AND no special method applies, use the **Tax Computation Worksheet** (label).

## 3.3 Do not use the tax rate schedules directly

The 2025 Form 1040 instructions provide the **Tax Table** and **Tax Computation Worksheet** for figuring line 16. Do **not** substitute the tax rate schedules as the direct line-16 computation method.

## 3.4 Schedule J — out of scope

The 2025 Form 1040 instructions also list **Schedule J** (income averaging for farmers and fishermen) as an official line-16 method. This codebase **does not implement** Schedule J because farming / fishing income (Schedule C / Schedule F path) is out of scope per CLAUDE.md and is blocked upstream via blocking flags at the income lines. No Schedule J branch exists in `computeLine16`.

# 4) Special method: Form 8615

Use **Form 8615** for the regularTax component if all of the following are true:

1. The child had **more than $2,700 of unearned income**.
2. The child is **required to file a return**.
3. The child was either:
   - under age **18** at the end of **2025**;
   - age **18** at the end of **2025** and didn't have earned income greater than half of support; or
   - a full-time student age **19 through 23** at the end of **2025** and didn't have earned income greater than half of support.
4. At least one parent was alive at the end of **2025**.
5. The child does **not** file a joint return for **2025**.

Manual-entry pattern: Form 8615 itself is **out of scope for auto-computation** (the multi-step net-investment-income allocation + parent's bracket comparison is complex). The user fills Form 8615 externally (paper or third-party software) and enters the line 18 result into `childFinalTaxLine18` on the `kiddie-income-taxpayer` personal form. The backend reads that value:

```text
regularTax = roundMoney(kiddieIncomeTaxpayer.childFinalTaxLine18)
```

Defensive fallback: if `hasKiddieTaxUnearnedIncome=TRUE` but `childFinalTaxLine18` is null, the code emits a WARN log and falls back to `computeTaxBracket(line15, status)`. The method label flips to `TAX_TABLE` or `TAX_COMPUTATION_WORKSHEET` based on the $100k boundary.

Important interaction:
- if the taxpayer also files **Form 2555**, the Form 8615 result does **not** go directly to Form 1040 line 16; it flows through the **Foreign Earned Income Tax Worksheet**. The decision tree handles this structurally via priority order (Branch 2 fires before Branch 3). Known limitation: the current FEITW implementation uses TCW or QDCG for line 4, NOT Form 8615 — the rare combination "Form 2555 + Form 8615" produces a slightly different FEITW line 4 than the spec ideally wants (vanishingly rare scenario).

# 5) Special method: Schedule D Tax Worksheet

Use the **Schedule D Tax Worksheet** if either of the following is true:
- the taxpayer must file **Schedule D**, **line 18 or 19** of Schedule D is more than zero, and **lines 15 and 16** of Schedule D are gains; or
- the taxpayer files **Form 4952** and has an amount on **line 4g**, even if Schedule D itself is not required.

Predicate `isScheduleDTaxWorksheetRequired(scheduleD)` (~line 3149) implements the first condition:

```text
hasSpecialGain = (Sched D line 18 > 0) OR (Sched D line 19 > 0)
required       = hasSpecialGain AND (Sched D line 15 > 0) AND (Sched D line 16 > 0)
```

The Schedule D Tax Worksheet handles two income types that the QDCG worksheet cannot:
- **28% rate gain** (collectibles and §1202 stock with 50% exclusion);
- **Unrecaptured §1250 gain** (depreciation recapture, 25% max rate).

Do **not** use the Schedule D Tax Worksheet if:
- **Form 2555** is filed; in that case use the **Foreign Earned Income Tax Worksheet** instead (Branch 2 priority).

Important exception from the 2025 Schedule D instructions:
- do **not** use either the **Qualified Dividends and Capital Gain Tax Worksheet** or the **Schedule D Tax Worksheet** if:
  - Schedule D line 15 or 16 is zero or less and there are no qualified dividends on Form 1040 line 3a; or
  - Form 1040 line 15 is zero or less (Branch 1 fires first).

Mapping:

```text
regularTax = computeScheduleDTaxWorksheet(line15, status, scheduleD, qualDiv, capGainLoss)
```

# 6) Special method: Qualified Dividends and Capital Gain Tax Worksheet

Use the **QDCG Worksheet** if the taxpayer does **not** have to use the Schedule D Tax Worksheet AND any of the following applies:
- qualified dividends are reported on **Form 1040 line 3a**;
- the taxpayer does **not** have to file Schedule D and reported **capital gain distributions** on **Form 1040 line 7a**;
- the taxpayer **is** filing Schedule D and **Schedule D lines 15 and 16 are both more than zero**.

Predicate `isQDCGWorksheetRequired(form1040, scheduleD)` (~line 3174) encodes these three OR-conditions.

Do **not** use this worksheet if:
- **Form 2555** is filed; use the **Foreign Earned Income Tax Worksheet** instead (Branch 2 priority);
- the Schedule D Tax Worksheet is required (Branch 4 priority — mutual exclusivity is structurally enforced by decision-tree branch order).

## 2025 worksheet thresholds

| Filing status | 0% threshold | 20% threshold |
|---|---:|---:|
| Single | $48,350 | $533,400 |
| Married filing jointly / Qualifying surviving spouse | $96,700 | $600,050 |
| Married filing separately | $48,350 | $300,025 |
| Head of household | $64,750 | $566,700 |

Below 0% threshold → 0% on qualified dividends + LTCG; between thresholds → 15% rate; above 20% threshold → 20% rate. Ordinary income taxed at bracket rates regardless.

Mapping:

```text
regularTax = computeQDCGWorksheet(line15, status, qualDiv, scheduleD, capGainLoss, line7bSchedDNotRequired)
```

# 8) Special method: Foreign Earned Income Tax Worksheet

If the taxpayer claimed the **foreign earned income exclusion**, **housing exclusion**, or **housing deduction** on **Form 2555**, the taxpayer **must** figure tax using the **Foreign Earned Income Tax Worksheet**.

This is the **highest-priority** branch in the decision tree (Branch 2; only Branch 1 ZERO precedes it). Per IRC §911 + spec §12.2: without this priority, a Form 2555 filer with qualified dividends would route to Branch 5 (QDCG) and miss the FEITW stacking math, understating tax.

Important 2025 rule:
- if **Form 1040 line 15 is zero**, **do not** complete the Foreign Earned Income Tax Worksheet — Branch 1 fires first and Branch 2 is unreachable.

Core mapping:

```text
ForeignEarnedIncomeTaxWorksheet.line1  = Form1040.line15
ForeignEarnedIncomeTaxWorksheet.line2a = Form 2555 line 45 (FEIE)            (taxpayer + MFJ spouse)
ForeignEarnedIncomeTaxWorksheet.line2b = Form 2555 line 50 (housing excl)    (taxpayer + MFJ spouse)
ForeignEarnedIncomeTaxWorksheet.line2c = line2a + line2b
ForeignEarnedIncomeTaxWorksheet.line3  = line1 + line2c                       (the "stack")
ForeignEarnedIncomeTaxWorksheet.line4  = tax on line3 using TCW / QDCG / SchedD
ForeignEarnedIncomeTaxWorksheet.line5  = tax on line2c using Tax Table / TCW
ForeignEarnedIncomeTaxWorksheet.line6  = max(0, line4 − line5)               ← the FEITW result
```

The stacking formula prevents tax-rate arbitrage: $100k taxable + $50k FEIE is taxed at the marginal rate that applies at $150k (high), not $100k (low).

Final line-16 mapping when Form 2555 controls:

```text
regularTax = computeForeignEarnedIncomeTaxWorksheet(line15, form2555Taxpayer, form2555Spouse, status, ...)
```

Known limitation: the implementation's FEITW line 4 uses TCW or QDCG; the rare combination of Form 2555 + Form 8615 (child with foreign earned income exclusion) feeds TCW/QDCG into line 4 rather than Form 8615.

### Form 2555 seed gap (observed behavior)

Per the project memory `project_form2555_doesnt_net_wages_from_agi.md`: `foreign-earned-income-*` seeding produces Form 2555 + FEITW correctly but does **not** auto-route the exclusion through Schedule 1 line 8d. Line 1z stays at full W-2 wages; FEITW stacks on a larger base than the IRS-canonical line 15. The line-16 result is consistent with line 15 as input, but line 15 itself is higher than it should be in those scenarios.

# 10) Box 1, Box 2, and Box 3 add-on taxes

Three independent add-on tax paths stacked on top of `regularTax`. Each box has an independent checkbox + amount; each can fire even when `regularTax = 0` (critical zero rule per spec §2.3).

## 10.1 Box 1: Form 8814

If the taxpayer elects to report a child's interest or dividends on **Form 8814**, include the tax from **Form 8814** on line 16 and check **box 1**.

Implementation: per-child summation across `form8814List`:

```text
form8814AddonTax = Σ over form8814List of f8814.getLine15ExtraTax()
box1Checked      = form8814AddonTax > 0
```

Multiplicity rule (per CLAUDE.md): one Form 8814 per child; child gross income must be < $13,500.

## 10.2 Box 2: Form 4972

If the taxpayer uses **Form 4972** for tax on a qualifying lump-sum distribution, include that tax on line 16 and check **box 2**.

Implementation: taxpayer + spouse `line30Total` summed (MFS-guarded):

```text
form4972AddonTax = (form4972Taxpayer?.line30Total ?? 0) + (form4972Spouse?.line30Total ?? 0)
box2Checked      = form4972AddonTax > 0
```

Form 4972 is per-person (one per participant born before Jan 2, 1936; not for IRAs / 403(b)s).

**MFS defensive guard**: `form4972Spouse` is null-shadowed at the `computeLine16` call site (~line 1213 in `prepare()`) when `isMfsReturn=true`. Without the guard, MFS taxpayer's line 16 would wrongly include the spouse's stale Form 4972 lump-sum tax. Bundled fix with `form2555Spouse` per 16 #1; 18th orchestrator in the MFS-guard cascade.

### Known issue — Form 4972 Part II cap-gain double-tax

Per the project memory `project_form4972_part2_capgain_not_removed_from_line5b.md`: when Form 4972 Part II is elected on a 1099-R lump-sum distribution, the capital-gain portion is **not** removed from line 5b. Effective double-tax: the cap-gain portion is taxed once at the ordinary bracket via line 5b AND again at 20% via Form 4972. Compute-layer follow-up needed; not blocked at line 16.

## 10.3 Box 3: write-in taxes (all 5 codes wired per V57)

The 2025 Form 1040 instructions list five Box 3 write-ins. **All five are now wired** in `computeLine16` (V57 closure, 2026-06-08); previously only ECR was implemented.

| Code | Source | 1040 line-16 treatment |
|---|---|---|
| `ECR` | Education credit recapture (from `16-tax-taxpayer.hasEcrRecapture` + `ecrAmount`) | Include on line 16 and write `ECR` |
| `962` | Section 962 election tax (from `hasSection962Election` + `section962TaxAmount` − `section962Form1118ForeignTaxCredits`) | Include amount net of Form 1118 FTC credits; attach statement |
| `1291TAX` | Form 8621 line 16e PFIC excess distribution (from `hasForm8621PficTax` + `form8621Line16eAmount`) | Include on line 16 and write `1291TAX` |
| `Form 8978` | Form 8978 line 14 BBA partnership pushout (from `hasForm8978Adjustment` + `form8978Line14Amount`) | Include only if **positive**; if **negative**, route to Schedule 3 line 6l (refundable credit) — see special handling below |
| `965INC` | Section 965(i) triggered deferred tax (from `hasSection965iTriggered` + `section965iTriggeredAmount`) | Include on line 16 and write `965INC` |

Safe implementation mapping:

```text
box3Total = ecrAddonTax + section962AddonTax + form8621AddonTax + form8978AddonTax (positive only) + section965iAddonTax
box3Code  = comma-separated list of all codes that fire (per V57 multi-code support)
```

### Form 8978 negative-amount routing

Per IRS instructions: a **negative** Form 8978 line 14 amount does **NOT** go on Form 1040 line 16; it goes on **Schedule 3 line 6l** as a refundable credit. The Schedule 3 line 6l field is **not yet plumbed** in this codebase, so when a negative amount is entered the backend emits a non-blocking advisory flag `LINE16_FORM_8978_NEGATIVE_ROUTES_TO_SCHEDULE3_LINE6L` asking the user to claim it manually. The negative amount itself is **not** added to line 16.

### Known issue — multi-code Box 3 e2e returns 500

Per the project memory `project_line16_box3_three_codes_e2e_500.md`: **any** multi-code Box 3 (2+ codes) on `16-tax-taxpayer` returns **HTTP 500** in e2e while the equivalent Java unit test passes. Single-code combos all work. Investigation deferred; both 2-code and 3-code e2e tests are `test.skip`-ped. The compute logic itself is correct (Java unit tests pass); the issue is in the REST / persistence / JSON-serialization layer that e2e exercises.

# 12) Validation rules

## 12.1 Core validation

```text
if Form1040.line15 is missing -> computeLine16 silently skips (gate at line 2765)
if filing_status is missing   -> defaults to bracket-table fallback
```

Line 16 itself emits **no blocking flags** — it gates on null line 15. Upstream blocking flags from lines 11ab / 12abcde / 13a / 13b / 14 / 15 propagate; line 16 does not add its own.

## 12.2 Method exclusivity validation (structural)

```text
Form 2555 priority           → Branch 2 fires before Branches 3-7 (structurally enforced)
Schedule D Tax Worksheet     → Branch 4 fires before Branch 5 QDCG (structurally enforced)
Branch 1 ZERO priority       → fires before Branch 2 FEITW (handles spec §6.3 FEITW zero gate)
```

## 12.3 Form 8978 sign rule

```text
if Form8978.line14 < 0
    do NOT add to line 16; emit advisory flag (LINE16_FORM_8978_NEGATIVE_ROUTES_TO_SCHEDULE3_LINE6L)
    user manually claims on Schedule 3 line 6l
```

## 12.4 Zero-line15 validation

```text
if line15 <= 0 and no Box 1/2/3 taxes apply and line16 != 0
    -> bug (should not happen — Branch 1 sets regularTax=0)
```

If line 15 ≤ 0 but Box 1 / Box 2 / Box 3 taxes apply, line 16 may still be positive (per spec §2.3).

# 13) Compute order

```text
1. Finalize Form1040.line15
2. Determine the correct tax-on-taxable-income method via 7-branch decision tree
3. Compute regularTax
4. Compute Form 8814 box 1 add-on (per-child sum)
5. Compute Form 4972 box 2 add-on (taxpayer + MFS-guarded spouse)
6. Compute Box 3 add-ons (5 codes, multi-code list)
7. Sum: line16 = regularTax + box1 + box2 + box3
8. Persist on TaxAndCredits (tax + regularTax + computationMethod + box*/code)
9. Mark Box 1/2/3 checkboxes via box1Checked / box2Checked / box3Checked
```

Line 16 is the **first orchestrator-based audit since 13b** — earlier deductions chain (12abcde → 13a → 13b → 14 → 15) is inline-computed; `computeLine16` (~line 2748) is a dedicated orchestrator with 4 helper methods (`computeTaxBracket` + `computeQDCGWorksheet` + `computeScheduleDTaxWorksheet` + `computeForeignEarnedIncomeTaxWorksheet`) and 2 predicates (`isScheduleDTaxWorksheetRequired` + `isQDCGWorksheetRequired`).

Call site: `prepare()` ~line 1583, after Form 2555 / Form 4972 / Form 8814 / Schedule D / kiddie income / Schedule 1-A / line 13a second pass / line 14 / line 15 are all finalized.

# 14) MFS protection at the call site

`computeLine16` is invoked once from `prepare()` (~line 1583). The call site null-shadows two spouse-side inputs on MFS, bundled into the 16 #1 fix:

```java
computeLine16(form1040, filingStatus, scheduleD, form8814List,
    form4972Taxpayer,
    isMfsReturn ? null : form4972Spouse,    // 16 #1: MFS guard (Box 2 leak)
    form2555Taxpayer,
    isMfsReturn ? null : form2555Spouse,    // 16 #1: MFS guard (Branch 2 FEITW leak)
    kiddieIncomeTaxpayer, line16TaxTaxpayer, flags, uid);
```

Without these guards:
- **Form 2555 spouse leak**: Branch 2 (`form2555Taxpayer != null || form2555Spouse != null`) would fire on MFS with stale spouse FEIE, stacking spouse FEIE into the FEITW bracket and producing OVER-stated tax.
- **Form 4972 spouse leak**: Box 2 would unconditionally add `form4972Spouse.getLine30Total()`, leaking spouse's stale lump-sum tax into MFS taxpayer's line 16.

`computeLine16` is the **18th orchestrator** in the MFS-guard cascade (per 16 #1) — new codebase maximum at the time.

```text
  + Form8814_box1_if_any

  + box3_writein_total_if_any   (5 codes: ECR / 962 / 1291TAX / Form 8978 / 965INC)
```

Method order for `regularTax`:

```text
ZERO → Form 2555 (FEITW) → Form 8615 → Schedule D Tax Worksheet → QDCG → Tax Table / Tax Computation Worksheet
```

Interpretation:
- `ZERO` short-circuits to 0 and gates the FEITW (spec §6.3);
- `Form 2555` is the top-priority override per IRC §911;
- `Form 8615` / `Schedule D` / `QDCG` are special methods;
- the **Tax Table** and **Tax Computation Worksheet** are fallback ordinary methods only;
- Schedule J is officially listed by the IRS but **not implemented** (out of scope).

# 17) Bottom line

For tax year 2025:
- **line 16** is the total of the tax on taxable income **plus** any applicable **Form 8814 / Form 4972 / box-3 taxes**;
- the ordinary fallback is **Tax Table under $100,000** and **Tax Computation Worksheet at $100,000 or more**;
- but **Form 2555, Form 8615, Schedule D Tax Worksheet, and QDCG Worksheet** can override that ordinary path (Schedule J is out of scope);
- **line 16 is not guaranteed to be zero just because line 15 is zero** — Box 1/2/3 add-ons can produce positive line 16 with line 15 = 0;
- `regularTax` (without add-ons) is preserved separately for the Form 1040 line 17 AMT comparison;
- multi-code Box 3 e2e tests are currently skipped due to a known 500 in the REST layer (compute logic is correct).
