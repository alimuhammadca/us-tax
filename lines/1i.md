# Form 1040 Line 1i - Nontaxable combat pay election

## 1. Line identity

- Form: `1040 / 1040-SR`
- Line: `1i`
- Label: `Nontaxable combat pay election`
- Concept: optional election to include nontaxable combat pay in EARNED INCOME for the Earned Income Credit (EIC) and Additional Child Tax Credit (ACTC) **without** making the combat pay taxable
- Semantic PDF field: `topmostSubform[0].Page1[0].f1_56[0]` (key `line1i_nontaxable_combat_pay`)
- Output field path: `form1040.income.nontaxableCombatPayElection`
- Backend method: `TaxReturnComputeService.computeCombatPay(...)` declared at line 19343
- Call site: line 426 in `prepare()`

This is a 2025 line spec.

## 2. Line intent for 2025

For tax year 2025, line 1i is used when a taxpayer elects to include nontaxable combat pay in earned income when figuring the EIC and ACTC. The elected amount is entered on `Form 1040 / 1040-SR line 1i`.

Important boundary:

- line 1i does **not** make the combat pay taxable
- line 1i does **not** increase wages on lines 1a through 1h
- line 1i does **not** enter line 1z, line 9, AGI, or taxable income
- line 1i is used for credit-worksheet earned-income computations only
- the election is optional and may sometimes **decrease** EIC (see ┬з6)

## 3. Source input

The authoritative source is `Form W-2 box 12 code Q` (Nontaxable combat pay). The implementation uses only `W-2 box 12 code Q` amounts, filtered by `employeeSSN`.

Constants:

- `COMBAT_PAY_CODES = Set.of("Q")` at TaxReturnComputeService.java:124

### Eligible recipients

Full exclusion under IRC ┬з112:

- Enlisted members
- Warrant officers
- Commissioned warrant officers

Commissioned officers receive a **limited exclusion** under IRC ┬з112(c): capped at the highest rate of enlisted basic pay plus hostile fire / imminent danger pay. The W-2 issuer applies the cap before reporting code Q. The backend trusts the issuer-reported amount and does NOT re-derive the officer cap.

## 4. Core 2025 rules

### Optional election

The election is optional.

**IRS caution (Pub. 596)**: the IRS explicitly states that electing to include combat pay may sometimes **decrease** rather than increase the allowable EIC. Taxpayers should compute the credit both with and without combat pay before deciding.

### All-or-none per person

If a person elects, that person must include **all** of their nontaxable combat pay. Partial inclusion is not allowed.

Code enforcement at lines ~19376-19386:

```java
taxpayerElected = Boolean.TRUE.equals(electTaxpayer)
    ? (taxpayerTotal == null ? BigDecimal.ZERO : taxpayerTotal)
    : null;
```

UI offers no partial-amount input (Yes/No radio only).

### MFJ spouse-by-spouse election

On an MFJ return, each spouse may elect independently. All four combinations are valid:

- taxpayer includes all, spouse includes none
- taxpayer includes none, spouse includes all
- both include all
- both include none

### MFS gating

The orchestrator passes `isMfsReturn` to `computeCombatPay`. On MFS, three explicit gates null out spouse-side processing (Code Validation #1, 2026-05-10):

```java
spouseSsn    = isMfsReturn ? null : normalize(spouse.ssn);
electSpouse  = isMfsReturn ? null : Boolean.TRUE.equals(...combat-pay-spouse.electCombatPay);
spouseTotal  = isMfsReturn ? null : sumW2Box12ByCodes(w2Entries, COMBAT_PAY_CODES, spouseSsn);
```

Three-point gate (vs the two-point pattern in lines 1cтАУ1h) is required because `sumW2Box12ByCodes(_, _, null)` sums ALL entries тАФ gating spouseSsn alone would not have prevented the leak.

## 6. Credit-worksheet effect

### Earned Income Credit (EIC) тАФ line 27a

```
earnedIncomeForEic = normalEarnedIncome + form1040.income.nontaxableCombatPayElection
```

EIC is then looked up from the EIC table using the **lesser of** `earnedIncomeForEic` or AGI at the applicable phaseout threshold. Combat pay raises earned income but NOT AGI:

- **EIC phase-in range** (low income): adding combat pay increases earned income тЖТ higher EIC. Election BENEFICIAL.
- **EIC phaseout range**: adding combat pay pushes earned income further into the phaseout table тЖТ lower credit. Election may be DETRIMENTAL.
- **Fully phased out**: election has NO effect.

**Source-of-truth fix (Code Validation #2, 2026-05-10).** `computeLine27aEIC` at lines ~16604-16612 now reads `form1040.getIncome().getNontaxableCombatPayElection()` as the single source of truth тАФ same pattern as Schedule 8812. Legacy `electNontaxableCombatPay` flag on the EIC personal form is ignored by the backend.

### Additional Child Tax Credit (ACTC) тАФ Schedule 8812 Part II-A

```
line18a = income.getTotalWages() + income.getNontaxableCombatPayElection()
line18b = income.getNontaxableCombatPayElection()   // IRS disclosure of combat-pay component
line19  = max(0, line18a - $2,500)                  // ACTC_EARNED_INCOME_FLOOR
line20  = line19 ├Ч 0.15                             // ACTC_RATE
```

Refundable ACTC per child capped at `$1,700` (2025; `ACTC_PER_CHILD_CAP`).

Combat pay is included alongside taxable earned income for the 15% calculation, so electing generally benefits ACTC for lower-income families. Wired CORRECTLY at lines 19087-19096.

## 7. What line 1i is not

Line 1i is **not**:

- taxable wages
- part of `Form 1040 line 1z`
- part of `Form 1040 line 9` (total income)
- part of `AGI` (line 11b)
- part of `taxable income` (line 15)
- a place for BAH, BAS, or other military allowances
- a place for military retirement pay
- a general military earned-income adjustment outside `code Q` combat pay

## 8. Related: IRA contribution compensation (out of scope)

Per IRC ┬з219(f)(7) and Pub. 590-A, nontaxable combat pay counts as compensation for IRA contribution limits. Members of the armed forces serving in a combat zone may base traditional IRA or Roth IRA contributions on tax-free combat pay.

**NOT currently wired** тАФ the backend IRA path covers only distributions (`computeIraDistributions` at line 4832 for lines 4a/4b). IRA contribution-deduction cap validation is not implemented at all. Deferred per Code Validation #7; outstanding.md entry tracks the broader IRA-cap feature.

When implemented, the ┬з219(f)(7) piece must read `sumW2Box12ByCodes(w2Entries, COMBAT_PAY_CODES, personSsn)` (the underlying sum) rather than `form1040.income.nontaxableCombatPayElection` тАФ IRC ┬з219(f)(7) is election-independent.

## 9. Statement / filing behavior

No separate IRS election form is filed. For 2025, the election is made by entering the elected amount on Form 1040 line 1i.

The IRS no longer requires writing `NCP` next to line 1i (per the 2023 revision of Pub. 3). **The line 1i value IS the election.**

## 10. Implementation requirements

### SSN attribution

W-2 box 12 code Q amounts are filtered by `employeeSSN` against taxpayer / spouse SSN via `normalize()` (digits-only). The `spouse` map fed into `computeCombatPay` is the resolved output of `resolveSpouseIdentification(legacySpouse, identificationSpouse)` at line 151, which overlays `identification-spouse.spouseSsn` into the resolved `spouse.ssn` slot via `overlayResolvedField` at line 3902. (Audit false positive тАФ Code Validation #3, 2026-05-10.)

### All-or-none enforcement

The UI offers no partial-amount input. Backend enforces via boolean ternary.

### No-code-Q behavior тАФ design choice

If `electCombatPay = true` but no code Q W-2 entries match the person's SSN, the elected amount is `BigDecimal.ZERO` (NOT null). This distinguishes "user explicitly elected with no Q" from "user did not elect" тАФ allows ACTC/EIC worksheets to see a deliberate `0`.

### Defensive null-SSN guards (Code Validation #4)

`sumW2Box12ByCodes(_, _, null)` returns the sum of ALL code Q entries regardless of attribution. Defensive guards added at the two combat-pay sum sites:

```java
taxpayerTotal = hasText(taxpayerSsn) ? sumW2Box12ByCodes(...) : null;
spouseTotal   = (isMfsReturn || !hasText(spouseSsn)) ? null : sumW2Box12ByCodes(...);
```

Lock-in test `combatPayWithNullTaxpayerSsnDoesNotAbsorbCodeQFromOtherW2s` prevents regression.

### UI gating

When no code Q is found for a person, the form shows a recovery-pointing empty-state message and hides decision controls.

When code Q is present, three labeled sections render (UX pass 2026-06-02):

1. "What we found on your W-2" тАФ derived combat pay total + provenance label + correction pointer
2. "See how each choice affects your credits" тАФ auto-running EIC comparison panel (precedes the radio)
3. "Your choice" тАФ Yes/No radio + field-help + optional mismatch chip

### EIC "both ways" decision support (`eic-comparison-panel.component.ts`)

Per IRS Pub. 596, the panel must:

- compute both ways and show both totals (not just pick the larger)
- auto-run on first visit when no election has been saved (`autoRun = !hasSavedElection()`)
- highlight the larger combined credit via a winning-cell class
- show absolute dollar delta and "Recommended: Yes / No / Either" chip
- restore the user's persisted election in a `finally` block тАФ comparing must never silently change the saved choice
- emit `comparisonComplete` so the parent form can react (mismatch chip)
- detect stale state via `PersonalDataService.updated` signal effect; show amber "These numbers may be out of date" banner

Edge cases:

- hidden when no code Q present
- ineligible EIC (Form 2555, age, etc.) тЖТ ACTC-only note
- backend 409 (blocking flags) тЖТ "Complete the rest of your return firstтАж" with flag messages
- transport error тЖТ retryable message; still restore saved state

### Mismatch chip

When the user picks Yes/No but the most recent successful comparison recommended the opposite, render an inline amber chip below the radio group with the difference amount.

### Save feedback

The Save button surfaces a descriptive status echoing the saved choice (held ~6 seconds), not a generic "Saved":

- Yes тЖТ "Saved тАФ combat pay will be counted as earned income for the Earned Income Credit and Additional Child Tax Credit."
- No тЖТ "Saved тАФ combat pay will not be counted as earned income for these credits."
- Unset тЖТ "Saved." (partial-progress save with no choice yet тАФ per the Save vs. compute validation principle)

## 12. Validation flags

**Line 1i emits NO flags.** Per spec ┬з13 invariants, line 1i never produces a `TaxReturnFlag`. Line 1i is purely an OPTIONAL ELECTION; there is no scenario where the return cannot be filed because of line 1i state.

This makes line 1i unique among the 1aтАУ1hтАУ1i set тАФ all other lines emit at least one blocking or advisory flag.

| Scenario | Behavior | Where the user-side message lives |
|---|---|---|
| Elected but no code Q W-2 found | Result is `BigDecimal.ZERO`; no warning emitted | UI form-combat-pay-{taxpayer,spouse} disables the dropdown and shows recovery message |
| No election (`electCombatPay = null/false`) | Line 1i is null; no flag | UI shows election radio enabled when code Q exists |
| Officer code Q exceeds plausible cap | No backend recalculation; trusts issuer | No UI message |
| Line 1i exceeds taxable wages | Normal тАФ combat pay is nontaxable; can exceed wages on deployed servicemember | No flag |
| Multiple W-2s with code Q for same person | Summed correctly via `sumW2Box12ByCodes` | No flag |
| Code Q with `$0` amount | Summed as 0; if elected тЖТ ZERO (not null) | No flag |

## 14. Invariants

- `Form1040.Line1i >= 0`
- For each person: elected amount is either `null`, `BigDecimal.ZERO`, or the full code Q total
- `Form1040.Line1i <= sum(all W-2 code Q amounts on the return)`
- `Form1040.Line1i` must NOT flow into `line 1a`, `line 1z`, `line 9`, or `AGI`
- If `electCombatPay == true` and no code Q W-2 тЖТ result is `BigDecimal.ZERO`, not null
- Line 1z formula at line 7361 uses 8 operands (1aтАУ1h); line 1i intentionally absent
- Line 9 formula at line 7440 uses line 1z + 2b + 3b + 4b + 5b + 6b + 7a + 8; line 1i intentionally absent

Lock-in test `computesLine1zSumExcludesCombatPayElection` (TaxReturnComputeServiceTest:6291) protects the line 1z exclusion: `$5,000` wages + `$1,500` code Q + `electCombatPay=true` тЖТ `line 1z = $5,000` (combat pay NOT added).

## 18. 2025 corrections vs older / looser specs

A correct 2025 line spec must reflect:

- `line 1i` is for the combat pay election itself, not taxable wages
- the election is all-or-none per person
- MFJ spouses may elect independently
- only `W-2 box 12 code Q` belongs on line 1i
- the election is made by populating line 1i; no `NCP` notation required (post-2023 Pub. 3)
- line 1i affects credit earned-income calculations but does NOT change wage totals, line 9, AGI, or taxable income
- the election may DECREASE EIC for taxpayers in the phaseout range; compute both ways
- nontaxable combat pay also counts as IRA contribution compensation (IRC ┬з219(f)(7); out of scope here)
