# Form 1040 Line 1z - Total wages

## 1. Line identity

- Form: `1040 / 1040-SR`
- Line: `1z`
- Label: `Add lines 1a through 1h` (subtotal of the earned-income block)
- Concept: pure arithmetic subtotal of the eight earned-income wage lines (1a–1h); bridges per-source income computations into the Form 1040 total-income flow via line 9
- Semantic PDF field: `topmostSubform[0].Page1[0].f1_57[0]` (key `line1z_total_wages`)
- Output field path: `form1040.income.totalWages`
- Backend: inline computation in `TaxReturnComputeService.buildIncome()` at lines 7361-7363 (not a separate named method since line 1z is just an aggregation)

This is a 2025 line spec.

## 2. 2025 IRS label

For tax year 2025, the IRS label is literally `"Add lines 1a through 1h"`. The line 1z subtotal range is fixed; the only design choice the IRS exposes is whether line 1i is in or out — and the IRS answer is OUT, per the line 1i instruction:

> "Don't include the amount on line 1i in your total income (line 9) or wages (line 1z)."

## 4. What line 1z EXCLUDES — line 1i

The critical 2025 boundary: line 1z **stops at line 1h**.

Line 1i (nontaxable combat pay election) is a credit-worksheet earned-income marker, not a wage subtotal line. Even though line 1i appears in the same Form 1040 section, it MUST NOT enter line 1z.

### Why line 1i exclusion matters

Combat pay (W-2 box 12 code Q) is excluded from W-2 box 1 under IRC §112 — it never enters wages in the first place. Adding line 1i to line 1z would:

- erroneously increase total wages (already-excluded income reappearing)
- erroneously increase line 9, AGI, and taxable income
- defeat the purpose of the line 1i election, which is to grant CREDIT access (EIC, ACTC) without making the pay taxable

Line 1i RE-ENTERS only at:

- Schedule 8812 line 18a (`totalWages + nontaxableCombatPayElection`) — for ACTC earned income
- EIC line 27a worksheet (`normalEarnedIncome + nontaxableCombatPayElection`)

### Code enforcement

At `TaxReturnComputeService.java:7361-7362` the line 1z chain is exactly 8 operands:

```java
BigDecimal totalWagesRaw = addNonNull(addNonNull(addNonNull(addNonNull(addNonNull(addNonNull(addNonNull(
    line1a, line1b), line1c), line1d), line1e), line1f), line1g), line1h);
BigDecimal line1z = roundMoney(totalWagesRaw);
```

Line 1i (`combatPayElection`) is set on `income` separately at lines 7477-7479 (`income.setNontaxableCombatPayElection(...)`).

Multi-layer lock-in tests protect against regression:

- `computesLine1zSumExcludesCombatPayElection` — composite scenario, asserts `line 1z = -$5,280` with adoption + combat pay
- `line1zMinimalLine1iExclusion` — minimal isolation: W-2 `$4,000` + code Q `$2,000` + elect=true → `line 1z = $4,000`, `line 1i = $2,000`
- `line1zEqualsLine1aWhenOnlyW2Wages` — single-sub-line baseline

### Other exclusions from line 1z

- Interest / dividend lines (`2a`, `2b`, `3a`, `3b`)
- IRA, pension, Social Security, capital gain lines (`4a/b`, `5a/b`, `6a/b/c/d`, `7a/b`)
- Schedule 1 income (`line 8`)

## 5. Top-level formula

```
line1z = roundMoney( addNonNull(addNonNull(addNonNull(addNonNull(addNonNull(addNonNull(addNonNull(
                       line1a, line1b), line1c), line1d), line1e), line1f), line1g), line1h) )

line9  = roundMoney( addNonNull(addNonNull(addNonNull(addNonNull(addNonNull(addNonNull(addNonNull(
                       line1z, line2b), line3b), line4b), line5b), line6b), line7a), line8) )
```

Each sub-line helper applies `roundMoney()` before returning, so line 1z sums already-rounded values. The `roundMoney(totalWagesRaw)` final wrap is typically a no-op (aggregate is already integral) and a null-passthrough otherwise.

## 6. Signed pass-through — line 1z may be negative

`addNonNull` performs algebraic addition with NO `max(0, ...)` floor. So when a sub-line is negative under IRS rules, the negative amount remains part of the subtotal.

Within current scope:

- **Line 1f** may be negative because Form 8839 line 31 may be negative under IRC §137(a)(3) special-needs adoption finality cases (lifetime exclusion exceeds current-year reported benefits)

The negative carries through:

- line 1z → line 9 → AGI (line 11b)
- floor at `max(0, ...)` is applied only at line 15 (taxable income)

Inline breadcrumb at lines 7364-7369 documents this with the lock-in test reference.

## 7. Null behavior

- If all 8 sub-lines are null, `addNonNull(null, null) = null` propagates through the chain → `line1z = null` → `income.totalWages` stays unset → PDF cell blank.
- This avoids printing `$0` on returns with no wage activity (interest-only filers, retirement-only filers).

### `hasEmploymentFlag` zero-override invariant

`hasEmploymentFlag` is derived inline at lines 7349-7350 from either spouse's `employment-income-{taxpayer,spouse}.hasEmploymentIncome == true`. When this flag is true AND line 1a is null, the backend force-sets `income.wages = BigDecimal.ZERO` at lines 7449-7451 (so the UI renders `$0` rather than blank on a "yes, expect employment" return with no W-2 yet).

**Mirror invariant on `totalWages`** (lines 7496-7500):

```java
if (line1z != null) {
    income.setTotalWages(line1z);
} else if (hasEmploymentFlag) {
    income.setTotalWages(BigDecimal.ZERO);
}
```

This enforces: `income.totalWages` is non-null whenever `income.wages` is non-null. Pre-2026-05-10, the backend left `income.totalWages = null` in the hasEmploymentFlag-true-but-no-W-2 case, relying on a frontend fallback `totalWages ?? wages` for PDF rendering. The fallback has been REMOVED; the invariant is enforced at the source.

Lock-in test `line1zStoresZeroWhenHasEmploymentFlagAndNoWageInputs` protects the override.

**Practical limitation.** The end-to-end "all 8 sub-lines null → line 1z null" path is currently UNREACHABLE in observable scenarios because of a cross-sub-line stylistic 0-vs-null choice — line 1e (`computeDependentCareBenefits`) returns `BigDecimal.ZERO` (not null) when the childcare-expenses form is absent, so line 1z is always non-null whenever any sub-line compute runs. The defensive `else if (hasEmploymentFlag)` branch is therefore dormant but future-proofs the invariant if line 1e is ever refactored.

## 8. Cross-sub-line MFS attribution — single-guard cascade

Line 1z aggregation has NO spouse-attribution logic of its own. Each sub-line owns its MFS gating:

- Lines 1c, 1d, 1e, 1f, 1g, 1h, 1i all accept an `isMfsReturn` parameter (single-guard cascade — 7-line pattern as of 1i Issue #1 closure 2026-05-10)
- Lines 1a/1b use SSN-filter inside `computeLine1aWages` / `sumHouseholdEmployeeWages`

Adding an MFS guard at line 1z would be:

- redundant (sub-lines already gate spouse contributions)
- potentially harmful (could mask sub-line bugs by double-suppressing legitimate values)

Architecturally, the aggregator is the WRONG layer for MFS logic — its job is summing, not filtering.

## 9. Rounding

- Each sub-line is whole-dollar HALF_UP rounded inside its compute helper
- `roundMoney(totalWagesRaw)` at line 7363 wraps the aggregate (HALF_UP at 50¢ per `ROUNDING_MODE` constant)
- Dual-rounding is consistent — no risk of "sum of rounded sub-lines differs from rounded sum" because each sub-line is already integral by the time line 1z sees it

## 10. Validation flags

**Line 1z emits NO flags.** Line 1z is a pure aggregation; it never blocks the return. Sub-line blocking flags propagate FROM the sub-line computations (e.g., `MISSING_W2_EMPLOYMENT_INCOME_TAXPAYER`, `FORM8919_IRS_DATE_REQUIRED_*`, `ADOPTION_BENEFITS_*`, `LINE1H_DISABILITY_*`).

| Scenario considered | Why no flag at line 1z | Where blocking actually happens |
|---|---|---|
| Negative line 1z (e.g., from negative line 1f) | Permitted per §6 + IRC §137(a)(3) | No flag; downstream `max(0, ...)` at line 15 |
| Line 1z exceeds plausible wage range | No invariant; high earners can legitimately exceed `$1M` | No flag |
| All sub-lines null → line 1z null | Expected per §7 | No flag; PDF cell blank |
| `hasEmploymentFlag=true` but no W-2 | Sub-line responsibility | `MISSING_W2_EMPLOYMENT_INCOME_TAXPAYER` from line 1a; line 1z stores `BigDecimal.ZERO` via the override |
| Line 1f negative + special-needs gate fails | Sub-line responsibility | Per-child flags from line 1f, not line 1z |

Line 1z is UNIQUE among the 1a–1h–1i set in emitting nothing. Every other line has at least one blocking or advisory flag. This is intentional — line 1z is a passive sum.

## 13. Cross-line interactions

- Line 1z carries the SIGN of line 1f through to line 9 / AGI (negative pass-through)
- Line 1z does NOT contain line 1i — but line 1i RE-ENTERS at Schedule 8812 line 18a and EIC line 27a (separately added)
- Line 1z is the FIRST operand of line 9; null/zero/positive/negative all flow correctly

## 14. Negative / null contract

| Value | Meaning |
|---|---|
| `null` | All 8 sub-lines null AND `hasEmploymentFlag=false` — most rare in practice (see §7 practical limitation) |
| `BigDecimal.ZERO` | Either sub-lines aggregate to 0, or `hasEmploymentFlag=true` with no W-2 inputs |
| Positive | Standard case |
| Negative | Special-needs adoption (line 1f negative) per IRC §137(a)(3); reduces total wages |

## 15. Reference data

Line 1z has **ZERO** `ReferenceData` constants. It is a pure arithmetic subtotal. All constants relevant to line 1z arrive pre-applied via the sub-line computations:

- Line 1e earned-income limit
- Line 1f `$17,280` per-child adoption max, `$259,190` / `$299,190` phaseout
- Line 1g `$176,100` SS wage base; 6.2% / 1.45% / 0.9% FICA rates
- Line 1h `$23,500` §402(g) limit, `$16,500` SIMPLE limit, `$11,250` / `$5,250` enhanced catch-ups; `$3,000` PSO cap
- Line 1i `Set.of("Q")` combat pay codes

Only `ROUNDING_MODE` (WHOLE_DOLLAR HALF_UP at `TaxReturnComputeService.java:111`) is consumed at line 1z itself.

## 16. Invariants

Required invariants:

- `Line1z = addNonNull(line1a, line1b, line1c, line1d, line1e, line1f, line1g, line1h)` — exactly 8 operands
- `Line1z` MUST NOT include `line 1i`
- If a component line is negative and valid under IRS rules (line 1f from IRC §137(a)(3)), line 1z must include the negative amount (no `max(0, ...)` floor at line 1z level)
- If all 8 component lines are null AND `hasEmploymentFlag=false`, line 1z is null and `income.totalWages` is unset
- If `hasEmploymentFlag=true`, `income.totalWages` is non-null (either the computed sum or `BigDecimal.ZERO`)
- `income.totalWages` is non-null whenever `income.wages` is non-null (zero-override mirror invariant)

## 17. 2025 corrections vs older / looser specs

A correct 2025 line 1z spec must explicitly avoid these mistakes:

- treating line 1z as `1a through 1i` (line 1i is excluded)
- including the nontaxable combat pay election in wages totals
- treating line 1z as only W-2 wages (excludes other 1b–1h categories)
- dropping negative includable wage-line amounts from the subtotal (line 1f signed pass-through)
- relying on the deleted frontend fallback `totalWages ?? wages` for the `hasEmploymentFlag=true` zero-override (the invariant is now enforced at the backend)
