## 1. Line Identity

**Form 1040 line 21** is:

```text
Add lines 19 and 20
```

Exact mapping:

```text
Form1040.line21 = Form1040.line19 + Form1040.line20
```

For tax year 2025:

- `line 19` = child tax credit or credit for other dependents from `Schedule 8812 line 14`;
- `line 20` = amount from `Schedule 3 line 8`;
- `line 21` = total nonrefundable credits claimed at this stage of page 2.

## 2. Core 2025 Formula

```text
Form1040.line19 = Schedule8812.line14                         (CTC + ODC; already credit-limited)
Form1040.line20 = Schedule3.line8                             (other nonrefundable; already summed)
Form1040.line21 = Form1040.line19 + Form1040.line20
```

Implementation convention (null-as-zero coercion):

```text
Form1040.line21 = nz(Form1040.line19) + nz(Form1040.line20)
```

where `nz(x)` means treat blank / null as zero for arithmetic. Backend uses `safeAmount(...)`.

---

## 3. What Line 21 Means

Line 21 is the **subtotal of nonrefundable credits** shown directly on Form 1040 page 2 before subtracting them from the line-18 tax subtotal.

It is **not**:

- a direct-entry line,
- a standalone credit,
- a payment line,
- a refundable credit line, or
- the final tax after credits.

Its only job is to combine the two upstream page-2 nonrefundable credit lines before line 22 applies the zero floor.


So the surrounding page-2 flow is:


## 6. 2025 Guardrails

### 6a. Line 21 includes only lines 19 and 20

Do not add anything else here.

In particular, line 21 does **not** include:

- refundable credits such as `line 28` (ACTC), `line 29` (refundable AOTC), `line 30`, or `line 31` (Schedule 3 line 15);
- Schedule 2 Part II other taxes;
- withholding or estimated payments;
- direct Schedule 3 Part II amounts.

### 6b. Line 21 should not exceed line 18 in a correct computation

Because `line 19` and `line 20` are nonrefundable-credit amounts with their own upstream limitations (Schedule 8812 CLW-A + each Schedule 3 credit's own CLW), a correctly computed return should satisfy:

```text
Form1040.line21 <= Form1040.line18
```

If the engine ever produces `line21 > line18`, treat that as an upstream-credit bug or ordering problem, not a valid tax result. Pre-G-new-1-fix scenarios (CLW-A reading stale line 18 before FTC correction) could violate this; current code refreshes line 18 inside `correctLine17ForFtc` to prevent it.

This is a **soft constraint** — NOT runtime-validated. Documented as informal invariant.

### 6c. Line 21 itself is not floored

The floor happens on `line 22`, not on line 21:

```text
Form1040.line21 = line19 + line20                             (raw sum; ≥ 0 by addends ≥ 0)
Form1040.line22 = max(0, line18 - line21)                     (explicit floor at 0)
```

Both addends are non-negative so the sum is structurally non-negative even without an explicit floor.

## 8. Validation Rules

```text
Form1040.line21 >= 0                                          (STRUCTURAL — both addends ≥ 0)
Form1040.line21 = nz(Form1040.line19) + nz(Form1040.line20)   (STRUCTURAL — single addition)
Form1040.line21 <= Form1040.line18                            (SOFT — after correct upstream limiting)
Form1040.line21 stored as null when zero                      (CONVENTION)
```

Practical validation notes (per lock-in tests):

- if `line19 = 0` and `line20 = 0`, then `line21 = null` (test `line21_isNullWhenNoCreditsPresent` at `TaxReturnComputeServiceTest.java:24187`);
- if either upstream line is present, line 21 must equal their exact sum (test `line21_equalsLine19PlusLine20_ctcAndAotc` at line 24139);
- if line 22 = 0 (credits fully absorb tax), `line21 = line18` (test `line21_line22FlooredAtZeroWhenCreditsAbsorbAllTax` at line 24215).

## 9. Implementation Notes For This Project

This project documents the two feeder lines separately:

- `line 19` in `19.md`;
- `line 20` in `20.md`.

The line-21 implementation stays intentionally small (3-line inline addition):

```text
inside computeLine20ThroughLine24() at TaxReturnComputeService.java:26942:

  BigDecimal line19 = safeAmount(tac.getChildTaxCredit());
  BigDecimal line20 = safeAmount(tac.getOtherCreditsSchedule3());
  BigDecimal line21 = roundMoney(line19.add(line20));
  tac.setTotalCredits(line21.compareTo(BigDecimal.ZERO) > 0 ? line21 : null);
```

Call order (all inside same method):

1. compute `line 20` (read from Schedule 3 line 8);
2. compute `line 21` (line 19 + line 20);
3. compute `line 22` (max(0, line 18 − line 21));
4. compute `line 24` (line 22 + line 23).

The 4-field method handles lines 20 + 21 + 22 + 24 in a single pass.
