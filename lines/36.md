# Form 1040 (2025) — Line 36: Amount Applied to 2026 Estimated Tax

> Source of truth: 2025 Form 1040, 2025 Form 1040 Instructions.
> Verified 2026-06-14.

---

## 1. Line Identity

**Form 1040 line 36** label (2025):

```text
Amount of line 34 you want applied to your 2026 estimated tax
```

Exact formula:

```text
line36 = min(amountToApply, overpaid)    when user elects AND overpaid > 0
line36 = null                             otherwise
```

Line 36 is the **user-elected portion of the overpayment** that the taxpayer wants the IRS to credit toward 2026 estimated tax, instead of receiving as a 2025-return refund. The election is **irrevocable** once filed (IRS rule): the taxpayer cannot reverse the election by amending or filing a separate request.

## 2. Top-Level Formula

```text
Stage 1 (raw read, lines 27317-27326):
   line36 = roundMoney(applyNextYearData.amountToApply)
       WHEN applyNextYearData.electsApplyToNextYear == TRUE
        AND amountToApply > 0
   line36 = ZERO  otherwise (default)

Stage 2 (cap + setter, lines 27342-27346, inside refund branch):
   line36Capped = line36.min(overpaid)        // cannot exceed line 34
   if (line36Capped > 0)
       refund.setAmountAppliedToNextYear(line36Capped)
```

The two stages are **non-adjacent** in the code: Stage 1 runs before the refund-branch gate; Stage 2 runs inside the refund branch (`if (line33 > totalTax)`). They share state via the local `BigDecimal line36` variable.

## 6. Cap Rule — Line 36 ≤ Line 34

The hard structural rule is:

```text
line36 ≤ line34 (overpaid)
```

Enforced via `.min(overpaid)` at line 27343. If the user enters `amountToApply = $10,000` but the overpayment is only `$3,000`, the stored value is `$3,000` (capped to overpaid); the remaining `$7,000` is **silently dropped** — no advisory flag is emitted.

**UX gap (informational):** the silent cap is a potential UX issue — the user may believe they applied $10,000 when the actual credit was $3,000. The UI may surface this at the form-entry level, but the backend does not emit a `LINE_36_CAPPED` flag. The behavior is structurally correct (IRS rule) but the visibility is left to the UI.

## 7. IRS Rules (Informational)

### 7a. Irrevocability

Once the return is filed with a non-zero line 36, the election cannot be reversed. The IRS will not unwind the credit even via Form 1040-X (amended return). This is documented in the IRS instructions for line 36 but **not enforced in the backend** (the project does not implement amended-return processing).

### 7b. First-installment crediting

By default, the IRS credits the line 36 amount to the **Q1 2026 estimated-tax installment** (due April 15, 2026). The taxpayer can direct it to a later installment by attaching a written statement to the paper return, but the project does not implement an alternate-installment routing path — the default Q1 crediting is the assumed behavior.

Both rules are documented in IRS Form 1040 instructions; neither is structurally enforced in the backend.

## 8. Null Behavior — If-Gate-Around-Setter (Convention 1)

Line 36 uses the **if-gate-around-setter** Convention 1 mechanism:

```text
if (line36Capped > 0)
    refund.setAmountAppliedToNextYear(line36Capped)
// else: setter not called; field remains at default null
```

When the user did not elect (Stage 1 produced ZERO), the cap produces ZERO, the setter is never called, and `Refund.amountAppliedToNextYear` stays at default `null`. When the user elected but the overpayment was zero, the same outcome.

This is the same mechanism used by line 31 and lines 35b/c/d.

---

## 9. Effect on Line 35a

Line 36 directly reduces line 35a (refund amount):

```text
line35a = overpaid − line36Capped       (line 27349)
```

| Scenario | overpaid | user election | line36Capped | line35a |
|---|---|---|---|---|
| No election | 1000 | 0 (no elect) | 0 | 1000 (full refund) |
| Partial apply | 1000 | 300 | 300 | 700 (refund check) |
| Full apply | 1000 | 1000 | 1000 | 0 (no refund, all credited) |
| Over-apply (silent cap) | 1000 | 5000 | 1000 (capped) | 0 (no refund) |

When `line35a == 0` (full election), `refund.refundAmount` stays at default `null` via the line 27350 ternary-at-setter. The `Refund` object still has `overpaid` and `amountAppliedToNextYear` set; only the refund-amount field is blank.

## 14. Summary Rule

For tax year 2025:

```text
Stage 1: line36 = roundMoney(amountToApply)    when user elected AND amount > 0
         line36 = 0                              otherwise

Stage 2 (only inside refund branch):
         line36Capped = min(line36, overpaid)
         if line36Capped > 0:
             refund.amountAppliedToNextYear = line36Capped

Effect on line 35a:
         line35a = overpaid − line36Capped
```

Stored at `Refund.amountAppliedToNextYear`. Capped at line 34. Directly reduces line 35a. Election is informally irrevocable; backend does not enforce. Default IRS crediting is Q1 2026 (April 15, 2026).
