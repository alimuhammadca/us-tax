# Form 1040 (2025) — Line 37: Amount You Owe

> Source of truth: 2025 Form 1040, 2025 Form 1040 Instructions.
> Verified 2026-06-14.

---

## 1. Line Identity

**Form 1040 line 37** label (2025):

```text
Subtract line 33 from line 24. This is the AMOUNT YOU OWE.
For details on how to pay, go to www.irs.gov/Payments or see instructions.
```

Exact formula (IRS-canonical):

```text
line37 = line24 − line33    (computed only when line24 > line33)
```

Line 37 is the **amount-owed pivot** — the dollar amount by which total tax (line 24) exceeds total payments (line 33). It is mutually exclusive with line 34 (overpayment): exactly one of the two lines is populated per return (neither when line 33 == line 24).

---

## 2. Top-Level Formula (Pure vs. Stacked)

### 2a. Pure IRS-canonical value

```text
pure_line37 = roundMoney(totalTax − line33)    when totalTax > line33
```

This is the strict IRS-canonical value that satisfies the rule `line 37 == line 24 − line 33`. It is what the printed Form 1040 line-37 PDF box emits.

### 2b. Project-stacked value (what is stored)

```text
stored.amountOwed = pure_line37 + Form2210.totalPenalty       // when Form 2210 fires
stored.amountOwed = pure_line37                                 // when Form 2210 does NOT fire
```

The project stores the **combined** line-37-plus-line-38 amount in `AmountOwed.amountOwed` per the **stacking convention** documented in `AmountOwed.java:35-149` Javadoc. This is so the in-app "Amount you owe" summary shows the total remittance amount to the user.

The **pure** value is recoverable via `AmountOwed.getLineThirtySevenPure()` (`AmountOwed.java:145-149`):

```java
public BigDecimal getLineThirtySevenPure() {
    if (amountOwed == null) return null;
    if (estimatedTaxPenalty == null) return amountOwed;
    return amountOwed.subtract(estimatedTaxPenalty);
}
```

## 3. Compute Order and Wiring

`TaxReturnComputeService` two-stage wiring:

### Stage 1 — Pure line 37 (`computeLine31ThroughLine38`, lines 27440-27449)

```text
} else if (totalTax.compareTo(line33) > 0) {
    BigDecimal owed = roundMoney(totalTax.subtract(line33));
    AmountOwed amountOwed = form1040.getAmountOwed();
    if (amountOwed == null) {
        amountOwed = new AmountOwed();
        form1040.setAmountOwed(amountOwed);
    }
    amountOwed.setAmountOwed(owed);
}
```

After Stage 1, `amountOwed.amountOwed == pure_line37`.

### Stage 2 — Form 2210 penalty stack (lines 2073-2088)

```text
Form2210 form2210 = computeForm2210(form1040, priorYearTaxData, filingStatus);
if (form2210 != null && form2210.getTotalPenalty() != null
        && form2210.getTotalPenalty().compareTo(BigDecimal.ZERO) > 0
        && !"WAIVED".equals(form2210.getComputationMethod())) {
    AmountOwed ao = form1040.getAmountOwed();
    if (ao == null) {
        ao = new AmountOwed();
        form1040.setAmountOwed(ao);
    }
    ao.setEstimatedTaxPenalty(form2210.getTotalPenalty());
    BigDecimal currentOwed = safeAmount(ao.getAmountOwed());
    ao.setAmountOwed(roundMoney(currentOwed.add(form2210.getTotalPenalty())));
}
```

After Stage 2, `amountOwed.amountOwed == pure_line37 + form2210.totalPenalty`.

This 4-condition gate excludes WAIVED Form 2210 (Box A full waiver → `computationMethod == "WAIVED"`), zero penalty, or null Form 2210 (returned by helper when balanceDue < $1,000 or l9 ≥ l8).

## 5. Stacking Convention — Why Both Stages

The project keeps pure line 37 + line 38 stacked into the same `AmountOwed.amountOwed` field for one reason: the in-app summary shows the user a single "Amount you owe" number, which is `line 37 + line 38` for the actual remittance amount. The printed Form 1040 still satisfies the IRS rule by emitting the pure value at the line-37 PDF box and the penalty separately at the line-38 PDF box.

This is documented in `AmountOwed.java:35-149` Javadoc:

> **Form 2210 stacking convention:** the stored value of `amountOwed` carries the *combined* line 37 plus line 38 amount when Form 2210 fires. The pure IRS-canonical line 37 (the strict `line 24 − line 33`) can be recovered via `getLineThirtySevenPure()`. The printed Form 1040 line-37 PDF box displays the pure value; the in-app summary displays the stacked total.

This means the printed return is correct (IRS-canonical) AND the user-facing summary is intuitive ("here is the total you need to send").

## 6. Null Behavior — GATED-NOT-SET (Convention 1)

Pure line 37 uses the **GATED-NOT-SET** Convention 1 mechanism: when `line33 ≥ totalTax`, the `else if` branch at line 27440 is skipped, the setter is never called, and `AmountOwed` is not lazily initialized. The amount-owed branch is structurally cleaner than ternary-at-setter (no empty `AmountOwed` container is created on refund or balanced returns).

The Stage 2 stack site uses an **if-gate-around-setter** mechanism (Convention 1): when Form 2210 returns null OR the penalty is zero OR `computationMethod == "WAIVED"`, the setter is skipped. If pure line 37 was set in Stage 1, it remains untouched.

**Edge case:** if pure line 37 is null (refund return) but Form 2210 still fires with a non-zero penalty, Stage 2 lazy-creates the `AmountOwed` object and sets both `estimatedTaxPenalty` and `amountOwed` (the latter to just the penalty). This is unusual — Form 2210 normally only fires on owed returns — but the code handles it (lines 2080-2087). The Stage 2 gate at line 27551 (`balanceDue < $1,000 → return null`) prevents the obvious case.

## 7. Mutual Exclusivity with Line 34

```text
if (line33 > totalTax) {
    // Refund branch — line 34 set
} else if (totalTax > line33) {
    // Owed branch — pure line 37 set
}
// Implicit balanced branch: line33 == totalTax → NEITHER set
```

The mutual exclusivity applies to the **pure** values. When Form 2210 fires on a refund return (rare but possible), the stacked Stage 2 value may touch `AmountOwed.amountOwed` even though pure line 37 is null. Strictly speaking the IRS rule "line 34 and line 37 cannot both be populated" applies to the pure values; the stacking does not violate this since the printed Form 1040 emits the pure value at the line-37 PDF box (which is null on a refund return).

## 8. MFS Inheritance via M2 (Transitive)

Pure line 37 has no per-spouse code or MFS-specific logic:

- `line33` — M2 transitive via 25d + 26 + 32 sub-line MFS protection
- `totalTax` (line 24) — per-return; each MFS spouse files separately with their own totalTax

Stage 2 inherits via the upstream values plus the Form 2210 MFS-aware safe-harbor routing (see `lines/38.md` §5 for the isMfs tax-rule routing at line 27609). Neither stage has an MFS-leakage check.

| Invariant | Rationale / How Enforced |

## 12. Summary Rule

For tax year 2025:

```text
Pure line 37 (printed PDF, IRS-canonical):
    line37 = roundMoney(totalTax − line33)    when totalTax > line33
    line37 = null                              otherwise

Stored AmountOwed.amountOwed (in-app summary):
    stored = pure_line37 + Form2210.totalPenalty     when Form 2210 fires (not WAIVED)
    stored = pure_line37                              otherwise

Pure value recovery:
    AmountOwed.getLineThirtySevenPure() = amountOwed − estimatedTaxPenalty
```

Mutually exclusive with line 34 (overpayment) on the pure value. The stacked value may be touched on a refund return if Form 2210 fires (rare).
