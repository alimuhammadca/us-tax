### Test 7 — `Line 11a/11b — negative AGI is preserved when adjustments exceed income`

```
test('Line 11a/11b — negative AGI is preserved when adjustments exceed income')
```

**Purpose.** Locks in the "no zero-floor on AGI" rule — negative AGI is a legitimate IRS state (a taxpayer can have a Net Operating Loss that drives AGI below zero). Verifies the boundary case where the educator cap exactly equals the user's input ($300 entered, $300 cap, $300 deducted).

#### Inputs the test seeds

| Field | Value |
|---|---|
| W-2 entry: `wagesTipsOtherCompAmount` | $100 |
| `hadIncomeAdjustmentsForSchedule1` | `true` |
| `educatorExpensesLine11` | $300 (= the cap) |

#### IRS rule that applies

```
Form 1040 line 9 = $100  (tiny wages)

Schedule 1 line 11 = min($300, EDUCATOR_EXPENSES_CAP_PER_PERSON ($300)) = $300
                   (at boundary — no clamp message, no advisory)
Schedule 1 line 26 = $300

Form 1040 line 10 = $300
Form 1040 line 11a = $100 − $300 = −$200  (negative AGI preserved; no zero-floor)
Form 1040 line 11b = −$200
```

IRS rule (2025 Form 1040 Instructions, line 11a): *"Subtract line 10 from line 9. This is your adjusted gross income."* No zero-floor instruction. Negative AGI is valid and propagates to downstream forms.

#### Code path

`computeIncomeAdjustments.clampToEducatorCap($300, "TAXPAYER", ...)` returns $300 unchanged (`raw.compareTo(cap) <= 0`).
`buildAdjustments.subtractNonNegativeAllowNegative($100, $300)` returns −$200 (the function name promises sign preservation).

| # | Assertion | IRS-expected | Verdict |
|---|---|---|---|
| 1 | `income.totalIncome === 100` | $100 | **OK** |
| 2 | `adjustments.adjustmentsSchedule1 === 300` | $300 (at cap; no clamp) | **OK** |
| 3 | `adjustments.line11aAdjustedGrossIncome === -200` | −$200 (no zero-floor) | **OK** |
| 4 | `adjustments.line11bAmountFromLine11aAdjustedGrossIncome === -200` | −$200 | **OK** |
| 5 | `line11a === line11b` (invariant) | true | **OK** |

All 5 assertions match IRS-computed values.

| Gap | Severity | Recommendation |

| Does not test the OVER-CAP path. $300 is exactly at the cap; a $400 input should clamp to $300 plus emit the `SCHEDULE1_LINE11_EDUCATOR_EXPENSES_CAPPED_TAXPAYER` non-blocking advisory. ### Top 5 most-impactful gaps (punch-list)

These are the gaps most likely to let a real regression slip through:

1. **No e2e coverage for the four §17 non-overrideable blocking flags this round added.** `SCHEDULE1_LINE19A_ALIMONY_POST2018_AGREEMENT_DISALLOWED_TAXPAYER` / `_SPOUSE`, `SCHEDULE1_LINE21_STUDENT_LOAN_INTEREST_MFS_DISALLOWED`, `SCHEDULE1_LINE21_STUDENT_LOAN_INTEREST_DEPENDENT_DISALLOWED` are all registered, emitted, and enforced in `TaxReturnComputeService.computeIncomeAdjustments` — but no e2e test verifies the user actually sees the 409 response with the flag in the body. A regression that registered the flag but failed to emit it (silent over-deduction restored) would not be caught by the current suite. **Recommended:** add five tests using the raw `page.evaluate(fetch)` + 409-body pattern from `test_cases/8.md` GAP-E29-* — one per flag.

2. **No coverage of the line 21 $2,500 cap.** Today a taxpayer with $4,000 of 1098-E interest will silently get $2,500 instead — but no test asserts that. **Recommended:** add a test that seeds `studentLoanInterestDeductionLine21 = $4000` and asserts `adjustmentsSchedule1 === 2500`. Without this, removing the cap call would not regress the e2e suite.

3. **No coverage of line 11 educator over-cap path.** Test 7 uses exactly $300 (the cap boundary); no test seeds $400+ to verify (a) the value is clamped to $300, (b) the `SCHEDULE1_LINE11_EDUCATOR_EXPENSES_CAPPED_TAXPAYER` non-blocking advisory fires. **Recommended:** see the row in Test 2's gap table.

4. **No spouse-side coverage for any of the new Line 10 rules.** All current tests use only the taxpayer side. The spouse-side aggregation (`addNonNull(taxpayer, spouse)`), spouse-side post-2018 alimony block, spouse-side `_SPOUSE` flag code variants, and the dual-educator MFJ $600 combined cap are all unexercised. **Recommended:** add a paired spouse-side test per new flag.

A regression that swapped line 11 ($100) with line 14 ($20) at the storage layer would not be detected because the sum is invariant under field swaps. **Recommended:** add per-line `expect(schedule1.adjustments.educatorExpenses).toBe(100)` etc.
