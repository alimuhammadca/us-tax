Audit date: 2026-06-08
Method: For every `test(...)` in the spec — (1) read the inputs the test seeds, (2) compute by hand what the IRS rule requires from those inputs (Form 8995 / 8995-A; IRC §199A; Publication 17 chapter on QBI), (3) read the Java compute path that handles those inputs, (4) compare against the `expect(...).toBe(...)` assertions, (5) classify each assertion **OK** / **MISMATCH** / **GAP**.

Shared 2025 references:
- Form 8995 (simplified) — used when `taxable_income_before_qbi ≤ $197,300` (Single) and the taxpayer is not a cooperative patron.
- Form 8995-A (full) — used above the threshold or for cooperative patrons.
- Single standard deduction (2025): **$15,750**.
- QBI deduction = `min(tentative, taxable_income_limit)`, where:
  - `tentative = 20% × max(0, net_QBI) + 20% × max(0, net_REIT_and_PTP)`
  - `net_QBI = QBI_total − prior_year_QBI_loss_carryforward`
  - `net_REIT_and_PTP = REIT_and_PTP_dividends − prior_year_REIT_PTP_loss_carryforward`
  - `taxable_income_limit = 20% × max(0, taxable_income_before_QBI − net_capital_gain_and_qualified_dividends)`
  - `net_capital_gain` per Form 8995 spec = Schedule D long-term net (floored at zero) + qualified dividends.

Backend computation entry point: `TaxReturnComputeService.computeLine13a(...)` at `src/main/java/com/ustax/microservices/TaxReturnComputeService.java`.

### Hand computation per IRS rule
The IRS **would** compute a QBI deduction for Schedule C / F self-employment income — IRC §199A(d)(1)(A) explicitly includes any qualified trade or business other than performing services as an employee, so a Schedule C net profit is QBI. Form 8995 line 1 column (c) accepts QBI from Schedule C / F.

Under strict IRS rules with `Schedule C QBI = $100`, the deduction would be `20% × $100 = $20` (assuming taxable-income limit doesn't bind), and the final line 13a = `$20`.

**However**, this application **declines to support that path** (per `CLAUDE.md`: "Self-employment (Schedule C / SE / F) is out of scope"). The application's documented behavior is to refuse to compute and raise a blocking flag so the user knows the return cannot be filed through this product. This is correct from an application-scope standpoint — silently zeroing the deduction would risk taxpayer harm.

So the IRS-correct outcome is **$20 QBI deduction**, and the application's documented-correct outcome is **blocked with a flag**. These differ, but the divergence is intentional and within stated scope.

### Java compute path
- `validateQbiThresholdPath` / supplemental-form gate at ~`TaxReturnComputeService.java:6040` reads `hasScheduleCOrScheduleFQbiSources` from the QBI form.
- When `true` → emits `TaxReturnFlag` with code `LINE13A_SELF_EMPLOYMENT_OUT_OF_SCOPE_TAXPAYER`, `blocking = true`, registered in `NonOverrideableFlags.CODES`.
- Sets `hasBlockingIssue = true` → `computeLine13a` returns `null` (line ~6060: `if (!hasAnySupportedAmount || taxableIncomeBeforeQbi == null || hasBlockingIssue) return null`).
- Caller sets neither `form8995` nor `form8995A` on the output.
- The compute endpoint returns `200` with the flag in the payload (blocking flags ride along with the 200 — the 409 path is only when `overrideFlags=false` AND a non-overrideable flag fires; the §17 blocker pattern documented in `MEMORY.md` confirms this).
