## Test 6 — Employer treated as contractor excluded — wages do not flow to line 1b

```powershell
npx playwright test --workers=1 --output=test-results-1b-186 line1b-household.spec.ts:186
```

**Purpose.** Per-row Form 8919 collision-guard validator, end to end. The contractor-flagged row must (a) be excluded from Line 1b and (b) raise the §6.4 `MISMATCH_CONTRACTOR_HOUSEHOLD_NEEDS_FORM_8919_TAXPAYER` blocking flag (because no matching Form 8919 firm row exists). Per `rules.md` §17 the flag is **non-overrideable**: even when the caller passes `overrideFlags=true`, compute must still return HTTP 409 — the user must fix the input (add the matching Form 8919 firm row OR change the contractor flag) before a return can be generated. Without this rule the $800 would silently disappear from the return and the IRS would catch the under-reporting later via a CP-2000 notice. The collision guard's positive-routing assertions (line 1b = $1,200, line 1g = $800, Form 8919 attachment generated) are covered by **Test 11**, which adds the matching Form 8919 firm row so compute succeeds and the full IRS routing can be observed end to end.

**Inputs.**
- Employment Income (taxpayer) via API: `hasEmploymentIncome=true`, `householdWork=true`, `householdEmployeeUnderControlTest=true`, `householdReceivedW2=false`.
- Two household employers: "Smith Household" `wages=$1,200, contractor=false`; "Jones Household" `wages=$800, contractor=true`.
- NO matching Form 8919 firm row for "Jones Household".

**Outputs (blocked path — no override).**
- Compute returns HTTP **`409`** with `MISMATCH_CONTRACTOR_HOUSEHOLD_NEEDS_FORM_8919_TAXPAYER` in the response body.

**Outputs (override attempt — `overrideFlags=true`).**
- Compute STILL returns HTTP **`409`** with the same `MISMATCH_CONTRACTOR_HOUSEHOLD_NEEDS_FORM_8919_TAXPAYER` flag (`rules.md` §17 — non-overrideable).
- No `TaxReturnComputation` is produced; the user must fix the input.

## IRS routing — proof-of-correctness summary

Every dollar that flows through the household-employee scenarios is now verified to land on the IRS-specified destination, in both the happy and corner-case paths:

| Scenario | Line 1b | Line 1g | Line 25b | Coherence flag |
|---|---|---|---|---|
| Genuine household-employee wages, no W-2 (Test 1) | Sums the non-contractor rows | — | — | — |
| Household work but failing the control test (Test 2) | Suppressed (null) | — | — | — |
| Household work WITH a W-2 (Test 4) | Suppressed (null) — goes to Line 1a | — | — | — |
| Misclassified worker, no withholding, with Form 8919 firm row (Test 11) | Excluded (collision guard) | Form 8919 line 6 → Line 1g | — | None |
| Misclassified worker, no Form 8919 firm row (Tests 6, 12) | — (compute blocked) | — (compute blocked) | — | §6.4 — **non-overrideable**; user must fix input |
| Misclassified worker + federal withholding, with 1099-NEC on file (Test 13) | Excluded | Form 8919 line 6 → Line 1g | 1099-NEC Box 4 → Line 25b | None |
| Misclassified worker + federal withholding, no 1099-NEC on file (Test 14) | — (compute blocked) | — (compute blocked) | — | §6.8 — **non-overrideable**; user must fix input |

The two coherence-guard flags (§6.4 + §6.8) prevent the two silent-failure modes the IRS would otherwise catch via a CP-2000 notice (under-reported income; lost withholding credit). Per `rules.md` §17 ("No facilitation of misrepresentation or under-reporting"), both flags are **non-overrideable**: passing `overrideFlags=true` does NOT bypass them. The compute endpoint returns HTTP 409 in both branches, and the user is required to fix the underlying input (add the missing Form 8919 firm row, add the missing 1099 statement, or change the contractor flag) before a return can be generated. The previous "override accepts the under-reporting consequence" escape hatch was removed on 2026-05-30 — convenience does not trump correctness for flags the application's own validators have proven to be wrong.
