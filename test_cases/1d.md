A note on the 2026-05-20 form restructure: the Medicaid waiver form now asks the home-care-business question FIRST (sections 3–6 are hidden until it is answered), and the earned-income election LAST (so the user has seen the qualified amounts before electing). ## Resolution log (line 1d audit)

The five items below were originally surfaced as forward-looking observations during the 2026-05-30 audit. As of 2026-05-31 **all five are resolved end-to-end** (backend + frontend + tests + docs where applicable). The original observation text is retained verbatim under each item alongside its resolution record so future readers can see what the concern was and how it was addressed. The line 1d audit currently has **zero open observations and zero open implementation gaps**.

1. **`MEDICAID_WAIVER_SCHEDULE_C_*` promoted to §17 non-overrideable on 2026-05-30.** Added to `NonOverrideableFlags.CODES`. The application knows Line 1d should NOT be populated when the user runs a home-care business — the income belongs on Schedule C (which this application does not produce). Filing as-is would leave Line 1d empty AND produce no Schedule C either, so the entire Medicaid waiver income would be missing from the return — the IRS would catch the gap via a CP-2000 document-matching notice. The flag message was rewritten to §17 format (named consequence + IRS reaction + two concrete resolutions, no "proceed anyway"). Test 5 rewritten to use `computeReturnRaw` (asserts 409 + flag with AND without override) rather than the prior UI-confirm-dialog assertion. Test 5 continues to pass.
2. **Test 11 picker pre-fill double-count resolved on 2026-05-30.** Fixed via a combination of (a) frontend picker fix and (b) backend compute-time blocking flag:
   - **Frontend picker fix.** `pickStatement` in `form-medicaid-waiver-{taxpayer,spouse}.component.ts` no longer pre-fills `taxablePaymentsNotInW2Box1` for the `w2-box1` case. The picker creates the new entry with payer metadata only (`payerName`, `payerEIN`, `sourceType = W-2`). The 1099 cases (`1099misc-box3`, `1099misc-box6`, `1099nec-box1`) keep their taxable pre-fill because 1099 amounts are genuinely NOT in W-2 box 1, so routing to `taxablePaymentsNotInW2Box1` → Line 1d is IRS-correct. Test 11 updated to assert the new pre-fill behavior (both amount fields stay blank).
   - **Backend compute-time blocking flag.** New flag `MEDICAID_WAIVER_LINE1D_TAXABLE_OVERLAPS_W2_BOX1_{TAXPAYER,SPOUSE}` fires when `sourceType = W-2` AND matched W-2 Box 1 > 0 AND `taxablePaymentsNotInW2Box1 >= matched Box 1`. Added to `NonOverrideableFlags.CODES` per §17 — filing as-is would over-report wages by at least the Box 1 amount and trigger a CP-2000 document-matching notice. The flag is the safety net for (a) existing user data created via the old picker behavior and (b) future cases where a user manually re-introduces the smoking-gun pattern. A new e2e test (`save-vs-compute: W-2 source with taxable >= matched-W-2 box 1...`) asserts the 409 + flag with AND without `overrideFlags=true`. The `taxable < Box 1` case is allowed without flag (could be a legitimate split where the W-2 covers some wages and an additional separate payment is recorded on the form).
