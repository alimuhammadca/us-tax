The `OK` / `MISMATCH` / `GAP` codes mean:

- **OK** — the assertion matches what the IRS rule would compute.
- **MISMATCH** — the assertion expects a value the IRS rule says is wrong; the test would pass against buggy code.
- **GAP** — the assertion is silent on an output the IRS rule says should also be populated, or the assertion is loose (e.g., `> 0` when a specific value is computable).

### Why Fix #3 was reverted

The fourth proposed change added `expect(computation?.form1040?.income?.taxableIraDistributions).toBe(500)` to Test 5. While the IRS rule does route IRA / SEP / SIMPLE corrective distributions to line 4b, the application's IRA compute path (`computeIraDistributions` in `TaxReturnComputeService.java:7334`) requires:

1. The 1099-R to have a `recipientTIN` matching the taxpayer or spouse Social Security Number for attribution, AND
2. The user to have opened the `ira-income-taxpayer` (or `ira-income-spouse`) personal form and indicated they had IRA distributions.

Test 5 has neither. Its 1099-R is unattributed (`recipientTIN` is blank), and no IRA personal form is opened. So the `taxableIraDistributions` field stays null. Forcing the assertion would have caused Test 5 to fail.

The proper home for verifying line 4a / 4b routing is `line4abc-ira-income.spec.ts`, which already covers it correctly at lines 182 / 184 / 284 / 285 with the full IRA personal-form setup. Test 5 in `line1h-other-earned-income.spec.ts` correctly scopes itself to the line 1h EXCLUSION only. A comment in the test now records this scope boundary.
