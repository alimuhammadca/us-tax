**Purpose.** Verifies the IRS caveat (Pub 596 + lines/1i.md §6 + Computations/1i.md §1.3): the combat-pay election INCREASES earnedIncome but NOT AGI, so for filers in the EIC PLATEAU (where election pushes them into PHASEOUT) the with-election credit is SMALLER than the without-election credit. **Inputs.**
- `you`: SSN `111-11-1111`, DOB `1980-06-01`.
- W-2: `wages: 9000`, `box12Entries: [{ code: 'Q', box12Amount: 5000 }]`.
- No 1099-INT (AGI = $9,000 = wages only).

- First compute: `combat-pay-taxpayer` NOT saved.
- Second compute: `savePersonalFormApi(page, 'combat-pay-taxpayer', { electCombatPay: true })`.

**IRS hand calculation:**

Without election:
1. Earned income = $9,000.
2. AGI = $9,000.
3. Childless single: $9,000 in plateau ($8,490 < $9,000 ≤ $10,620) → creditFromEarned = $649.
4. AGI same → creditFromAGI = $649.
5. **EIC without election = min($649, $649) = $649**.

With election:
1. Earned income = $9,000 + $5,000 = $14,000.
2. AGI = $9,000 (combat pay non-taxable → not in AGI).
3. Childless single: $14,000 > phaseoutStart $10,620, $14,000 < phaseoutEnd $19,104 → PHASEOUT.
4. creditFromEarned = $649 - ($14,000 - $10,620) × 0.0765
                   = $649 - $3,380 × 0.0765
                   = $649 - $258.57
                   = $390.43 → floor → **$390**.
5. creditFromAGI = $649 (AGI $9,000 still plateau).
6. **EIC with election = min($390, $649) = $390**.

Comparison: $390 < $649 → directional assertion `with < without` holds.

**Java compute path.** Two consecutive `computeLine27aEIC` calls. First call: `nontaxableCombatPayElection = null` → earnedIncome $9,000 → plateau → $649. Second call: `nontaxableCombatPayElection = $5,000` (sourced from line 1i via Form1040.income) → earnedIncome $14,000 → phaseout branch → `$649 - ($14,000 - $10,620) × 0.0765` = $390.43 → floor → $390.

**Per-assertion triage.**

| Line | Assertion | Computed value | Triage |
|---|---|---|---|
| 328 | `expect(eicWithoutElection).toBeGreaterThan(0)` | $649 (> 0) | ⚠️ OK but **GAP** — directional `> 0`; exact-value pin (`.toBe(649)`) would be tighter. Per `feedback_e2e_exact_value_pins` this is the kind of assertion that hides hand-calc errors. |

The directional `< without` check at line 337 IS the test's purpose (the caveat) and can stay alongside the exact pins as a redundant safety net. Test comment justifies the directional assertion but the project's `feedback_e2e_exact_value_pins` rule explicitly tightens this kind of pattern.
