## Why no dedicated tests exist

Line 27b is structurally a degenerate constant in the current implementation. The triggers are:

1. The IRS canonical rule (`XLS/Computations/27b.md §1b`) requires THREE conditions to be true before the checkbox is checked: (Category A clergy OR Category B church employee income) AND filing Schedule SE AND Schedule SE line 4c overlaps Form 1040 line 1z.
2. The project is policy-blocked at Gap G9 (Schedule SE / self-employment out of scope per `outstanding.md`). The Schedule SE intake form does not exist. The clergy declaration form does not exist. The overlap declaration mechanism does not exist.
3. With those intake fields absent, none of the three conditions can ever evaluate to true. The implementation therefore hardcodes the value to `false` everywhere it appears:
   - Backend helper at `TaxReturnComputeService.java:26831` and `:27339` — two inline comments documenting the constant-`false` policy decision. Zero compute code.
   - Frontend renderer at `form-tax-return-1040.component.ts:820` — explicit `values['line27b_clergy_schedule_se'] = false;`.
4. A unit test or end-to-end test of the form "send empty inputs → expect line 27b false" would assert the same thing whether the code worked or not. There is no input combination that could produce `true`; therefore there is no scenario that distinguishes correct from incorrect behaviour.

This matches the §11 inventory in `XLS/Computations/27b.md`:

> **End-to-end tests:** ⚠️ None needed for the degenerate constant.

## Why the 37 line 27a tests do NOT cover line 27b

The 37 tests in `tests/line27a-earned-income-credit.spec.ts` exercise the **compute** pipeline (the call to `/api/tax-return/compute` and the JSON response under `form1040.payments`). They do NOT exercise the **PDF render** pipeline (the client-side `pdf-lib` call that writes Acroform field values). The render code at `form-tax-return-1040.component.ts:820` runs only when the user clicks "Download PDF" in the UI; no e2e test triggers that action.

Concretely:
- Every line 27a test calls `computeReturnApi(page, { overrideFlags: true })` which returns a JSON payload.
- That payload contains `form1040.payments.earnedIncomeCredit` (the line 27a value) — which the line 27a tests assert against.
- The payload does NOT contain a `line27b_clergy_schedule_se` field. That field exists only at the PDF render layer, written from the constant `false` literal in `form-tax-return-1040.component.ts:820`.
- A regression that changed the literal to `true` (for example, "Line 27b ignored as no-op — always check the box") would still let all 37 line 27a tests pass.

Therefore the 37 line 27a tests provide ZERO direct or indirect coverage of the line 27b render path.

### Why this is a real gap (not merely an audit artifact)

The frontend renderer literally has the line:

```typescript
values['line27b_clergy_schedule_se'] = false;
```

This single character (`false`) being changed to `true` (whether by an erroneous refactor, a copy-paste mistake from another field, or an automated rename tool) would silently break correctness for every tax return the project generates — and no e2e test in the entire codebase would fail.

The IRS canonical for "no clergy, no Schedule SE, no overlap" is `false`. Producing `true` would mean every Form 1040 PDF asserts that the taxpayer is a clergy member filing Schedule SE with overlap, which is incorrect for every return the project currently supports.

## Closing the GAP in the no-tests-today reality

There are two pragmatic options to close the current gap without waiting for Gap G9:

### Option A — Add a single render-layer assertion to an existing line 27a test

Inside any existing line 27a test (for example test 1: `childless single filer in phase-in range receives EIC`), after the compute call, render the PDF via the same `form-tax-return-1040.component.ts` code path and inspect the rendered AcroForm value for `line27b_clergy_schedule_se`. Assert `false`.

Cost: one new helper function in `tests/helpers/` to render the PDF via the client-side path (currently no e2e helper does this); a few lines of assertion in one test. Closes the regression risk for the literal flip from `false` to `true` in `form-tax-return-1040.component.ts:820`.

### Option B — Add a TypeScript unit test for the renderer

Skip Playwright entirely. Instantiate the `Form1040RenderingService` (or whichever class owns the `values[...]` map population) with a fake `TaxReturnComputation` and assert that `values['line27b_clergy_schedule_se']` is `false`. This bypasses the full e2e overhead.

Cost: a few lines in a `.spec.ts` file under the UI's existing unit-test runner. Closes the same regression risk at lower cost.

Both options are deferred-decision items: neither is required by IRS canonical, and the risk being mitigated is a programming defect (literal-flip in the constant) rather than an IRS-correctness gap. Per `feedback_principled_diagnosis_over_test_tweaking`, the gap is logged here without auto-closing; the user decides whether the regression risk justifies the test investment.
