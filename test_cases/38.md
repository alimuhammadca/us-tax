## Line 38 in one sentence

Form 1040 line 38 (the **estimated tax penalty**) is the Form 2210 underpayment penalty stacked onto `AmountOwed.estimatedTaxPenalty`. It fires only when (a) balance due ≥ $1,000 per Internal Revenue Code §6654(e)(1), (b) neither safe harbor is met (90% of current-year tax OR 100% / 110% of prior-year tax per Internal Revenue Code §6654(d)(1)(B-C)), and (c) Box A full waiver is NOT checked. The four-period regular-method computation per Internal Revenue Code §6654(d) divides the required annual payment into four equal installments and computes a daily-rate penalty for each quarter's underpayment using the 2025 annual rate of 7% (federal short-term + 3% per Internal Revenue Code §6621).

The project's Form 2210 stacking convention (`lines/33.md` §4 + `38.md` Section 2b) adds the line-38 penalty on top of `AmountOwed.amountOwed` (the line-37 storage) so the user-facing total carries both numbers. The IRS-canonical separation is preserved at the PDF emit layer per today's `37.md` Gap D + G closure: the line-37 PDF box shows the pure value (`amountOwed − estimatedTaxPenalty`) while the line-38 PDF box shows the standalone penalty.

The `38.md` audit (2026-06-14) closed three actionable gaps: Gap G (UI hint distinguishing `paymentNAmount` Form 2210 timing from `installmentNAmount` line-26 estimated tax payments), Gap H (yearly-refresh documentation for the `F2210_PENALTY_ANNUAL_RATE` constant), and Gap I (two new Java unit tests pinning the $1,000 trigger boundary at the compute layer). Four gaps remain documented out-of-scope by policy (Gap A Schedule AI; Gap B Box D actual withholding dates; Gap C Box C annualized income method; Gap D Form 2210-F farmers/fishermen).

## Project conventions — the four-stage gate + stacking

> **Stage 1 — Trigger gate** (`TaxReturnComputeService.computeForm2210:27548-27553`): when balance due < $1,000, return null (no Form 2210 generated; line 38 stays blank).
>
> **Stage 2 — Part I required-annual-payment** (`:27555-27624`): compute line 1 (Form 1040 line 22) + line 2 (Schedule 2 line 21) − line 4 (refundable credits) = line 5; then line 6 = 90% × line 5 and line 7 = 100% (or 110% when prior-year AGI > $150k / $75k MFS) × prior-year tax. Required annual payment (line 8) = min(line 6, line 7) when prior-year tax is known; else line 6 alone.
>
> **Stage 3 — Safe-harbor check** (`:27642-27646`): when total payments (withholding + Form 2210 estimated payments) ≥ required annual payment, set `computationMethod = "NO_PENALTY"`, `totalPenalty = 0`, and return — Form 2210 fires diagnostically but the stacking gate skips it.
>
> **Stage 4 — Waiver short-circuits** (`:27648-27662`): when Box A (full waiver) is checked, set `computationMethod = "WAIVED"`, `totalPenalty = 0`, and return. Box B (partial waiver) and Box E (joint-separate mismatch) flag but do NOT zero the penalty (project convention per `lines/2210.md`).
>
> **Stage 5 — Four-period regular-method loop** (`:27664-27717`): compute per-quarter underpayment × daily rate × days underpaid. Carry-forward surplus to next period.
>
> **Stage 6 — Storage** (`:27719-27726`): `totalPenalty = roundMoney(sum of period penalties)`. Stacking onto `AmountOwed` happens AFTER `computeForm2210` returns, at `TaxReturnComputeService:2073-2088`.

**Today's `38.md` Gap I closure** added two Java unit tests at the compute layer pinning Stage 1 (the $1,000 trigger boundary) from both sides — `form2210_noPenalty_balance_due_under_1000_boundary` and `form2210_fires_when_balance_due_over_1000_and_safe_harbor_missed`. The e2e tests in this cluster pin the same boundary at the request layer for defense in depth.
