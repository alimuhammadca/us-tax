## Test 1 — Child qualified dividends flow to parent line 3a and 3b — line 3c checkboxes set

`line8814-child-interest-dividends.spec.ts:360`

**Purpose (Line 3c lens):** Core positive case. Verifies BOTH Line 3c checkboxes fire when Form 8814 line 9 > 0 (the symmetric-trigger contract). Also verifies the upstream paired Line 3a and Line 3b symmetric arithmetic.

**Inputs:** One dependent + one `child-interest-dividends` entry — `line1aTaxableInterest = 1200`, `line2aOrdinaryDividends = 2000`, `line2bQualifiedDividends = 800`. No parent dividends.

**IRS rule for Line 3c:**
- Per 2025 Form 8814 line 9 instructions: "Include the amount from line 9 on the parent's Form 1040 line 3a (Qualified dividends) AND on line 3b (Ordinary dividends). Check the appropriate boxes on Form 1040 line 3c."
- Form 8814 line 4 = $1,200 + $2,000 + $0 = $3,200; line 5 base = $2,700; line 6 = $500; line 7 ratio (qualified) = $800 / $3,200 = 0.25; line 9 = $500 × 0.25 = $125.
- $125 > 0 → BOTH Line 3c checkboxes must be checked.

**Hand-compute Line 3c:**
- `form8814_qualified_dividends_total` = $125.
- `line3c_child_dividends_in_line3a` = TRUE.
- `line3c_child_dividends_in_line3b` = TRUE.

**Code path:**
- `buildForm8814` at `:9829-9849` computes line 9 = $125 (stored on the Form 8814 instance).
- `prepare()` at `:406-408` aggregates `form8814QualifiedDividendsTotal = $125`.
- `computeDividendIncome` at `:7621` derives `hasChildQualifiedDividends = true`.
- `:7752-7753` returns `Boolean.TRUE` for both checkbox fields on the `DividendComputation` record.
- `computeInterestIncome` at `:7156-7157` propagates them through to `Income.line3cChildDividendsInLine3a` / `..InLine3b`.

Matches IRS.

**Assertions touching Line 3c:**

**Line 3c GAPs:** None — both checkbox assertions plus paired arithmetic lock-ins are in place. The new render-layer test (Test 2 below) further hardens this case.

## Test 3 — Child with no qualified dividends — line 3c checkboxes not set

`line8814-child-interest-dividends.spec.ts:490`

**Purpose (Line 3c lens):** Core negative case. Verifies BOTH Line 3c checkboxes stay null when Form 8814 line 9 = 0 (child has only interest income, no dividend income). Symmetric trigger means "neither" not "one or the other."

**Inputs:** One dependent + one `child-interest-dividends` entry — `line1aTaxableInterest = 3000`. No dividends, no cap gain distributions.

**IRS rule for Line 3c:**
- Per 2025 Form 8814 line 9 instructions, line 9 = `line 6 × line 2b / line 4`. With `line 2b = 0` (no qualified divs), line 9 = $0 regardless of how much interest the child has.
- $0 is not > 0 → BOTH Line 3c checkboxes must be unchecked (blank — i.e., null in TRUE-or-null semantics).
- The child's $3,000 of interest above the $2,700 line-5 base routes to Schedule 1 line 8z via Form 8814 line 12, NOT to parent's Line 3a/3b → there is nothing to disclose on Line 3c.

**Hand-compute Line 3c:**
- `form8814_qualified_dividends_total` = null (or 0).
- `line3c_child_dividends_in_line3a` = null (unchecked).
- `line3c_child_dividends_in_line3b` = null (unchecked).

**Code path:**
- `buildForm8814` at `:9835`: `hasPositiveAmount(line2b) || hasPositiveAmount(line3)` is false → line9 = 0 → stored as null at `:9847`.
- `prepare()` at `:406-408`: aggregate stays null.
- `computeDividendIncome` at `:7621`: `hasChildQualifiedDividends = hasPositiveAmount(null) = false`.
- `:7752-7753`: returns `null` for both checkbox fields (TRUE-or-null pattern).

Matches IRS.

**Assertions touching Line 3c:**
1. `income?.line3cChildDividendsInLine3a ?? null` is `null` (`:502`) → **OK**.
2. `income?.line3cChildDividendsInLine3b ?? null` is `null` (`:503`) → **OK**.
3. `income?.qualifiedDividends ?? null` is `null` (paired Line 3a parent-isolation lock-in).
4. `income?.ordinaryDividends ?? null` is `null` (paired Line 3b parent-isolation lock-in).
5. `additionalIncome.otherIncomeForm8814 === 300` (confirms child's $300 above-base interest routed to Schedule 1 line 8z, not anywhere near Line 3a/3b — the safety net for the Line 3c blank state).

**Line 3c GAPs:** None — the negative case is fully covered at the JSON layer. A complementary DOM-render lock-in for the unchecked state could be added in a future tightening pass (would assert the `<input>` elements are attached but NOT `[checked]`), but no functional gap exists since the Test 2 positive render lock-in proves the rendering chain is correct, and this test proves the negative trigger condition holds.

## Test 4 — Two children with qualified dividends — parent lines 3a and 3b aggregate both

`line8814-child-interest-dividends.spec.ts:542`

**Purpose (Line 3c lens):** Multi-child aggregation. Verifies the symmetric trigger fires under multi-child aggregation when the SUM of per-child line 9 amounts > 0. Even if no single child individually had a positive line 9, an aggregate > 0 still fires both boxes.

**Inputs:** Two dependents + two `child-interest-dividends` entries — each child has `line2aOrdinaryDividends = 3000`, `line2bQualifiedDividends = 3000`. Per child: line 4 = $3,000 + $0 interest + $0 cap gain = $3,000; line 6 = $300; line 7 ratio = $3,000 / $3,000 = 1.0; line 9 = $300. Aggregate line 9 = $600.

**IRS rule for Line 3c:**
- Per 2025 Form 8814 instructions, one Form 8814 per child. Each child's line 9 contributes to the parent's lines 3a and 3b independently.
- Aggregate `form8814_qualified_dividends_total` = $300 + $300 = $600 > 0 → BOTH Line 3c checkboxes fire.

**Hand-compute Line 3c:**
- `form8814_qualified_dividends_total` = $600.
- `line3c_child_dividends_in_line3a` = TRUE.
- `line3c_child_dividends_in_line3b` = TRUE.

**Code path:**
- `buildForm8814` runs twice (one per child), each producing line9 = $300.
- `prepare()` at `:406-408` sums to $600.
- `computeDividendIncome` at `:7621-7625` adds $600 symmetrically to BOTH Line 3a and Line 3b.
- `:7752-7753` returns `Boolean.TRUE` for both checkbox fields.

Matches IRS.

**Assertions touching Line 3c:**
1. `income?.line3cChildDividendsInLine3a === true` (`:547`) → **OK** (aggregation lock-in for box 1).
2. `income?.line3cChildDividendsInLine3b === true` (`:548`) → **OK** (aggregation lock-in for box 2).
3. `income?.qualifiedDividends === 600` (paired Line 3a aggregate).
4. `income?.ordinaryDividends === 600` (paired Line 3b aggregate; symmetric value to Line 3a).
5. `list21[0].line9QualifiedDividends === 300`, `list21[1].line9QualifiedDividends === 300` (per-child line 9 isolation — proves aggregation is correct rather than one child contributing $600 and the other $0).
6. `sch1Addl21.otherIncomeForm8814 === null` (confirms line 12 = $0 per child since line 6 fully consumed by line 9).
7. `form6251 === null` (orthogonal).

**Line 3c GAPs:** None — multi-child aggregation is fully locked in.
