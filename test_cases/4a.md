## Test 1 — Taxpayer IRA form blocks save when no IRA 1099-R statements are uploaded

`line4abc-ira-income.spec.ts:111`

**Purpose (Line 4a lens):** UI-side gate that prevents the user from declaring "I had IRA distributions and uploaded all 1099-Rs" when zero IRA-flagged 1099-R statements actually exist in the user's `1099-r` collection. Protects Line 4a/4b from being computed against missing source data.

**Inputs:** `seedRequiredDob` + `ensureStatementSelected` for `1099-r` *registers the form selection* but does **not** create any 1099-R entries. IRA form fields set:
- `hadIraDistributions=Yes`
- `hasUploadedAtLeastOneIra1099RStatement=Yes`
- `confirmAllReceived1099RUploaded=Yes`
- `received1099R=Yes`, `uploaded1099R=Yes`

User then clicks **Save**.

**IRS rule:**
- IRS 2025 Form 1040 instructions p. 27 (Line 4a/4b): "If you received a Form 1099-R, see the instructions for those forms first." The taxpayer is required to report every Form 1099-R received.
- Internal Revenue Code §408(d)(1): traditional IRA distributions are includible in gross income to the extent they exceed the owner's basis.
- The IRS rule itself does not specify a UI gate — this is a defensive application gate that prevents the user from getting into an inconsistent state where the IRA form claims distributions exist but no statement is present. Without the gate, Line 4a/4b would compute to `null` (no statements → no gross), producing a return that silently omits real IRA income.

**Hand-compute Line 4a:** Given the inconsistency, there is nothing for the IRS rule to compute against — the application correctly refuses to persist the form rather than let an empty IRA flow downstream.

**Code path:**
- IRA Income form has `statementUploadCheck` block in `4abc-ira-income-taxpayer.yaml` (per `C:\us-tax\yamls\`) that gates `hasUploadedAtLeastOneIra1099RStatement = Yes` on the actual statement count.
- UI directive `statement-form` (in `us-tax-ui/src/app/directives/`) cross-validates the answer against the live `1099-r` statement collection and disables the Save button (the `PUT /api/personal/ira-income-taxpayer` request never fires).
- Test races a 2-second `waitForResponse` predicate; expects it to resolve to `null` (timeout = no PUT happened).

**Assertions touching Line 4a:**
1. `await saveResponse === null` (`:131`) — the PUT did not fire within 2 seconds → **OK**. The save-gate prevents an inconsistent IRA personal form from being persisted, indirectly protecting Line 4a/4b from a "phantom IRA distribution" state.

**Line 4a GAPs:**
- **GAP-T1-1 (low priority): Closed 2026-06-03.** Network-level lock-in that distinguishes the two `onSubmit` early-return branches at `form-ira-income-taxpayer.component.ts:893-901`. Two preconditions: (1) wait for the IRA 1099-R count pill to settle on `"IRA 1099-R uploaded: 0"` — proves ngOnInit's initial refresh GET has completed, so any subsequent GET must come from `onSubmit`. (2) install BOTH waiters before the click — PUT to `/api/personal/ira-income-taxpayer` (must NOT fire) AND GET to `/api/statements/1099-r/entries` (MUST fire from onSubmit's refresh call). Only the correct `isValid() returns false → return early` path satisfies both. The silent `if (this.saving || this.loading) return` branch fails the refresh-GET assertion; a regression that drops through to fire the PUT fails the PUT assertion. ## Test 3 — Compute returns 409 with `LINE4_EXCEPTION_BREAKOUT_STATEMENT_REQUIRED` when multiple exceptions apply and statement not prepared

Per IRS 2025 Form 1040 instructions p. 27, when more than one Line 4 exception applies, the filer must attach a written statement explaining how each amount is attributable to each exception.
