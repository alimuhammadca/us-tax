# Form 1040 Line 5b — Per-Test IRS Audit

**Status:** Tax Year 2025. Per-test review of every end-to-end and Java unit test that touches Form 1040 Line 5b (Pensions and annuities — taxable amount) or any code path that contributes to it. Each test is audited against the IRS source documents (`docs/books/i1040gi_2025.pdf`, IRS Publication 17, Publication 525, Publication 575, Publication 939, Internal Revenue Code §§61, 72, 72(t), 402(c), 402(l)) and the Java compute path in `TaxReturnComputeService.java`.

Each assertion is graded:

- **OK** — assertion matches what the IRS rule would compute by hand AND the code implements that rule correctly.
- **MISMATCH** — assertion value disagrees with the IRS rule (either the test asserts the wrong value, or the code computes the wrong value and the test locks it in).
- **GAP** — test is silent on an output the IRS rule says should be populated.

Companion docs:
- `XLS/Computations/5b.md` (line-level IRS rule + code implementation)
- `test_cases/5a.md` (Line 5a per-test audit — many tests assert both 5a and 5b; the Line 5a perspective is in 5a.md, the Line 5b perspective is here)

## Test 2 — Line 5a/5b/5c flow computes pension outputs and generates Form 4972/Form 5329

**Run:**
```powershell
npx playwright test --workers=1 --output=test-results-5b-t2 `
  line5abc-pension-withdrawals.spec.ts -g "Line 5a/5b/5c flow"
```

**Purpose:** Comprehensive Line 5b chain: rollover + Public Safety Officer (PSO) exclusion + Form 5329 + Form 4972 election absorbs the lump-sum.

**Inputs:**
- 1099-R: gross=5000, taxable=4500, iraSepSimple=false, code=1, box 5=100 (employee basis pre-netted by payer)
- Pension form: `hadPensionRollover=true`, `totalPensionRolloverAmount=200`, `box5Offset=100`, PSO eligibility+election, $500 PSO premiums, Form 5329 with exception code "01"
- Lump-sum form: `electsForm4972=true`, `sourceGrossDistributionAmount=5000`, `sourceTaxableAmount=4500`

**IRS rule chain per Publication 575:**
- Stage 0 taxable_base = $4,500 (box 2a)
- Stage 3 rollover: rolloverBase = box 1 gross = $5,000; `reducedBase = max(0, 5,000 − 200 − 100) = $4,700`; `taxableAfterRollover = 4,700` (no RRB)
- Stage 4 PSO: `psoExclusion = min(500, 3,000) = 500`; `taxableAfterPso = 4,700 − 500 = 4,200`
- Form 4972 subtraction: `current5b − sourceTaxableAmount = max(0, 4,200 − 4,500) = 0`

**By-hand expected:** Line 5b = 0; Line 5a = null (gross fully absorbed); Form 4972 generated; Form 5329 = 10% × $4,500 = $450.

**Code path:** `computePensionForPerson:9136-9419` (4-stage chain), `applyForm4972ToPensionLines:23560` (Form 4972 subtraction).

**Assertions affecting Line 5b:**

| # | Assertion | Expected (IRS) | Verdict |
|---|---|---|---|
| 1 | `taxablePensionsAnnuities === 0` | 0 | **OK** — chain produces exactly 0 |
| 2 | `pensionsAnnuities === null` | null | **OK** (Line 5a — see test_cases/5a.md) |
| 3 | `form5329.additionalTaxOnEarlyDistributions === 450` | 450 | **OK** (Form 5329 separate from Line 5b) |

**GAPs identified for Line 5b coverage:**
- **GAP B-A2.1** — Test does not assert the post-Form-4972 chain by asserting an intermediate value (e.g., that line 5b before Form 4972 would be 4,200). Locking in the Stage 1–4 chain's intermediate value would catch regressions in the chain itself.
- **GAP B-A2.2** — Test does not exercise the case where Form 4972 source amount is LESS than the chain result (e.g., chain → 4,200; Form 4972 absorbs 3,000 → Line 5b residual = 1,200). The "fully-absorbed" case is the only scenario tested.
