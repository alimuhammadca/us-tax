**Tax year: 2025.** Per-test review of every end-to-end test that touches **Form 1040 Line 6a (Social Security benefits — gross net-benefit amount)** in the named spec file. Each test is audited against:

- The 2025 Instructions for Form 1040 / 1040-SR (`C:/us-tax/docs/books/i1040gi_2025.pdf` and the parallel `.txt`)
- Publication 17 — Your Federal Income Tax (`C:/us-tax/docs/books/p17.pdf`), Chapter 7 (Social Security and Equivalent Railroad Retirement Benefits)

Each assertion is graded:

- **OK** — assertion value matches what the IRS rule would compute by hand, AND the code computes that same value

- **GAP** — the test is silent on an output the IRS rule says should be populated, or the assertion is too lenient (e.g., `toBeFalsy()` when IRS would specify a precise value)

### Purpose

Verifies that the *front-end* `social-security-benefits-taxpayer` form refuses to issue a PUT save request when the user has answered `hadSocialSecurityBenefits = Yes` and `receivedSsa1099 = Yes` and `uploadedSsa1099 = Yes` (so the user is claiming to have an uploaded statement), but no actual SSA-1099 statement entry exists in the database. The test does *not* exercise the Form 1040 computation — it only verifies a user-experience guard that prevents the user from accidentally saving an inconsistent state.

### Inputs

| Field | Value |
|---|---|
| Filing status | Default (Single) |
| SSA-1099 statement entries | None |
| RRB-1099 statement entries | None |
| `hadSocialSecurityBenefits` | true (via UI radio) |
| `receivedSsa1099` | true (via UI radio) |
| `uploadedSsa1099` | true (via UI radio) |
| `confirmAllReceivedSsaRrbStatementsUploaded` | true (via UI radio) |

### IRS rule (what we expect)

There is no IRS rule about the user interface. The applicable IRS-side rule is the upload-gating principle expressed in Pub. 17 Chapter 7 and the 2025 Form 1040 Line 6a instructions: a taxpayer who received Social Security benefits is required to report them, and Form SSA-1099 is the source document. If the user claims to have received and uploaded the statement but no statement exists in the system, no reliable Line 6a value can be produced.

### By-hand expected outputs

None — this is a UI test, not a compute test. The expected behavior is that the front-end Save button does not produce a PUT request to `/api/personal/social-security-benefits-taxpayer`.

### Code path

- Front-end Angular component `form-social-security-benefits-taxpayer.component.ts` — its `isValid()` / equivalent guard refuses to save when the upload-claim flags conflict with the actual statement count.
- Backend would also reject at compute time via the blocking flag `SOCIAL_SECURITY_STATEMENT_UPLOAD_REQUIRED` emitted from `validateSocialSecurityStatementGating()` in `TaxReturnComputeService.java` around line 12810. The compute-time gate is independent of the UI-time gate.

### Assertions

| # | Assertion | Verdict |
|---|---|---|
| 1 | `expect(await saveResponse).toBeNull()` — no PUT response received within 2.5 seconds | **OK** — the front-end correctly suppresses the save call. Matches the project's defense-in-depth pattern (UI gate + compute gate). |

### GAPs identified

- **GAP T1.1** — **CLOSED 2026-06-05** — New contrapositive test added: `"Taxpayer Social Security form SAVES once an SSA-1099 statement is on file (contrapositive to Test 1)"`. Seeds an SSA-1099, sets the same UI state that Test 1 blocks (hadSocialSecurityBenefits=Yes + receivedSsa1099=Yes + uploadedSsa1099=Yes + confirmAllReceivedSsaRrbStatementsUploaded=Yes), and asserts the PUT save returns 200. Proves the `isValid()` guard at `form-social-security-benefits-taxpayer.component.ts:640` is triggered only by the empty-statement state.
  - *Note on the original gap framing:* the audit suggested testing with `receivedSsa1099 = false`, but the form's `isValid()` requires `ssa1099Count + rrb1099Count > 0` whenever `hadSocialSecurityBenefits = true`, so the realistic contrapositive must seed a statement rather than drop the upload-claim flags.

## Test 2 — Line 6a/6b/6c/6d flow computes taxable benefits and sets checkbox flags

**Run:**
```powershell
npx playwright test --workers=1 --output=test-results-6a-t2 `
  line6abcd-social-security-benefits.spec.ts -g "Line 6a/6b/6c/6d flow"
```

### Purpose

The most comprehensive test in the spec. Exercises all four Line 6 sub-lines in a single Married-Filing-Separately + lived-apart-all-year + lump-sum-election scenario. Verifies that:

- Line 6a equals the full Form SSA-1099 Box 5 amount (lump-sum portion included).
- Line 6b reflects the **lump-sum election method** computation per Publication 915 Worksheet 3, not the simplified approximation.
- Line 6c (lump-sum election checkbox) is checked because the election reduces the taxable amount.
- Line 6d (Married Filing Separately + lived apart all year checkbox) is checked.

### Inputs

| Field | Value |
|---|---|
| Filing status | Married filing separately |
| SSA-1099 entry: `beneficiarySSN` | 111-11-1111 (matches taxpayer) |
| SSA-1099 entry: `netBenefitsAmount` (Box 5) | $30,000 |
| `hadSocialSecurityBenefits` | true |
| Upload gating fields | All true (statement counted as uploaded) |
| `livedWithSpouseAnyTimeDuringTaxYear` | false |
| `livedApartFromSpouseEntireTaxYear` | true |
| `hasLumpSumBackPaymentForPriorYears` | true |
| `electsLumpSumElectionMethod` | true |
| `manualOtherIncomeLine8` | $30,000 (manual override flowing into Form 1040 Line 8) |
| `lumpSumDetails` [0] `priorTaxYear` | 2024 |
| `lumpSumDetails` [0] `lumpSumAllocatedToPriorYear` | $5,000 (the part of the $30,000 that is a back payment for 2024) |
| `lumpSumDetails` [0] `priorYearTaxableBenefitsPreviouslyReported` | $3,000 |

### IRS rule (what we expect)

**Line 6a:** Per the 2025 Form 1040 Line 6a instruction quoted in `6a.md`: "Report your net benefits (the total amount from box 5 of all your Forms SSA-1099 and Forms RRB-1099) on line 6a." The lump-sum back-payment portion is included in Form SSA-1099 Box 5 by the Social Security Administration and is not separated out. Line 6a is the full $30,000 regardless of the election.

**Line 6b:** Computed by Publication 915 Worksheet 3 ("Lump-Sum Election Method") which adds two parts:

1. Recompute the *prior year* with the lump-sum portion added to that year's income. Take the increase in prior-year taxable benefits as the lump-sum portion's taxable contribution to the current year.
2. Run the standard Social Security Benefits Worksheet on the *current year* using only the non-lump-sum SS portion ($30,000 − $5,000 = $25,000).

Add (1) + (2) and compare to the result the standard worksheet would have produced on the full $30,000. Use whichever is smaller as Line 6b (and check Line 6c if the election method is smaller).

**Line 6c:** True when the election method produces a smaller Line 6b than the standard method.

**Line 6d:** True because filing status is Married Filing Separately AND the taxpayer lived apart from the spouse for the entire tax year.

### IRS rule (what we expect)

Per Publication 17 Chapter 7 ("Repayment of benefits" subsection, last paragraph) and Publication 915 ("Repayments More Than Gross Benefits"):

- **The taxpayer does NOT enter a negative number on Line 6a.** When repayments exceed gross, the gross-side Line 6a is reported as $0, and Line 6b is also $0.
- The taxpayer may be entitled to:
  - A **deduction on Schedule A (Itemized Deductions) under "Other Itemized Deductions"** if the excess repayment is $3,000 or less.
  - A **tax credit under Internal Revenue Code §1341 (the "claim of right" credit)** if the excess repayment is more than $3,000. The taxpayer chooses whichever of the deduction or the credit produces the lower overall tax.
- The application should at minimum *flag* the case so the user does not silently miss the deduction or credit.

### By-hand expected outputs

**IRS-correct expectation:**
- Line 6a: $0 (explicit zero, not blank)
- Line 6b: $0
- Schedule A "Other Itemized Deductions" line should be auto-populated with $1,000 (the excess) OR a §1341 credit should be auto-computed (whichever is smaller, since the $1,000 excess is below the $3,000 threshold the deduction is more typical here).
- An advisory non-blocking flag should at minimum alert the user.

**Current code behavior:**
- The code returns the SS computation as null (Line 6a and Line 6b are absent from the response — i.e., "falsy" in JavaScript sense).
- The code emits an advisory non-blocking flag `SOCIAL_SECURITY_NEGATIVE_NET_BENEFITS_MANUAL_REVIEW`.
- The Schedule A deduction / §1341 credit is **not** auto-computed.

### Code path

- `computeSocialSecurityBenefits()` (~line 12443) detects `line6a < 0` and:
  - Emits the advisory flag `SOCIAL_SECURITY_NEGATIVE_NET_BENEFITS_MANUAL_REVIEW` (non-blocking).
  - Returns null from the orchestrator, suppressing the SS computation entirely so neither Line 6a nor Line 6b appears in the output JSON.
