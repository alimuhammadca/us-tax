## Test 5 — `Lump-sum election with missing prior-year previously-reported amount fires §17 (closes 6b.md Gap 9 e2e)`

### Command

```powershell
cd C:\us-tax\us-tax-be\e2e
npx playwright test --workers=1 --output=test-results-6c-t5 --grep "Lump-sum election with missing prior-year previously-reported" line6abcd-social-security-benefits.spec.ts
```

### Purpose

Negative-path §17 test for the Gap 9 closure. Verifies that when the user elects the lump-sum method AND provides a row with positive `lumpSumAllocatedToPriorYear` BUT omits `priorYearTaxableBenefitsPreviouslyReported`, the orchestrator emits `SOCIAL_SECURITY_LUMP_SUM_PRIOR_YEAR_REPORTED_REQUIRED` as a §17 non-overrideable blocker (HTTP 409 with `overrideFlags=true`). Without this guard the recompute silently treats prevTaxable as zero, inflating `additional` to the full recomputed amount and defeating the election.

### Inputs seeded

| Source | Field | Value |
|---|---|---|
| filing-status | (not set) | defaults to Single |
| SSA-1099 | `netBenefitsAmount` | `$30,000` |
| social-security-benefits-taxpayer | `hasLumpSumBackPaymentForPriorYears` | `true` |
| social-security-benefits-taxpayer | `electsLumpSumElectionMethod` | `true` |
| lumpSumDetails[0] | `priorTaxYear` | `2024` |
| lumpSumDetails[0] | `lumpSumAllocatedToPriorYear` | `$5,000` |
| lumpSumDetails[0] | **`priorYearTaxableBenefitsPreviouslyReported`** | **(omitted)** |

All three V48 prior-year override columns also omitted.

### IRS-rule expectation

Per IRC §86(e) + Pub. 915 Worksheet 3: the recompute requires the per-prior-year *actually-reported* taxable amount as the subtrahend. Without it the worksheet cannot be completed correctly.

The IRS does not block the filing at the form level (the worksheet is taxpayer-completed and the IRS only sees the result), but a tax-preparation application that silently fills the missing value with zero would silently defeat the user's election. The application's §17 disposition is the right defense-in-depth.

### Java code path

1. `validateLumpSumPriorYearReportedAmounts` runs first when `electsLumpSumElection = true`. Detects the missing field on the row with positive `allocated`. Emits §17 `SOCIAL_SECURITY_LUMP_SUM_PRIOR_YEAR_REPORTED_REQUIRED`.

### X4 incident details (post-validation finding)

**Discovered:** 2026-06-06, when the user ran the full 22-test e2e spec after the initial closure. Got 21/22 — only the X2 test failed.

**Symptom:**
```
Expected: 409
Received: 200
  at C:\us-tax\us-tax-be\e2e\tests\line6abcd-social-security-benefits.spec.ts:1331:25
```

**Root cause:** Promoting a flag to §17 non-overrideable requires **two** changes:
1. Set `blocking=true` on the `TaxReturnFlag` constructor at the emission site (done).
2. Add the flag code string to `NonOverrideableFlags.CODES` (missed initially).

Without #2 the flag is just a "blocking flag." Without `overrideFlags=true` in the HTTP call, ANY blocking flag yields 409 — so the default compute path was correct. But the X2 e2e test (correctly) used `?overrideFlags=true` to specifically validate the §17 enforcement layer; unregistered codes are treated as overrideable, the override succeeded, compute returned 200.

**Why Java unit tests didn't catch it:** the lock-in tests assert `f.isBlocking() == true` on the flag, not the HTTP response code. The flag IS blocking at the orchestrator level; the registry check happens later at the HTTP layer.

**Why this matters as a doc-level gap (not just a code bug):** the closure pattern "promote advisory to §17" was documented as one step (flip `false` → `true`) but is actually two steps. The follow-up-action table for GAP T5.1 / GAP X2 silently assumed one step. Future §17 promotions in OTHER lines (not just Line 6c) would hit the same trap unless the two-step pattern is documented.

**Test-design lesson captured:** §17 e2e lock-in tests MUST use `?overrideFlags=true` in the compute POST. Tests that use the default compute path return 409 for ANY blocking flag and so do not distinguish "flag is blocking" from "flag is also registered as non-overrideable." All three lump-sum §17 e2e tests (existing Test 5 for Gap 9, new X2 for Gap 2, existing Test 6 in `XLS/Computations/6a.md` for excess-repayment) already follow this pattern.
