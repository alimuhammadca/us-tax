Line 7b on the 2025 Form 1040 is not a money amount in its own right. It is a disclosure strip composed of **three discrete outputs**:

| Output | JSON field on `form1040.income` | PDF field | Source rule |

## Test 1 — `Taxpayer capital gain/loss form blocks save when no statements are uploaded`

### Command

```powershell
cd C:\us-tax\us-tax-be\e2e
npx playwright test --workers=1 --output=test-results-7b-t1 --grep "blocks save when no statements" line7ab-capital-gain-loss.spec.ts
```

### Purpose

Lock in the client-side save-guardrail at `form-capital-gain-loss-taxpayer.component.ts:865` (`isValid()`). When the user affirms "I received a 1099-DIV with capital-gain distributions" AND "I uploaded the 1099-DIV" but no 1099-DIV statement actually exists, the form refuses to PUT. This protects line 7b: incoherent intake data can never feed the line 7b computation (Outputs A, B, C are never produced for this user state).

### Inputs seeded

| Source | Field | Value |
|---|---|---|
| (none — `seedRequiredDob` only) | — | (no statements, no spouse, no dependents) |
| capital-gain-loss-taxpayer (UI) | `hadCapitalGainOrLoss` | `Yes` |
| capital-gain-loss-taxpayer (UI) | `hasUploadedAtLeastOneCapitalStatement` | `Yes` |
| capital-gain-loss-taxpayer (UI) | `confirmAllReceivedCapitalStatementsUploaded` | `Yes` |
| capital-gain-loss-taxpayer (UI) | `received1099DivWithCapitalGainDistributions` | `Yes` |
| capital-gain-loss-taxpayer (UI) | `uploaded1099DivWithCapitalGainDistributions` | `Yes` |

### IRS-rule hand verification (line 7b angle)

If the user reaches line 7b computation with claimed-but-unuploaded 1099-DIV box 2a, Output A would be set to whatever the 8-condition Exception 1 gate produces from the empty per-person totals (probably TRUE — no QOF, no sales, no carryover, no special boxes), Output C would be `null` (no child amount), and Form 1040 line 7a would be `$0`. **This is wrong** — the taxpayer asserted they received capital-gain distributions; printing `$0` on the return is silent misreporting. The IRS-correct behavior is therefore: do not produce line 7b output at all until the user provides the underlying statement. The application implements this correctly by blocking the save.

### Java / TypeScript code path

1. `form-capital-gain-loss-taxpayer.component.ts:865` — `isValid()` short-circuits to `false` because `received1099DivWithCapitalGainDistributions === true && (uploaded1099Div !== true || divCount <= 0)` and `divCount = 0` (no statement seeded).
2. Save click in `save()` returns early without calling `personalDataService.savePersonal(...)` → no PUT.
3. Compute is never invoked → line 7b outputs are never produced.

### Assertion report

| # | Assertion | Expected | Actual | Verdict |
|---|---|---|---|---|
| 1 | `await saveResponse === null` (no PUT in 2.5 s) | null | null | **OK** |

### Line 7b coverage gaps

- **GAP T1.7b.1 — ACCEPTED 2026-06-06**: the no-PUT assertion IS the load-bearing IRS-relevant guarantee at this layer (compute is not invoked → no line 7b outputs to assert). The compensating positive coverage lives in Test 2 (server-side mirror for the 1099-DIV branch) and in the three new Tests 9 / 10 / 11 (server-side mirrors for the 1099-B / Form 2439 / other-source branches added during the 2026-06-06 gap-closure pass).
